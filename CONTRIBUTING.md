# 仓库组织与写作规范

本文件定义仓库的组织规则与笔记写作规范。

## 快速导读

如果你带着具体任务阅读，优先按任务入口跳转：

| 你要做什么 | 优先阅读 | 目的 |
|---|---|---|
| 新增一篇笔记 | 第七章 7.1、7.2、7.4 + 第二章 | 先判断材料是否适合写成正文，再确定目录和命名 |
| 新增一个子目录 | 第七章 7.5 + 第一章、第三章 | 先确定目录定位、层级和 README 要求 |
| 处理外部 AI 调研、网页搜索或临时笔记 | 第七章 7.3、7.8、7.10 | 先做证据分级和材料分流，不直接写正文 |
| 记录主题缺口或待研究对象 | 第五章 L4/L5 + 第七章 7.2、7.9 | 区分问题缺口与研究对象 |
| 处理术语、事实或结论冲突 | 第五章 L7 + 第七章 7.7 | 先记录冲突，再决定是否修改正文 |
| 查某个元信息文件的职责 | 第五章 | 理解 `README.md`、`overview.md`、`backlog.md` 等文件边界 |
| 优化单篇文档表达 | 第六章 | 调整结构、公式、表格或论文笔记写法 |

如果只记一条默认策略：**拿不准时，先写元信息文件，不抢写正文。**

## 一、层级设计

仓库按纵向技术栈分层，辅以横向跨学科层：

| 层级 | 定位 | 目录 |
|---|---|---|
| 基础方法层 | 经典算法与理论基础 | traditional-ml、deep-learning、reinforce-learning |
| 核心领域层 | 仓库主线 | llm、cv、knowledge-graph |
| 应用与集成层 | 技术落地与组合 | rag、agentic、world-models、embodied-intelligence |
| 基础设施层 | 训练与工程支撑 | training-infra |
| 跨学科层 | 横向交叉视角 | interdisciplinarity |
| 支撑资源 | 辅助资料 | learning-materials、assets |

## 二、目录与文件命名

- 一级目录：英文短名，语义自解释
- 二级及以下目录：`编号-英文短名` 格式（如 `01-foundations/`），编号保证排序稳定
- 目录名不使用中文
- 笔记文件：英文短名，语义自解释（如 `lora.md`、`rlhf-overview.md`）
- 论文笔记：`论文简称_arXiv编号.md`（如 `ReAct_2210.03629.md`）或 `论文简称_会议年份.md`（如 `BeingH05_ICLR2026.md`）

## 三、展开深度

根 README 的目录树展开深度按内容活跃度区分：

- **核心领域**（traditional-ml、deep-learning、reinforce-learning、llm、cv、rag、agentic、knowledge-graph）展开到 4 级——内容充实、更新频繁，需要更细粒度的导航
- **次要领域**（embodied-intelligence、world-models、training-infra、interdisciplinarity、learning-materials）展开到 3 级——内容量较少或更新节奏较慢
- 各一级目录有自己的 README，承载完整目录结构（不受根 README 深度限制）

## 四、README 规范

每个一级目录下必须有 README.md。二级及更深层目录仅在存在或未来可能容纳子目录时必须写 README；纯文件目录（仅含笔记文件，不会有子目录）不需要。

### 通用规则

所有 README 必须遵守：

- **标题**：使用清晰自然、语义自解释的标题即可，不要求中英对照
- **目录结构**：必填，展开到已有内容的最深层级，不预占位。根 README 的展开深度受"展开深度"规则限制，各一级目录 README 不受此限制。**原则上目录树只展示目录，不展示文件**（除非是用于说明组织结构的元信息文件，如 README.md、template.md）
- **元信息位置**：如需标注 `最后更新`、来源、适用范围、状态等元信息，统一放在标题下方、正文开始之前，便于 AI 和读者在进入正文前快速获取上下文；不强制要求在文末单独标注更新时间
- **跨目录引用**：基础内容放在所属领域目录下，其他目录通过相对路径引用，避免重复维护

### 根 README

根 README 面向首次访问者，结构建议：

- 仓库定位与一句话说明
- 层级关系表或 Mermaid 图（纵向技术栈 + 横向跨学科层）
- 各一级目录导航链接（带一句话定位）
- 目录结构（按活跃度区分展开深度，见第三章）
- 如何阅读本仓库（可选，推荐路径或适合人群）

### 按类型细化的结构建议

不同类型 README 在通用规则基础上，建议包含以下内容（按需增减）：

**一级目录（技术类）**

- 一句话定位
- 分类依据（解释二级目录的组织逻辑）
- 边界说明（什么放本目录、什么应放其他目录）
- 目录结构
- 同目录导航（优先提供正文主题、子目录入口或阅读路径，不主动列出元信息文件）
- 开源仓库与工具存放指南
- 学习路径或推荐阅读顺序
- 与其他目录的关系

**一级目录（跨学科类）**

- 一句话定位引言
- 排序逻辑说明（放在目录结构之前，先建立框架再列细节）
- 目录结构（子目录按 **个体→系统→群体→制度** 递进排列）
- 各领域概览（用映射表关联纵向技术目录）
- 与纵向技术栈的关系

  示例（`interdisciplinarity/`）：
  ```
  个体层面   01 认知科学与神经科学
            02 语言学与语用学
            03 心灵哲学
  系统层面   04 控制论与系统论
            05 复杂系统与涌现
  群体层面   06 社会学与组织管理
            07 经济学与博弈论
  制度层面   08 法学与治理
  ```

**子目录（论文集）**

- 研究主题
- 创建日期
- 论文列表（推荐表格：论文、文档链接、arXiv ID、来源、核心贡献、重要性）
- 主题分类
- 阅读建议
- 工具与资源
- 相关资源
- 参考文献 BibTeX

  重要性三级标注：⭐⭐⭐ 必读 / ⭐⭐ 值得关注 / ⭐ 简单参考

**子目录（技术主题）**

- 目录结构（推荐表格：目录、说明，可加"状态"列标注填充进度）
- 领域特色内容
- 相关资源

## 五、元信息文件模型

本章回答“仓库中哪些文件属于元信息文件、它们分别服务什么认知任务、应该在什么情况下出现”。

- 它是**参考手册**，用于查清 `README.md`、`overview.md`、`backlog.md`、`research-queue.md` 等文件的职责边界。
- 它不负责给出材料处理流程；拿到新材料后如何分级、分流和落地，见第七章。
- 它不讨论单篇正文如何写得更清楚；那属于第六章。

如果你只是想判断“这个目录下应该有哪些元信息文件”，读本章；如果你手上已经有一份材料，想知道下一步怎么处理，直接读第七章。

除笔记正文外，目录中的“元信息文件”（非内容文件）按以下七层模型组织。每层对应一种**读者认知任务**，按需出现，不强制全部存在。

| 层级 | 认知任务 | 文件名 | 职责 | 出现条件 | 典型内容 |
|:---|:---|:---|:---|:---|:---|
| L1 定向 | Orient | `README.md` | “这是什么地方？跟我有什么关系？” | 目录存在或未来可能容纳子目录时**必选**；纯叶子目录（仅含文件，无子目录可能）不需要 | 一句话定位、快速入口、目录树 |
| L2 发现 | Discover | `index.md` | “我要的文件/主题在哪？” | 文件/子目录 > ~10，或主题交叉关系复杂 | 关键词索引、反向映射、问题-文档对照表 |
| L3 理解 | Understand | `overview.md` | “这个领域的全貌和现状是什么？” | 该目录代表一个需要建立整体认知的领域 | 技术分类、代表工作、能力边界、关键 trade-off |
| L4 推进 | Track | `backlog.md` | “这个领域还有什么没被覆盖？” | 目录处于活跃建设期，内容有明显缺口 | 已知空白、待填补主题、与现有内容的关联 |
| L5 选题 | Queue | `research-queue.md` | “接下来值得研究哪些对象？” | 需要持续跟踪论文、仓库、官方文档、产品、协议、benchmark 等待研究对象，且对象列表不宜塞进 `backlog.md` 时可选创建 | 待研究对象、对象类型、关联问题、研究理由、当前状态、产出链接 |
| L6 规划 | Plan | `roadmap.md` | “从哪开始、按什么顺序阅读/建设？” | 有明确的内容组织顺序或学习路径 | 阶段划分、推荐顺序、前置依赖 |
| L7 校验 | Reconcile | `conflict.md` | “这里是否存在定义冲突或口径不一致？” | **仅当发现冲突时可选创建** | 冲突点、涉及文档、冲突类型、待核验问题 |

**命名约定**：L3 的标准文件名为 `overview.md`。`summary.md` / `SUMMARY.md` 已废弃——`summary` 语义过载（既可指内容摘要，也可指领域综述），`overview` 明确指向“领域全貌”。若目录下已有内容摘要性质的简短文件，应扩展为领域全貌描述后再使用 `overview.md`。

### L1 `README.md`：定向

- **一句话定位**：本目录是什么、面向谁
- **快速入口**：按背景推荐路径、核心文档链接
- **目录结构**：树状图（遵循第四章通用规则）。允许用注释标注未创建目录的定位（如 `├── 02-linguistics/  # 语义/语用、言语行为`），方便读者理解知识体系完整结构；注释应简洁，详细内容在对应目录创建后放入其 README
- **同目录导航**：如需提供导航，优先列出正文主题、子目录入口或阅读路径；不要在 README 中主动列出 `overview.md`、`backlog.md`、`research-queue.md`、`roadmap.md`、`conflict.md` 等元信息文件
- **禁止放入**：详细技术分类、论文列表、长篇学科导论

### L2 `index.md`：发现

- 当层级目录树不足以支持快速定位时使用
- **主题索引**：按关键词/问题领域扁平聚合，而非层级展开
- **反向索引**：如跨学科目录可建立“技术领域 → 关联学科视角”的反向映射
- **与 overview.md 的区别**：`index.md` 只回答“在哪”，不回答“是什么”；禁止放入概念解释、技术判断、领域分析
- 本质是**网**不是**树**

### L3 `overview.md`：理解

- 回答**领域**现状：该技术领域已覆盖什么范围、能力边界在哪（**不是目录划分，而是技术本身的适用范围**）、关键概念如何关联
- 技术目录（**技术分类学，不是目录结构**）：技术分类、代表工作、能力边界、关键 trade-off
- 跨学科目录：学科地图、映射关系、覆盖度仪表盘
- **与 index.md 的区别**：`overview.md` 回答“是什么”和“为什么重要”，不列出具体文件位置；读者在 overview 中获得认知后，再去 index 或目录树中定位具体文件

### L4 `backlog.md`：内容缺口

- 记录**该领域已被识别但尚未被覆盖的内容空白**
- 按优先级分组（P0 已识别高价值 / P1 有价值但模糊 / P2 长期方向）
- 每个条目标注：关联子目录、现有相关资源、为何尚未覆盖
- **不是任务列表**：不分配责任人、不设 deadline、不追踪完成状态；目的是让读者快速判断“这个领域目前覆盖到什么深度，还有什么值得了解”
- **与 `research-queue.md` 的边界**：如果关注点是“问题还没讲清楚”，放入 `backlog.md`；如果关注点是“有一个具体对象值得读/看/验证”，放入 `research-queue.md`

### L5 `research-queue.md`：待研究对象队列（可选）

- 记录**该领域下一步值得研究的对象**，如论文、GitHub 仓库、官方文档、产品、协议、benchmark 等
- 每个条目建议包含：对象名、对象类型、关联问题、为何值得研究、当前状态（待看 / 进行中 / 已完成 / 已转专题）、产出链接
- **按需创建**：仅当对象队列已形成稳定维护需求，且不适合塞进 `backlog.md` 时使用；不要为每个目录机械配齐
- **与 `backlog.md` 的边界**：`backlog.md` 记录问题缺口，`research-queue.md` 记录待研究对象。比如“Agent 环境边界还没讲清楚”是 backlog；“需要读 OpenAI Agents SDK 文档、AutoGPT Forge 仓库、某篇 sandbox 论文”是 research-queue
- **不确定时的默认判断**：先问“我现在掌握的是一个问题，还是一个对象？”问题进 `backlog.md`，对象进 `research-queue.md`；如果两者都有，允许在 backlog 中写问题，在 research-queue 中列支撑该问题的研究对象，并互相引用

### L6 `roadmap.md`：路径指引（可选）

- 按阶段或模块组织，有明确的**阅读/学习顺序**和前置依赖
- 适合：学习路线图（告诉读者按什么顺序阅读本目录下的笔记）、目录建设计划（说明未来内容扩展的方向和先后逻辑）
- **不是项目管理工具**：不设时间 deadline、不追踪完成状态；重点是“顺序”和“依赖”，而非“进度”

### L7 `conflict.md`：冲突记录（可选）

- 用于记录**当前所在层级目录范围内**的冲突，以及其**直接子目录之间（含跨子目录层级）**的冲突；以便快速定位责任边界与维护范围
- **仅在发现冲突时创建**；无冲突则不创建该文件
- 每条记录建议包含：冲突描述、涉及文档路径、冲突类型（术语/事实/结论/版本）、当前证据、待核验问题、建议处理方向
- 该文件是**研究与修订输入**，不是最终结论文档；冲突解决后应在相关文档修正，并同步更新或移除对应记录

## 六、文档写作与表达建议

本章回答“内容已经决定写成一篇文档后，怎样写得更清楚、更易读”。

- 它主要关注**表达形式**，例如结构、公式、表格、论文笔记写法。
- 它不负责判断材料该进入 `overview.md`、`backlog.md`、`conflict.md` 还是正文；这属于第七章。
- 它也不负责定义元信息文件职责；这属于第五章。

以下为推荐做法，不强制：

- 知识性笔记：可包含数学公式（LaTeX）、代码块、表格等
- 论文笔记：建议包含核心贡献、方法描述、关键结果，不必逐段翻译
- README 中如有流程或阶段关系，可用代码块/ASCII 图展示

## 七、材料处理与文档增改流程

本章回答“拿到一份新材料，或准备新增、迁移、修改文档时，实际应该按什么顺序动手”。

- 它是**操作手册**，从材料判断开始，到落入元信息文件或正文专题结束。
- 它引用第五章的元信息文件模型，但不重新展开每个文件的完整定义。
- 它也不讨论单篇文档的表达风格；那属于第六章。

本章的核心原则是：

> **正文负责表达经过整理和判断后的知识；元信息文件负责承接尚未稳定、尚待分流、尚待核验的材料。**

对人类读者，本章用于减少返工；对 AI 协作者，本章用于避免“先写正文、后补判断”的顺序错误。

### 7.1 总流程

> 先判断，再落位，最后写正文

拿到新材料时，默认按以下顺序处理：

```text
新材料
  → 判断材料类型
  → 判断证据强度
  → 选择落位
  → 执行新增、修改、迁移或保留
  → 必要时同步 README / overview / backlog / conflict
```

四个前置问题：

1. **材料类型**：它是整体认知、主题缺口、待研究对象、冲突问题，还是正文专题材料？
2. **证据强度**：它属于 `Verified`、`Observed`，还是 `Unverified / Unreliable`？
3. **目标落位**：它更适合进入 `overview.md`、`backlog.md`、`research-queue.md`、`conflict.md`，还是某篇正文专题？
4. **临时落点**：如果今天不写正文，最合适的暂存位置是什么？

如果以上问题没有想清楚，默认不要直接新建正文专题。

常见材料的入口：

| 材料或场景 | 默认入口 | 说明 |
|---|---|---|
| 新论文阅读笔记 | `7.2`、`7.3`、`7.4` | 论文原文可作为强证据，但笔记中的解读仍需判断是否适合进入正文 |
| 外部 AI 调研报告 | `7.3`、`7.8`、`7.10` | 默认视为上游输入，先分级、分流，再决定是否回流正文 |
| 网页搜索结果或博客整理 | `7.3`、`7.8` | 多数属于候选证据，不直接写成主干结论 |
| 发现概念、事实或版本冲突 | `7.2`、`7.7` | 先进入 `conflict.md`，不要直接改写正文定论 |
| 发现主题缺口 | `7.2`、`7.9` | 优先进入 `backlog.md`，不要用正文占位 |
| 发现具体论文、仓库、benchmark 值得研究 | `7.2`、`7.9` | 优先进入 `research-queue.md` |

### 7.2 新材料先分类型

新材料通常包括：临时笔记、`temp/` 文档、网页搜索结果、外部 AI 调研报告、博客摘录、issue/PR 讨论、论文阅读笔记、开源仓库观察记录。

这些材料**默认都不是正文**，也不默认等于主干结论；它们首先是“输入材料”或“候选证据”。

| 材料类型 | 优先落位 | 判断标准 |
|---|---|---|
| 整体认知 | `overview.md` | 已经较稳定，能帮助建立领域全貌、分类、边界或关键 trade-off |
| 主题缺口 | `backlog.md` | 关注点是“还有什么问题没讲清楚、还缺哪类内容” |
| 待研究对象 | `research-queue.md` | 关注点是“有哪篇论文、哪个仓库、哪份文档、哪个 benchmark 值得研究” |
| 冲突问题 | `conflict.md` | 涉及术语、事实、版本状态、适用边界或结论力度的不一致 |
| 正文专题 | 下沉到最贴近主题的正文文件 | 问题边界清晰，证据相对充分，适合围绕一个具体问题系统展开 |

### 7.3 外部材料先做证据分级

从 `temp/`、网页搜索、外部 AI 调研或临时笔记回流主干时，先按以下三级判断：

- **Verified**：可追溯到论文原文、官方文档、GitHub 源码、release notes、维护者正式说明等一手材料，且关键结论能被直接支撑
- **Observed**：有多个间接来源或高质量二手材料支持，方向上值得重视，但关键细节或原始依据仍不完整
- **Unverified / Unreliable**：主要来自单篇博客、媒体报道、社区帖子、第三方转述、AI 摘要，或虽有启发但无法回溯到足够强的原始出处

处理方式：

- `Verified`：可作为进入 `overview.md` 或正文专题的候选材料
- `Observed`：优先进入 `backlog.md`、`research-queue.md` 或 `conflict.md`，等待后续补证或澄清
- `Unverified / Unreliable`：只保留为线索，留在 `temp/`、阅读笔记或对象队列中；不要直接写成主干定论

> **外部 AI 调研结果默认只是上游输入，不是可直接发布的主干正文。**
>
> 不要因为它写得完整、结构漂亮、像综述，就跳过核验和分流步骤。

### 7.4 新增笔记

1. 先完成 `7.1` 的四个前置判断；不要把“还未整理的材料”直接写成正文
2. 确定所属领域，找到对应一级目录
3. 根据主题找到最匹配的编号子目录（如 LLM 训练相关 → `llm/03-training/`）
4. 按第二章规则命名文件
5. 如无法归入现有子目录，可新建子目录（遵循 `编号-英文短名` 格式），并同步更新该领域的 README
6. 如果材料主要来自外部调研、网页搜索或临时笔记，优先考虑先写入 `backlog.md`、`research-queue.md` 或 `conflict.md`，而不是立即形成正文

### 7.5 新增子目录

1. 确定在层级中的位置和编号（接续现有编号）
2. 判断该目录主要承载正文专题，还是先承接元信息文件；目录定位先于内容填充
3. 如果该目录存在或未来可能容纳子目录，创建目录并添加 `README.md`，内部用一句话说明该主题的定位和研究计划（无需展开）；纯文件目录（无子目录可能）不需要 README
4. 更新该一级目录的 README 目录结构
5. 如新增子目录使该领域的层级深度超出根 README 的展示限制，同步更新根 README

### 7.6 新增一级目录

1. 评估该目录属于哪个层级（基础方法 / 核心领域 / 应用与集成 / 基础设施 / 跨学科 / 支撑资源）
2. 创建目录、README.md（一句话定位）
3. 按层级顺序更新本文件的层级设计表
4. 更新根 README 的层级关系、知识体系（Mermaid 图）、目录结构三个章节
5. 若该目录短期内主要处于调研积累阶段，优先先建立 `README.md` 与必要的元信息文件，而不是预先创建大量正文空壳

### 7.7 修改已有正文

1. 先判断新增材料是在**强化现有结论**，还是在**引入新的冲突/限制条件**
2. 如果只是强化已稳定结论，可直接补入正文，并视需要更新 `overview.md`
3. 如果会改变术语、事实、版本状态、适用边界或结论力度，先更新 `conflict.md` 或相关元信息文件，再决定是否修改正文
4. 如果只是发现“这里还值得继续写”，优先更新 `backlog.md`，不要用正文占位
5. 修改正文后，如目录导航、边界说明或阅读路径受到影响，再同步更新 README

### 7.8 处理外部调研结果

拿到一份外部 AI 调研、网页搜索结果、博客整理或阅读笔记后，默认只做以下最小动作：

1. 标记其中哪些结论已经能追到一手来源，哪些只是间接观察
2. 把“值得继续展开的主题缺口”写入 `backlog.md`
3. 把“值得继续跟踪的对象”写入 `research-queue.md`
4. 把“定义/事实/结论冲突”写入 `conflict.md`
5. 只有剩余的稳定部分，才考虑回流 `overview.md` 或正文专题

外部 AI 特别容易在以下场景中产生“看起来可信、实际上不稳”的内容：

- 把第三方博客中的实现细节写成官方事实
- 把趋势判断写成行业共识
- 把单篇论文的表述写成稳定结论
- 把量化数字、性能比较或安全结论写得过满

### 7.9 默认策略

- **拿不准写哪里**：先写元信息文件，不抢写正文
- **拿不准是否可靠**：先降级处理，不抢写定论
- **拿不准是 backlog 还是 research-queue**：问题进 `backlog.md`，对象进 `research-queue.md`；两者都有时允许互相引用
- **拿不准是否该新建专题**：先补 `README.md`、`backlog.md` 或 `conflict.md`
- **拿不准是否该迁移出 `temp/`**：先保留在 `temp/`，等落位和证据强度明确后再迁移

### 7.10 AI 协作者执行前检查清单

AI 协作者在创建新文件、修改正文或迁移 `temp/` 内容前，必须先检查：

- 是否已经判断材料类型，而不是看到内容完整就直接写正文？
- 是否已经判断证据强度，区分 `Verified`、`Observed` 和 `Unverified / Unreliable`？
- 如果材料来自外部 AI、网页搜索或博客，是否已经把它当作上游输入，而不是主干结论？
- 是否存在更合适的元信息落点，例如 `backlog.md`、`research-queue.md` 或 `conflict.md`？
- 是否把“问题缺口”和“研究对象”混在同一个文件里？如混合出现，是否做了互相引用？
- 是否正在为某个目录机械创建所有 L1-L7 元信息文件？若是，应停止；元信息文件按需出现。
- 是否因为拿不准而新建了正文专题？若拿不准，应先降级到元信息文件或保留在 `temp/`。

## 八、其他规则

- **空目录**：
  - 如果该目录存在或未来可能容纳子目录：优先放入 `README.md`，用一句话说明该主题的定位和研究计划。既保持结构完整，又为未来贡献预留语义接口
  - 文件非常少的底层目录，或快速调整目录结构时的临时状态，可用 `.gitkeep` 占位
  - `.gitkeep` 仅作为中间状态，目录稳定后应替换为 `README.md`
  - 纯文件目录（不会容纳子目录）不需要 README 或 .gitkeep，直接等文件填充即可

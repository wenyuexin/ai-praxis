# Research Ingestion Design

本文回答：**“从外部临时材料到稳定知识层”的处理，为什么不只是若干规则，而应被看作一条 skill-like workflow；这条链在系统里应放在哪一层，又分别由哪些文档承接。**

它属于 design layer：不直接替代 `docs/contributing/` 的执行规则，也不直接替代 `docs/capabilities/ingest.md` 的能力说明。本文只处理一个更上位的问题：**为什么这条研究型输入吸收链应该被单独建模。**

## 1. 问题是什么

这个仓库长期会接收大量外部输入，例如：

- `temp/` 下的临时整理稿
- 外部 AI 调研结果
- 联网检索 / 网页抓取结果
- 博客摘录、社区讨论、issue / PR 线索
- 论文阅读笔记
- 对开源仓库、产品或 benchmark 的观察记录

这些输入的共同问题不是“怎么写得更顺”，而是：

- 它们默认还不是正文
- 它们往往混合了一手材料、二手转述和模型综合判断
- 它们经常已经足够完整，以至于 AI 很容易误把它们当成正文草稿
- 它们如果不经过中间层处理，就会污染正文、打断 Evidence 链，或者让 `temp/` 变成事实上的长期来源库

因此，这里真正需要建模的不是单次写作动作，而是一条**研究型输入吸收链**。

## 2. 为什么它不只是规则集合

如果这件事只是规则集合，那么把“不要直接引用 `temp/`”“要补 Sources”“要写 Trace”列出来就够了。

但实际不是。

这条链有几个明显的 workflow / skill 特征：

1. **有输入类型判断**：先判断材料是什么，而不是先判断怎么写。
2. **有阶段顺序**：分型、分级、分流、暂存、回流，不是可任意交换顺序的动作。
3. **有中间稳定层**：`backlog.md`、`candidates.md`、`conflict.md`、对象目录 `notes/` 都不是正文，但也不是垃圾桶。
4. **有 stop-line**：例如不能把临时调研稿直接润色成正文，不能把 `temp/` 当成最终来源。
5. **有回退路径**：归属未定、证据不稳、来源未恢复时，应停留在更低层，而不是硬推正文。
6. **有跨文档协作**：规则层、能力层、对象研究结构层需要同时配合。

也就是说，它不是“几条文档规则的并列列表”，而是一条**具有执行顺序和中间状态的 skill-like workflow**。

## 3. 为什么它也不只是单篇 capability 文档

把它放进 `docs/capabilities/ingest.md` 是对的，但只放在那里也不够。

原因是：

- capability 文档回答的是“系统提供什么能力、为什么需要这项能力”
- 这条链还额外涉及“为什么原有 workflow 文档会越来越像 skill 手册”
- 它还涉及 design layer 的一个更高位判断：哪些东西应留在规则层，哪些东西应上提到 capability layer，哪些又只是结构落点规则

所以，`Ingest` 是这条链的核心能力位，但**研究型输入吸收链本身**是一个更接近系统设计的问题。

## 4. 这条链在系统中的位置

当前更合理的分层是：

- `docs/capabilities/ingest.md`：回答“系统如何识别并吸收新输入”
- `docs/contributing/documentation-workflow.md`：回答“真正执行时，哪些判断顺序、门槛和 stop-line 不能跳过”
- `docs/contributing/research-artifacts.md`：回答“当输入已经归属于某个对象目录时，中间产物如何落到 `notes/` / 正文 / 机制专题”
- `docs/contributing/evidence-and-traceability.md`：回答“Evidence 与 Trace 如何组合执行”

因此，这条链不是取代这些文档，而是解释它们为什么必须共同存在。

## 5. 当前最合理的系统理解

当前阶段，最稳的理解不是“已经有了一项独立 product skill”，而是：

> **这是一条由 `Ingest` 主导、并受规则层与研究结构层约束的 skill-like workflow。**

这里“skill-like”有两个意思：

- 它已经具备稳定输入、阶段顺序、中间状态和 stop-line
- 但它目前仍主要存在于仓库设计与协作流程中，还没有被单独实现成产品级 skill 制品

这能解释为什么：

- 它不能继续全部压在 `documentation-workflow.md` 里
- 但也不必现在就强行抽成一个独立 capability 位之外的新 product skill

## 6. 为什么 `documentation-workflow.md` 必须收薄

如果不把这条链单独设计出来，`documentation-workflow.md` 会自然膨胀成一份“伪装成规则的 skill 手册”。

它会同时承接：

- 原则约束
- 执行顺序
- 输入类型判断
- 中间稳定层分流
- 回流正文条件
- 失败回退路径

这样一来，它既不像纯规则文档，也不像 capability 文档，还会不断把协作者私有习惯写进规则层。

因此，`documentation-workflow.md` 更适合退回到：

- 最小执行门槛
- 硬 stop-line
- 执行前检查单
- 与 Evidence / Trace / metadata-files 的接口

而把更完整的 skill-like logic 上提到 `Ingest` 和 design layer。

## 7. 这条链的最小系统图

可以把它粗略理解成：

```text
外部输入
  → Ingest 识别与降级
  → 临时层 / 元信息层 / 对象 notes 中间层
  → 证据恢复与状态判断
  → Place 决定稳定落位
  → 正文 / overview / 机制专题
  → Maintain 观察是否需要长期校准
```

这里每一步都可以由不同文档重点承接，但整条链必须被看成同一类系统动作，而不是若干互不相干的局部规定。

## 8. 与其他能力的边界

### 8.1 与 Navigate 的边界

- `Navigate`：回答“从问题出发去哪里找”
- 研究型输入吸收链：回答“材料已经拿到手后，该如何被系统吸收”

前者更早，也更偏查找；后者更偏吸收与回流。

### 8.2 与 Place 的边界

- `Ingest` / 研究型输入吸收链：回答“还在临时层或中间层时，先别急着写正文”
- `Place`：回答“已经找到稳定区域后，应停在哪一层、写成什么文件”

两者衔接紧密，但一个更偏输入吸收，一个更偏稳定落位。

### 8.3 与 Maintain 的边界

- 研究型输入吸收链：解决“新输入如何安全进入系统”
- `Maintain`：解决“系统长期运行后，哪些点会反复出错、需要上升到维护视角校准”

当某类外部输入误判不断复发时，`Maintain` 会推动这条链继续制度化。

## 9. 为什么对象目录 `notes/` 很关键

对象目录 `notes/` 的重要性在于：它让系统第一次拥有了**临时输入与正文之间的对象内稳定中间层**。

如果没有这层，协作者的默认动作常常只剩两个极端：

- 要么继续留在 `temp/`
- 要么直接润色进正文

而 `notes/` 提供的是第三种路径：

- 对象归属已经稳定
- 但正文条件仍未满足
- 于是先进对象目录内的 `source / evidence / general` 层

所以，`notes/` 不是实现细节，而是这条 workflow 能成立的重要结构前提。

## 10. 当前阶段的设计判断

当前最合理的判断是：

1. 这条链已经足够稳定，应该在 design layer 被单独说明。
2. 它目前最适合作为 **skill-like workflow** 被理解，而不是简单当作若干规则补丁。
3. 它仍以 `Ingest` 为核心能力位，不必立刻拆成新的 capability 名位。
4. 规则层应继续收薄，只保留护栏与执行门槛。
5. 如果未来这条链继续增长，并出现更明确的用户触发方式、输入模板、步骤模板和可复用执行协议，再考虑是否值得映射成更显式的 skill 制品。

## 11. 作为执行协议时，最小应包含什么

如果未来要把这条 research ingestion chain 进一步压成更显式的执行协议，最小应至少包含四块：

### 11.1 触发条件

典型触发包括：

- 用户给出 `temp/` 下的临时材料，并希望回流到主线文档。
- 协作者刚完成一次外部 AI 调研、联网检索 / 网页抓取，需要决定如何吸收结果。
- 正文或正式元信息文件中的 `Sources` 仍停留在临时路径，且上游真实来源可恢复。
- 某批对象相关材料已经不适合继续只留在 `temp/`，但又不足以直接进入正文。
- 某段已写入正文的外部结论缺乏明确 Evidence / Trace，需回退并重跑吸收链。

### 11.2 固定步骤

最小协议不应只是“记得补来源”，而应有顺序明确的步骤：

1. 识别输入类型。
2. 拆出可单独判断的 Claim。
3. 判断 Evidence 状态上限。
4. 恢复论文原文、官方文档、上游仓库、release notes 等真实来源。
5. 判断当前最合适的落位：`temp/`、`backlog.md`、`candidates.md`、`conflict.md`、对象目录 `notes/`、正文。
6. 只回流当前主题直接相关的部分，不整段搬运调研稿。
7. 改写 `Trace`，让主干文档脱离对临时入口的依赖。
8. 如果仍未满足正文条件，明确停留在更低层，而不是半回流。

### 11.3 输出契约

一轮协议执行完成后，至少应保证下面几类结果之一成立：

- 正文或正式元信息文件已改写为更直接的正式来源，不再以 `temp/` 充当证据终点。
- 如果还不能进入正文，已经明确进入 `backlog.md`、`candidates.md`、`conflict.md` 或对象目录 `notes/`。
- 如果对象归属、Evidence 状态或来源恢复仍不充分，已明确说明为什么继续停留在 `temp/`。
- 协作者能说明剩余 gap 是什么，而不是用模糊话术假装已经闭环。

### 11.4 失败回退

当协议中途发现下面任一情况时，应回退而不是硬推：

- 真实来源无法恢复。
- Evidence 状态不足以支撑正文语气。
- 对象归属仍不稳定。
- 当前主题与这批材料的关联仍过宽，继续写只会造成正文污染。

回退动作通常是：降级到 `temp/`、转入元信息文件、或先进对象目录 `notes/`，而不是继续拼接正文。

## 12. 协议样例

本节给出第一批 protocol examples。它们不是执行规则，也不是完整 case library，而是帮助人类读者快速恢复这条协议如何分支运行。

如果未来 `docs/cases/` 成为独立 case layer，本节样例应优先迁移或拆分为可复用案例对象。

### 12.1 外部调研稿先回流到对象目录 `notes/`

- **Input**：`temp/` 中有一份关于某个开源项目的外部调研稿，内容包括项目架构、源码观察、博客转述和 AI 综合判断。
- **Trigger**：材料已经明确指向一个具体项目对象，但 claim 还没有压实到足以进入对象正文。
- **Claim Split**：先拆成“项目是否提供某机制”“源码中是否存在某实现路径”“博客是否只是转述”“AI 综合判断是否只是推断”等可单独判断的 Claim。
- **Evidence Ceiling**：能追到源码或官方文档的 Claim 最高可到 `Observed` / `Verified`；只来自博客转述或 AI 综合的 Claim 只能是 `Inferred` / `Unverified`。
- **Source Recovery**：优先恢复官方文档、上游仓库、源码路径、release notes；无法恢复的一律不写成对象正文定论。
- **Placement Decision**：对象归属稳定，但正文条件不足；先进入对象目录 `notes/source.md` 或 `notes/evidence.md`，而不是直接进入 `overview.md` / `architecture.md`。
- **Trace Update**：对象正文如果暂时不改，只在相关 `notes` 中记录输入来源、claim-source 对照和待补证点。
- **Stop-line Applied**：不把临时调研稿当成对象正文草稿直接润色；不把 `notes/` 当成 `temp/` 镜像备份。
- **Output**：形成对象内稳定中间层；后续只有已恢复真实来源、Evidence 状态足够、且与当前专题直接相关的 Claim 才继续回流正文。

### 12.2 正文 `Sources` 停在临时路径时，恢复真实来源后回填正文

- **Input**：某篇正文或正式元信息文件的 `Sources` 仍写着 `temp/web-search/...`，但该临时材料中保存了论文、官方文档或上游仓库 URL。
- **Trigger**：主干文档删除临时文件后会失去证据链，且真实上游来源可恢复。
- **Claim Split**：先识别正文中哪些结论实际依赖这份临时材料，逐条拆成可核验 Claim。
- **Evidence Ceiling**：能被论文原文、官方文档、源码或 release notes 直接支撑的 Claim 可升格；不能恢复来源的 Claim 降级或移出正文。
- **Source Recovery**：把 `Sources` 从临时路径改成论文、官方文档、上游仓库、release notes 或其他公开正式来源。
- **Placement Decision**：若 Claim 已被真实来源支撑，可继续留在正文；若只剩外部调研综合判断，应降级到 `backlog.md`、`candidates.md`、`conflict.md` 或对象目录 `notes/`。
- **Trace Update**：`Trace` 改写为“外部调研线索已回收到真实上游来源”，而不是要求读者回到临时文件还原论证。
- **Stop-line Applied**：不只是删除 `temp/` 路径；必须恢复证据终点，避免临时文件删除后证据链断裂。
- **Output**：正文或元信息文件不再依赖 `temp/` 作为正式来源终点；无法恢复真实来源的部分被降级或移出正文。

### 12.3 完整外部材料仍可能只进入 `backlog.md` / `candidates.md`

- **Input**：外部 AI 给出一组论文、项目、产品和机制总结，看起来像一份完整综述。
- **Trigger**：材料覆盖面广，但尚未确定哪些结论可作为正文定论，哪些对象值得深入研究。
- **Claim Split**：把材料拆成“待研究问题”“候选对象”“初步比较”“未验证趋势判断”等不同类型。
- **Evidence Ceiling**：大部分综合性判断先保持 `Inferred` / `Unverified`；只有能回到论文、官方文档或源码的一小部分可进入更高状态。
- **Source Recovery**：优先恢复每个候选对象的真实来源；如果只能恢复对象名而不能支撑结论，不写成正文比较。
- **Placement Decision**：稳定的问题进入 `backlog.md`；稳定的对象进入 `candidates.md`；冲突口径进入 `conflict.md`；正文暂不接收大段综述。
- **Trace Update**：元信息文件可以说明这批输入来自一轮外部调研，但正式来源表达仍应尽量回到具体论文、官方文档或对象来源。
- **Stop-line Applied**：不因为材料完整、结构漂亮，就把它当成正文候选；先把问题、对象和冲突拆开。
- **Output**：形成可继续研究的问题队列和对象队列；正文只在后续证据链和落位判断成熟后接收局部结论。

## 13. 未来可能的演化方向

未来这条链可能有三种演化路径：

- **继续作为 capability chain 存在**：`Ingest` + `Place` + `Maintain` 协同即可，不额外抽象。
- **形成更显式的 workflow / protocol 文档层**：在 design layer 继续补强，使其更接近可执行协议，但仍不变成 product skill。
- **演化成可操作 skill**：当触发条件、步骤模板、输入输出契约足够稳定时，再映射到更正式的 skill 设计或实现。

当前最稳的是第一种和第二种的组合。

## 14. 本文的边界

本文不负责：

- 规定具体的 Evidence 字段怎么写
- 规定 `backlog.md` / `candidates.md` / `conflict.md` 的细节格式
- 代替 `documentation-workflow.md` 给出执行清单
- 代替 `research-artifacts.md` 规定对象目录内部结构
- 决定现在就创建新的 product skill

它只负责把一个此前容易被误写成“规则习惯”的事实说清楚：

> **从外部临时材料到稳定知识层的处理，是一条应被单独建模的研究型输入吸收链；它在当前系统中最适合作为由 `Ingest` 主导的 skill-like workflow 被理解。**

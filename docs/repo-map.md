# Repository Map

本文件回答：**这个仓库里有哪些不同功能的文档，它们分别是干什么的，彼此是什么关系。**

它不是贡献规则，也不是目录导航；它的目标是帮助任何读者先建立一个最小心智：看到一份文档时，先知道自己面对的到底是哪一类内容。

## 1. 为什么要先区分文档类型

这个仓库里的 Markdown 文档并不是同一种东西。

如果不先区分文档类型，最常见的误读通常是：

- 把治理文档当成领域知识正文
- 把元信息文档当成主题主文
- 把研究辅助材料当成可直接引用的稳定结论
- 把 `temp/` 里的临时输入当成正式知识
- 把对象研究文档写成跨文档消费说明，或把主题研究文档写成对象流水

所以，阅读和写作都应先问一句：**这份文档的组织功能是什么？**

## 2. 仓库中的五类文档

如果先不看具体规则，可以把这五类文档先看成一张很粗的功能分组图：

```text
仓库文档
├── 治理文档
│   ├── docs/contributing/
│   ├── docs/capabilities/
│   └── docs/design/
├── 知识正文文档
│   ├── 主题专题文档
│   ├── 对象研究文档
│   ├── 单篇论文 / 报告研究文档
│   └── 案例机制文档
├── 元信息文档
│   ├── README.md
│   ├── index.md
│   ├── overview.md / landscape.md
│   └── backlog.md / candidates.md / roadmap.md / conflict.md
├── 研究辅助文档
│   ├── notes/
│   ├── notes.md
│   ├── source.md / evidence.md
│   └── reflections.md 等
└── 临时输入文档
    └── temp/
```

这不是精确目录树，而是一张按组织功能分组的阅读图。

### 2.1 治理文档

这类文档解释仓库如何协作、如何组织、如何判断与处理文档，而不是直接承接具体领域知识。

典型位置：

- `docs/contributing/`
- `docs/capabilities/`
- `docs/design/`

它们分别回答：

- `contributing/`：规则、流程、文档治理、Evidence 与 Traceability
- `capabilities/`：这套系统提供哪些公共能力、能力之间如何衔接
- `design/`：为什么要这样分层、当前系统设计如何演化

如果只是想理解某个机器学习、Agent、系统机制本身，这类文档通常不是终点。

### 2.2 知识正文文档

这类文档直接承接知识内容，目标是解释一个主题、对象、机制、论文或报告本身。

常见形态：

- 主题专题文档
- 对象研究文档
- 单篇论文 / 报告研究文档
- 案例机制文档
- 专题主文，如 `architecture.md`、机制说明文、对象专题正文

这类文档更强调：

- 原子性
- 独立性
- 可单独阅读
- 直接回答“它本身是什么”

它们可以引用其他文档，但不应把“当前仓库如何消费它”写成自己的主体职责。

### 2.3 元信息文档

这类文档不是知识正文，而是帮助读者和维护者完成说明、导航、理解、结构研究、推进、选题、路径、冲突校验等认知任务。

典型文件：

- `README.md`
- `index.md`
- `overview.md`
- `landscape.md`
- `backlog.md`
- `candidates.md`
- `roadmap.md`
- `conflict.md`

它们回答的问题不是同一个，因此不应机械补齐，也不应随意混写。详细边界见 [`contributing/metadata-files.md`](./contributing/metadata-files.md)。

对普通读者来说，可以先这样理解：

- `README.md`：这是什么地方
- `index.md`：这里下面有什么、该去哪找
- `overview.md`：如果只读一篇，先理解什么
- `landscape.md`：这个主题内部怎么切、各部分怎么衔接
- `backlog.md`：还有什么没覆盖
- `candidates.md`：接下来值得研究哪些对象
- `roadmap.md`：从哪开始、按什么顺序走
- `conflict.md`：哪些口径还冲突

### 2.4 研究辅助文档

这类文档服务研究过程、证据追踪和后续维护，不要求像正文一样完整叙事。

典型位置与形态：

- `notes/`
- `notes.md`
- `source.md`
- `evidence.md`
- `reflections.md`
- 其他以过程、摘录、对照、暂存分析为主的辅助文件

它们常承接：

- 源码笔记
- 长引用
- 证据对照
- 失败搜索
- 研究脚手架
- 阶段性观察与假说

这类文档很重要，但默认不是给第一次进入该主题的普通读者看的主入口。

### 2.5 临时输入文档

这类文档是上游输入，不是正式知识终点。

典型位置：

- `temp/`

常见内容：

- 外部 AI 调研稿
- 网页抓取结果
- 临时整理稿
- 尚未完成归属判断的输入材料

它们的价值在于：

- 暂存
- 分流
- 等待回流

而不在于直接成为稳定正文。详细流程见 [`contributing/documentation-workflow.md`](./contributing/documentation-workflow.md)。

## 3. 五类文档之间的关系

可以先用一句话理解它们的关系：

- 治理文档：约束整个仓库如何写、如何组织、如何协作
- 元信息文档：帮助读者找到、理解、推进知识内容
- 知识正文文档：承接稳定知识本身
- 研究辅助文档：支撑正文形成与后续维护
- 临时输入文档：作为尚未完成整理与回流的上游材料

如果画成一条很粗的流向，大致可以这样理解：

```text
临时输入文档 -> 研究辅助文档 / 元信息文档 -> 知识正文文档
                 ^
                 |
              治理文档提供规则、能力与设计约束
```

这不是严格流水线，而是一个帮助理解的最小模型。

如果更关心“读者阅读时怎么在不同文档之间跳转”，可以再看成下面这张关系图：

```text
治理文档 ------> 规定边界、写法、创建条件

元信息文档 ----> 帮你判断去哪里读、从哪开始、哪里有缺口/对象/冲突
      |
      v
知识正文文档 --> 承接主题、对象、论文、机制本身
      ^
      |
研究辅助文档 --> 为正文提供过程材料、证据摘录、对照与维护支撑

临时输入文档 --> 作为尚未整理完成的上游输入，等待分流和回流
```

前一张图偏“按组织功能分组”，这一张偏“按阅读和研究过程理解关系”。

## 4. 什么文档更适合独立，什么文档更适合引用别人

这份文档只给一个最小判断，不展开细则。

更适合保持强独立性的通常是：

- 单篇论文研究文档
- 单个对象的案例研究正文
- 机制专题正文
- 面向读者的主文型 `overview.md`

这里的 `overview.md` 指承担主题主线理解的一篇相对独立主文；它虽然属于元信息文件体系，但在写法上通常应接近可单篇成立的知识正文。

更适合承担路由、摘要引用、跨文档关联的通常是：

- `README.md`
- `index.md`
- 主题研究文档
- 应用层对基础对象的消费文档

默认原则是：

- 对象文档负责对象本身
- 主题文档负责抽象与引申
- 元信息文档负责说明、导航、推进与校验
- 辅助材料文档负责研究过程
- 临时输入文档负责暂存与等待回流

## 5. 最常见的误读

下面这些都是仓库里最容易反复出现的混淆：

- 把 `docs/contributing/` 当成知识正文目录
- 把 `README.md` 写成大而全导航页或综述页
- 把 `candidates.md` 写成压缩研究正文
- 把单篇论文文档写成“当前主线该怎么消费它”的说明
- 把 `notes/` 里的研究过程材料当成正式结论引用
- 把 `temp/` 里的输入材料当成正式来源终点
- 把对象研究、主题研究、材料整理三种东西混在同一份文档里

看到这些现象时，通常说明问题不是“写得不够完整”，而是文档类型已经混了。

## 6. 第一次进入仓库时怎么用这张图

如果你是第一次进入这个仓库，可以按自己的目标选择起点：

- 想理解仓库怎样协作、怎样治理文档：先看治理文档
- 想学习某个主题或对象本身：先找知识正文文档
- 想知道一个目录下面有什么、该去哪找：先看元信息文档
- 想看研究过程、证据整理或问题树：看研究辅助文档
- 想知道最近收进来的外部材料还没整理到哪：看临时输入文档

如果你拿不准自己正在看的文档属于哪类，先回到本文件重新判断，再决定下一跳。

# Repo Naming Checklist

这份文档用于沉淀当前仓库命名讨论中反复出现、且值得长期保留的判断框架。

它不是候选名列表，也不是最终结论；它的作用是把命名过程中真正有价值的思考点、关注点和评价标准留下来，避免后续继续讨论时反复回到同样的问题。 

## 0. 目录内文件分工

本目录当前几份文件的定位如下：

- `README.md`：当前主文档。只保留本地采纳后的可执行 checklist，不继承前面几轮材料里的隐性方向偏好。
- `candidates.md`：当前候选短名单，用于保存还值得继续看的名字，以及已经明确降权的词与方向。
- `reference/`：过程材料目录，保留命名讨论过程中的外部输入、发散稿与阶段性中间稿；默认不是当前主结论，也不直接替代这里的 checklist。
- `reference/1.md`：第一轮自由发散，主要用于打开命名想象空间，包含较强的类比性和启发性，不适合直接作为决策依据。
- `reference/2.md`：第二轮扩样本与风格归纳，开始区分“外部参照”和“风格判断”，但仍以发散为主。
- `reference/3.md`：第三轮补证据与方向池，重点是扩大样本、提高可回查性，并初步形成命名方向地图。
- `reference/4.md`：第四轮收束，转向“命名原则 / 排除原则 / 重复度与撞名风险框架”，适合作为外部输入的结构化中间稿。

使用顺序上，优先读 `README.md`；需要看当前还剩哪些候选时，再读 `candidates.md`；只有在需要回看某个方向是如何被提出、修正、再收束时，再回查 `reference/` 下的过程稿。


## 1. 命名问题的核心

这轮讨论里，最稳定、最值得保留的认识不是某一个具体候选名，而是：**仓库命名不能只看字面边界，也不能只看传播感觉。**

如果只追求技术分类上的准确，很容易得到：

- 技术上没错
- 传播上很弱
- 与大量同类仓库高度同质化

如果只追求气质或品牌感，又容易得到：

- 吸引力有了
- 但边界发虚
- 需要过度依赖 README 解释

因此，命名过程中反复需要同时考虑的关键矛盾是：

- **边界准确性**
- **传播可区分性**

二者都重要，不能只保一个。

## 2. 正向原则

一个更合适的仓库名，应尽量满足以下条件：

### 2.1 不把仓库缩回单一技术分类

名字不能把仓库误导成：

- 传统 ML 笔记仓
- 单纯 LLM 论文收集仓
- 纯工程系统设计仓

它需要允许仓库继续容纳：

- 大模型底层技术
- 大模型应用
- RAG / Agentic / infra
- 多模态 / CV 的适度扩展
- 跨学科研究
- 知识沉淀与研究循环

### 2.2 传达“活的研究仓库”气质

名字最好能让人感受到这不是一个静态资料归档，而是一个持续推进中的公开研究仓库。

因此，比起完成时、静态感很强的词，更优先考虑能承载以下气质的方向：

- 持续跟踪
- 结构化组织
- 全景观察
- 长期演化
- 研究推进

### 2.3 给 README 留解释空间，但不能完全靠 README 救

名字不需要把仓库全部内容解释完，但至少要：

- 不明显误导
- 有清晰气质
- 值得被进一步点开

也就是说，名字可以保留一定抽象度，但不能抽象到脱离内容边界。

### 2.4 优先考虑公开传播时的识别度

仓库名不是内部分类标签，而是公开入口。

因此它需要具备：

- 一眼能记住
- 不容易和别的 AI 仓库混淆
- 被别人提起时有唯一识别感

## 3. 排除原则

以下类型，默认优先排除。

### 3.1 教程 / 入门仓气质

例如这类方向通常不适合：

- `*-tutorial`
- `*-for-beginners`
- `*-starter`
- `*-101`

原因：会把仓库误导成入门材料，而不是研究型知识仓库。

### 3.2 个人学习笔记气质

例如：

- `my-*`
- `*-notes`
- `learning-*`
- `*-diary`

原因：会让仓库显得过于私人、辅助、未公共化。

### 3.3 静态资源收集 / 归档气质

例如：

- `awesome-*`
- `*-resources`
- `*-archive`
- `*-knowledge-base`

原因：这些名字天然更像资源列表、资料仓或完成态文档库，不像持续研究中的项目。

### 3.4 过于正式的机构 / 组织气质

例如：

- `*-institute`
- `*-foundation`
- `*-center`

原因：个人公开仓库使用这类名字，容易出现“名过其实”的落差。

### 3.5 纯技术分类修补

例如只是在这些词上继续线性修补：

- `machine-learning`
- `llm-research`
- `ai-notes`
- `llm-systems`
- `ai-systems`

这些名字的问题不一定是“错误”，而是它们太容易落入：

- 过度准确但不出众
- 技术分类感强
- 高同质化
- 传播识别度弱

### 3.6 编程语言默认用词 / 基础技术占位词

例如这类方向通常不适合：

- 编程语言常见基础词
- 运行时 / 并发 / 容器 / 数据结构强占位词
- 第一眼更像类型名、模块名、库名、关键字或底层概念名的词

这类词的问题不只是“常见”，而是会优先触发程序员已有的技术语义，很难再为仓库本身建立独立认知。

## 4. 重复度 / 撞名风险检查

这是后续筛选名字时必须显式执行的一步。

一个名字即使方向正确，只要重复度太高，也应谨慎甚至放弃。

### 4.1 精确撞名

先检查：

- GitHub 上是否已有完全同名项目
- 是否已有近同名高星仓库
- 是否已有相同词序的知名项目

如果答案是“有，而且已经占据心智”，风险很高。

这里要特别注意：**高关注度完全同名仓库**的优先级，高于单纯的总匹配数量。

也就是说：

- 一个词即使总命中量不算高
- 只要已有高 star 的完全同名仓库
- 或已有大公司 / 知名组织托管的强相关项目

也应优先视为硬风险，而不是继续把它当作安全候选。

### 4.1.5 强占位项目风险

除了看总命中量，还要额外检查：

- 搜索结果前几条是否已经被某个高关注度仓库占据
- 是否已有大公司、知名组织或本领域核心社区在使用这个名字
- 目标受众第一眼是否大概率先想到那个现有项目，而不是你的仓库

如果答案是“是”，那么这个名字的问题就不再只是重复度高，而是已经存在较强的认知占位风险。

这类风险通常比普通的数量拥挤更严重，因为它会直接带来认知混淆。

同时也要注意：这类风险并不必然构成一票否决。对于个人学习 / 研究仓库，更合理的做法通常不是机械追求“名字主权”，而是判断这种占位风险是否仍在可接受范围内，是否会实质影响目标读者的理解与进入意愿。

### 4.2 模板拥挤度

要警惕高频模板，例如：

- `ai-*`
- `*-lab`
- `*-research`
- `*-notes`
- `*-atlas`
- `*-guide`
- `open-*`

即使单个名字没完全撞上，只要模板本身过于拥挤，也会削弱识别度。

### 4.3 搜索可区分性

要看这个名字在：

- GitHub
- Google
- 社交平台

上是否容易被大量泛结果淹没。

如果一个名字搜出来几乎全是无关结果，或者全是同类泛用项目，那它的公开传播效率会很差。

### 4.4 记忆唯一性

要问自己：

- 听过一次后，能不能复述出来
- 一周后，还能不能把它和别的仓库分开
- 别人转述时，会不会自动说成另一个常见名字

如果名字很顺但记不住，本质上还是弱名字。

### 4.5 语义占位风险

还要检查这个词是否已经被某个强势项目、产品或领域语义强绑定。

如果某个名字一出现，别人首先想到的是别的产品、组织，或某个编程语言里的默认技术概念，那它就很难再为你的仓库建立独立认知。

即使 GitHub 命中量不算高，只要这个词天然属于程序员默认语义池，也应优先降权或直接排除。

## 5. 方向评估时该看什么

后续如果继续比较命名方向，不要先问“哪个名字最好”，而应先问：

### 5.1 这个方向在表达什么

它更偏向：

- 技术边界
- 研究动态
- 知识结构
- 全景观察
- 品牌气质

### 5.2 这个方向最容易造成什么误解

例如：

- 像教程仓
- 像研究机构
- 像媒体品牌
- 像基础设施工具
- 像静态资源站

### 5.3 这个方向的重复度高不高

即使方向气质对，也要看它是否已经成为 GitHub 上的高频套路。

### 5.4 这个方向是否撑得住仓库现状

越抽象、越有品牌感的名字，对 README 和仓库内容的要求越高。

如果名字太大、内容承接不住，会放大落差。

## 6. 命名时的判断顺序

后续如果继续推进改名，建议按这个顺序判断，而不是一上来就直接列词：

1. **先定风格方向**
   - 先判断更想要哪种气质，而不是直接列候选名。
2. **再定核心词边界**
   - 再判断名字里是否需要出现 `AI`、`intelligence` 或其他核心词。
3. **再做双层重复度检查**
   - 先看总匹配数量、模板拥挤度、是否属于编程语言默认语义池。
   - 再看是否存在高 star 完全同名仓库、大公司 / 知名组织强占位项目，以及搜索前几屏是否已被锁定语义。
4. **最后才看具体名字**
   - 名字是最后一步，不是第一步。

## 7. 当前阶段的使用方式

这份 checklist 的作用是：

- 帮助排除明显错误方向
- 帮助评估某个方向是否值得继续
- 防止被“看起来合理但过于常见”的名字带偏

它暂时**不负责**：

- 给出最终仓库名
- 预设唯一正确方向
- 强行收敛到少数具体候选

在最终选名之前，应优先用这份 checklist 做筛选，而不是继续无约束发散。

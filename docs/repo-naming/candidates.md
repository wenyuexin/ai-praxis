# Repo Naming Candidates

这份文档用于保存**当前值得继续看的候选名短名单**。

它不是最终结论，也不是单个名字的详细评测结果；它的作用是先把有潜力的名字收在一起，供后续继续比较、筛选和收敛。

## 使用方式

- 这里只保留当前讨论中仍有参考价值的名字。
- 第一组是当前主候选，第三组是新观察组，第二组只保留历史讨论中还值得回看的旧候选。
- 不再新增候选名，除非出现明显强于现有候选、且通过双层重复度检查的新名字。
- 不在这里做完整评测或展开长篇论证。
- 如果某个名字在下一轮中被确认问题很大，应直接从这里移除，而不是无限保留。
- 当前这份短名单的筛选标准，已经从“方向是否正确”调整为：**是否既不太假、又比安全词更有吸引力**。
- 已经查过 GitHub 搜索命中量的词，直接写在文档里；如果某个词命中量已经明显过高，默认降权，避免后续重复提起。

## 第一组：当前最值得继续看的名字

当前排序倾向：`ai-praxis` > `praxis` > `aperture` > `vantage`。

基于当前目标的修正：这个仓库本质上仍是个人学习 / 研究仓库，希望吸引同领域学习者；因此，强占位风险仍需检查，但不再单独作为硬否决项。当前更看重领域可识别性、入口吸引力，以及是否能让一般开发者更容易点进来。

这一组用于保留当前最值得继续比较的名字，不再承担开放式扩词池的作用。

### `ai-praxis`

- **GitHub 搜索命中量**：组合未检查；`ai` 约 `5.2M`，`praxis` 约 `4.9k`
- **为什么进短名单**：它用 `ai` 直接点明研究领域，用 `praxis` 保留“研究—实践回流”的核心理念，属于“领域 + 理念”的组合。
- **当前优点**：比单独 `praxis` 更直观，也更容易让同领域学习者第一眼知道仓库大致在做什么。
- **当前观察**：在当前目标下，它属于最值得优先比较的候选之一。
- **当前风险**：仍然带有 `ai-*` 模板感，也可能让 `praxis` 的独特性被 `ai` 前缀部分削弱。

### `praxis`

- **GitHub 搜索命中量**：约 `4.9k`
- **为什么进短名单**：这个词天然带“理论与实践结合”的意味，和你仓库“双核心：底层技术 + 应用”高度相关。
- **当前优点**：它最能同时承接研究推进、知识回流、方法体系和长期主义，也比 `forge`、`atlas` 这类高拥挤词更稀缺。
- **当前观察**：它仍然是理念表达最强的候选之一，但入口门槛相对更高。
- **当前风险**：哲学背景较强，不是所有工程师都会立即熟悉；也要检查前几页结果是否已有强语义占位。

### `aperture`

- **GitHub 搜索命中量**：约 `4.2k`
- **为什么进短名单**：这个词有“观察窗口 / 打开视野 / 控制进入的信息量”的意味，比较适合研究、筛选和组织复杂技术主题。
- **当前优点**：命中量很干净，词感也比 `observatory` 更轻，比 `compass`、`lighthouse` 更不模板化。
- **当前观察**：它仍然是公开入口气质较强的一条路线，但当前优先级低于前两者。
- **当前风险**：偏观察与视角隐喻，更像“怎么进入这个仓库”，不如 `praxis` 那样能表达仓库长期存在的理由。

### `vantage`

- **GitHub 搜索命中量**：约 `5.9k`
- **为什么进短名单**：它更像“观察位置 / 认知视角”，能承接你这个仓库既研究内容、也研究研究方法的气质。
- **当前优点**：重复度可接受，也没有明显落入编程语言默认用词或常见命名模板。
- **当前观察**：它更像平衡型候选，但张力弱于当前排在前面的几条路线。
- **当前风险**：偏品牌词、偏英文气质，而且既没有 `praxis` 那么准，也没有 `aperture` 那么像公开入口。

## 第二组：历史候选，保留参考

### `ai-compass`

- **GitHub 搜索命中量**：`compass` 约 `34k`，`ai` 约 `5.2M`
- **为什么保留**：比纯技术分类词更像公开入口，也有一定方向感。
- **为什么暂不优先**：`compass` 本身不算最低拥挤词，再叠加 `ai-*` 这个超高拥挤模板后，整体重复度明显偏高。

### `ai-observatory`

- **GitHub 搜索命中量**：`observatory` 约 `5k`，`ai` 约 `5.2M`
- **为什么保留**：`observatory` 单词本身稀缺度其实不错，也贴“持续跟踪 + 观察 + 研究推进”的气质。
- **为什么暂不优先**：问题不在 `observatory`，而在 `ai-*` 模板过度拥挤；同时这个方向仍偏“正确词”，记忆点不够强。

### `intelligence-compass`

- **GitHub 搜索命中量**：`intelligence` 约 `269k`，`compass` 约 `34k`
- **为什么保留**：抽象度足够容纳跨学科和未来扩展。
- **为什么暂不优先**：两个组成词都不算稀缺，叠加后仍然偏模板化，且更像分析框架名，不像自然主名。

### `frontier-intelligence`

- **GitHub 搜索命中量**：`frontier` 约 `13k`，`intelligence` 约 `269k`
- **为什么保留**：`frontier` 这部分能表达前沿跟踪和研究气质。
- **为什么暂不优先**：`intelligence` 过于通用，整体仍像内容栏目或研究计划名，缺少自然名字感。

### `ai-lighthouse`

- **GitHub 搜索命中量**：`lighthouse` 约 `24k`，`ai` 约 `5.2M`
- **为什么保留**：隐喻明确，方向感和传播性都不错。
- **为什么暂不优先**：`lighthouse` 不算极高拥挤，但 `ai-*` 模板本身过于泛滥，整体仍偏常见命名套路。

### `loom`

- **GitHub 搜索命中量**：约 `10k`
- **为什么保留**：它确实能表达“把分散线索编织成结构”这层研究气质。
- **为什么暂不优先**：你在其他开源仓库里已经用过“编织”意象，再用会让不同仓库之间的气质重复，因此这次直接降权。

### `confluence`

- **GitHub 搜索命中量**：约 `8k`
- **为什么保留**：它的“交汇 / 汇流”语义仍然贴你的仓库边界。
- **为什么暂不优先**：问题不是语义不对，而是主观吸引力一般，更像“方向正确词”，不够值得长期背负。

### `signal-and-scale`

- **GitHub 搜索命中量**：未检查
- **为什么保留**：有研究感，也暗示了底层技术与应用扩展之间的张力。
- **为什么暂不优先**：名字和仓库内容的直观关系还不够强，第一眼识别成本较高。

### `thinking-in-public`

- **GitHub 搜索命中量**：未检查
- **为什么保留**：很好地体现了“公开研究过程”这件事。
- **为什么暂不优先**：更像方法论口号或过程宣言，而不像仓库主名。

### `foundry`

- **GitHub 搜索命中量**：约 `41k`
- **为什么保留**：仍然比 `forge` 更稳，也确实有“持续产出、锻造体系”的感觉。
- **为什么暂不优先**：命中量已经偏高，产品 / 公司 / 工具语义占位明显，不适合再放在第一优先组。

### `prism`

- **GitHub 搜索命中量**：约 `67.9k`
- **为什么保留**：它能表达“把复杂输入拆成不同视角来理解”，意象本身并不差。
- **为什么暂不优先**：命中量已经明显偏高，不值得作为强候选继续投入。

### `horizon`

- **GitHub 搜索命中量**：约 `44.2k`
- **为什么保留**：它能表达前沿、视野和持续追踪的感觉。
- **为什么暂不优先**：命中量已经不低，而且更偏愿景词，不够值得作为优先候选继续推进。

### `kernel`

- **GitHub 搜索命中量**：约 `148k`
- **为什么保留**：工程语义强，和“核心、底层、承载”这条解释链仍然成立。
- **为什么暂不优先**：命中量过高，且语义占位太强，已经更像参考方向而不是现实候选。

### `signal`

- **GitHub 搜索命中量**：约 `203k`
- **为什么保留**：它的短促、清晰和“从噪声里找有效信息”的感觉仍然很有吸引力。
- **为什么暂不优先**：命中量过高，撞名风险和通用语义拥挤度都已经明显超线。

## 第三组：值得补充观察的新词

这一组里，`cadence` 和 `contour` 当前更值得继续观察；`waypoint` 和 `throughline` 暂时更适合作为补充观察项。

### `cadence`

- **GitHub 搜索命中量**：约 `6.3k`
- **为什么进观察组**：它能表达持续推进、节奏和长期演化，和“活的研究仓库”这层气质有一定贴合。
- **当前观察**：它是第三组里较值得继续看的词之一，但还没有明显强到足以进入第一组。
- **当前风险**：更像过程气质词或工作节律词，不太像能独立承载仓库边界的主名。

### `waypoint`

- **GitHub 搜索命中量**：约 `6.2k`
- **为什么进观察组**：它有路径感、阶段感和探索感，比 `compass` 更具体，也没那么像常见模板词。
- **当前风险**：容易偏向“导航 / 路线节点”语义，需要看是否会把仓库误导成路径工具或学习路线仓。

### `contour`

- **GitHub 搜索命中量**：约 `8.4k`
- **为什么进观察组**：它有“勾勒轮廓、刻画复杂对象边界”的感觉，比较贴结构化研究和问题建模。
- **当前观察**：它目前仍在观察范围内，但相对不像最值得优先推进的词。
- **当前风险**：语义相对冷静，也更像分析动作词，不一定有足够强的公开吸引力。

### `throughline`

- **GitHub 搜索命中量**：约 `107`
- **为什么进观察组**：它非常贴“主线、贯穿、研究脉络”这层意味，而且重复度极低。
- **当前风险**：词偏长，也更像表达口号或方法论短语，不一定像自然的仓库主名。

## 已明确降权的词与方向

以下词或方向已经有较明确的降权依据，后续不应再默认当作优先候选反复提出。

### 已查过 GitHub 搜索命中量的高拥挤词

- `signal`：约 `203k`
  - **降权原因**：命中量极高，通用词拥挤度过强，后续默认不再作为优先候选。
- `kernel`：约 `148k`
  - **降权原因**：虽然工程语义强、技术感足，但命中量极高，且语义占位过重。
- `forge`：约 `110k`
  - **降权原因**：虽然有动作感和吸引力，但命中量过高，通用词拥挤度明显，后续默认不再作为优先候选。
- `atlas`：约 `84k`
  - **降权原因**：不是“稍微常见”，而是已经进入明显高拥挤词池；方向可能对，但词本身已过于模板化、审美疲劳。
- `foundry`：约 `41k`
  - **降权原因**：虽然比 `forge` 略好，但命中量仍明显偏高，而且语义占位风险较强。

### 当前明确降权的方向

这些方向不是一定错误，但当前优先级明显下降：

- 所有 `*atlas`
- 常规 `*observatory`
- 常规 `*guide`
- 常规 `ai-*`
- `ai-systems`
- `llm-systems`
- `ai-research`
- `llm-research`
- 所有 `*-notes`
- 常规 `*-lab`
- 常规 `open-*` 技术分类修补

原因主要是：

- 有些词方向正确，但吸引力不足
- 有些词已经开始模板化、审美疲劳
- 有些词太硬、太像技术分类或组织名称
- 很容易落入“技术上合理，但没有记忆点”的问题
- 部分词已经有明确的高命中量依据，不值得继续反复讨论

## 下一步建议

这份文档后续不再承担开放式扩词池的任务，而是保留当前最值得继续比较的候选，以及这轮命名讨论中已经形成的判断重点。

后续如果继续推进命名讨论，优先回看：

- 是否能表达个人学习 / 研究仓库的真实定位
- 是否对同领域学习者有足够吸引力
- 是否存在高关注度完全同名仓库或强占位项目
- 是否需要用领域词来降低理解门槛
- 是否愿意长期背负这个名字

这轮讨论沉淀下来的价值，不只是当前候选排序本身，也包括：

- 不能只看字面语义，还要看传播入口
- 不能只看总命中量，还要看强占位项目
- 对个人学习仓库来说，强占位风险需要评估，但不必机械追求名字主权
- 组合词不必默认排除，但要区分“领域 + 理念”与普通模板拼接

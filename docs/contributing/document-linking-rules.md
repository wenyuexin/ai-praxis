# 文档引用与关联规则

本文件定义仓库内不同类型文档之间如何建立引用关系、如何保持边界，以及什么时候应链接、什么时候应把内容留在原文档或转移到其他承接位。

相关入口：[`CONTRIBUTING.md`](../../CONTRIBUTING.md) / [`docs/contributing/README.md`](./README.md) / [`仓库文档功能地图`](../repo-map.md) / [`元信息文件模型`](./metadata-files.md) / [`研究产物组织规则`](./research-artifacts.md) / [`README 规则`](./readme-rules.md)。

## 1. 适用范围

本文件回答的是：**文档之间应如何关联。**

它不负责：

- 元信息文件各自的职责边界：见 [`metadata-files.md`](./metadata-files.md)
- 案例研究内部的正文 / 机制专题 / 辅助材料组织：见 [`research-artifacts.md`](./research-artifacts.md)
- Evidence 状态判断与 Traceability 字段细则：见 [`evidence-rules.md`](./evidence-rules.md) 与 [`traceability-rules.md`](./traceability-rules.md)
- 材料进入主线前的分流与回流流程：见 [`documentation-workflow.md`](./documentation-workflow.md)

## 2. 核心原则

- **对象文档优先保持原子性与独立性。** 不要把“当前主线怎么消费它”写成对象文档的主体职责。
- **主题文档负责消费、抽象、比较与引申。** 当信息已经超出对象本身，进入跨对象判断时，应转到主题文档承接。
- **元信息文档、研究辅助材料与临时输入的职责边界，不在本文重述。** 分别以 [`metadata-files.md`](./metadata-files.md)、[`research-artifacts.md`](./research-artifacts.md) 与 [`documentation-workflow.md`](./documentation-workflow.md) 为准；本文只补它们与正文、主题文档、对象文档之间的关联方式。

## 3. 四种常见关联方式

### 3.1 Markdown 链接引用

用于把读者导向另一篇文档。

适合场景：

- 入口分流
- 延伸阅读
- 指向对象目录、案例目录或规则文件

边界提醒：

- 链接引用解决“去哪看”，不自动等于“当前文档已消费并吸收了对方内容”。
- 不要因为加了链接，就省略当前文档本应承担的关键判断。

### 3.2 内容引用

用于在当前文档中使用另一篇文档中的结论、定义、观察或边界。

适合场景：

- 主题文档消费对象文档
- 应用层文档消费基础层文档
- 比较文档引用单对象研究结论

边界提醒：

- 内容引用应服务当前文档自己的主问题，而不是把对方文档改写成自己的附录。
- 被引用对象的完整消费方式、跨对象比较与进一步引申，应留在当前主题文档，而不是回写到原对象文档中。

### 3.3 方向性说明

用于告诉读者“想看哪类问题，接下来去哪里”。

适合场景：

- README 中的入口说明
- index 中的按问题找入口
- 根 README 中的仓库级分流

边界提醒：

- 方向性说明是入口层提示，不是正文分析。
- 不要把方向性说明膨胀成大量重复导航文案。

### 3.4 维护链路字段

例如 `Source / Decision / Placement / Gap`、`Trace`、`Relation to Topics`。

适合场景：

- 说明材料来源与当前落位
- 保留跨文档的维护链路
- 记录为什么内容放在这里而不是那里

边界提醒：

- 维护链路字段不等于读者向正文引用。
- 不要让 Trace 字段取代正文中本应出现的核心判断。

## 4. 默认方向

默认应按下面的方向建立关系：

- **对象文档 -> 向内负责对象本身**
- **主题文档 -> 向外负责跨对象组织、比较与引申**
- **README / index 的职责边界，以 [`metadata-files.md`](./metadata-files.md) 为准；在本文里，它们只讨论自己如何链接、分流和指向其他文档。**
- **notes / temp 的组织边界，以 [`research-artifacts.md`](./research-artifacts.md) 与 [`documentation-workflow.md`](./documentation-workflow.md) 为准；在本文里，它们只讨论自己如何支撑正文与引用链路。**

默认流向：

```text
对象文档 -> 被主题文档消费
主题文档 -> 可以链接或引用对象文档
README / index -> 通过说明、分流与导航把读者送到对的位置
notes / temp -> 为正文和主题文档提供上游材料
```

**Stop-line：不要把对象文档改写成“别人如何使用它”的说明文。**
**Stop-line：不要把主题文档应承担的跨对象比较，回写进单对象文档。**

## 5. 什么时候允许双向关联

默认不鼓励强双向耦合，但允许少量轻量互链。

允许：

- 对象文档链接回引用它的主题文档，作为“相关主题 / 延伸阅读”
- 主题文档链接到其主要证据来源或代表对象

不鼓励：

- 双方互相写大段“对方如何使用我 / 我如何被对方消费”的说明
- 因为想保留互相发现能力，就在两边重复一整套相同内容

**Stop-line：双向可发现，不等于双向重写内容。**

## 6. 最小判断流程

遇到跨文档引用问题时，按下面顺序判断：

1. 先判断当前文档是什么类型：对象、主题、元信息、辅助材料，还是临时输入。
2. 再判断当前信息是在回答“对象本身是什么”，还是“别人如何使用它”。
3. 如果是对象本身、实现边界、结构机制，优先留在对象文档。
4. 如果是跨对象消费、比较、抽象、引申，优先放到主题文档。
5. 如果只是把读者送过去，用 Markdown 链接或方向性说明，不要重写内容。
6. 如果当前内容主要是在解释来源、迁移与落位，用 Trace / Source / Decision / Placement / Gap 等维护链路字段，而不是强行写成正文。

## 7. 常见误判

- 把单篇论文文档写成“当前主题怎么消费这篇论文”的说明
- 把主题文档中的比较结论回写到单对象文档
- 把 README / index 扩写成正文式的内容消费文档
- 把 `notes/` 中的研究过程材料当成普通读者的主入口
- 把 `temp/` 中的调研稿长期当成正式主线来源

看到这些情况时，优先怀疑：**当前问题是不是已经跨了文档类型边界。**

## 8. 默认策略

- **对象文档保持原子性，主题文档承担引申。**
- **能用链接解决的，不要重写成重复内容。**
- **能在主题文档完成的消费，不要回写到对象文档。**
- **能用维护链路字段说明的，不要硬挤进正文主叙事。**

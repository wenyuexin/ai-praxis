# 结构重构规则

本文件沉淀目录重构、知识树调整与 README 导航收敛过程中提炼出的稳定规则。

它关注的问题不是“新材料写到哪里”，而是：**当既有目录结构已经形成后，如何低风险地重构、改名、收敛导航，并把过程中的结构性判断沉淀为可复用规则。**

相关入口：[`CONTRIBUTING.md`](../../CONTRIBUTING.md) / [`README 规则`](./readme-rules.md) / [`元信息文件模型`](./metadata-files.md) / [`材料处理与文档增改流程`](./documentation-workflow.md)

## 1. 适用范围

本文件适用于以下场景：

- 目录改名、合并、拆分、迁移
- README 目录树与导航结构调整
- 材料型目录、资源型目录与知识型目录边界清理
- 子领域内部分类轴重整
- 已有内容较多时的低风险结构演进

不适用于：

- 新增单篇知识笔记的日常落位
- 证据分级与 Traceability 标注本身
- 具体主题正文如何写作

## 2. 重构前先判断什么

结构重构前，先回答以下问题：

1. 当前问题是**命名问题**、**分类轴问题**、**导航问题**，还是**材料落位问题**？
2. 调整对象是仓库级通用规则，还是某个领域内部的知识组织规则？
3. 当前目录名是否会在未来内容增长后显著抬高迁移成本？
4. 当前目录是否把主题、对象、材料、资源、临时占位混在同一层级？
5. 当前重构是稳定规则沉淀，还是一次性任务判断？

默认原则：**先判断问题类型，再选择落位，不要一边迁移一边临时发明规则。**

## 3. 目录改名与迁移原则

### 3.1 先移动，再小改

目录重构默认遵循 [`AGENTS.md`](../../AGENTS.md) 的“先移动再小改”原则。具体到结构重构场景，推荐按以下顺序执行：

1. 先重命名或移动原目录
2. 再同步更新 README、overview、backlog、conflict 和相对链接
3. 最后再补充必要的边界说明

不要把“目录迁移”和“大幅重写内容”混在同一轮里完成。

### 3.2 优先改结构性问题，不优先改措辞美观

优先处理以下问题：

- 目录名明显过窄，未来会阻碍内容进入
- 目录名明显过宽，未来会吞并相邻主题
- 同层级混用了不同分类轴
- README 把材料目录误当知识目录展示
- 旧路径大量存在，已经影响导航一致性

不要因为措辞更优雅就轻易触发大规模重命名。

### 3.3 改名时区分“路径名”和“概念词”

路径迁移时，只同步：

- 路径引用
- 目录树
- 导航文字
- 目录定位描述

不要机械替换正文中的普通概念词。例如：

- 讨论 `reactive vs deliberative` 这一概念时，可以保留原概念表达
- 讨论 `competition` 作为机制时，不必因为目录改名而全部改成新目录名
- `patterns` 作为普通英文词汇时，不应机械替换成目录名

## 4. README 导航收敛规则

### 4.1 目录树深度服从 README 层级规则

目录树展开深度必须服从 [`readme-rules.md`](./readme-rules.md)：

- 根 README：按活跃度规则展开
- 一级目录 README：允许展开到三级子目录
- 二级及更深层 README：默认只记录直接子目录

不要因为内容熟悉或想“展示得更全”就擅自展开更深层目录。

### 4.2 README 目录树只展示知识骨架，不展示材料噪音

README 目录树原则上只展示目录，不展示内容文件；同时应避免把以下目录当知识骨架并列展示：

- `images/`
- `papers/`
- `notes/`
- 纯 `.gitkeep` 占位目录
- 临时中间目录

如果这些目录必须存在，也应尽量：

- 不在一级目录 README 的目录树中展示
- 不在“核心主题入口”中当作知识主题导航
- 通过边界说明解释其辅助性质

### 4.3 README 不承接一次性重构判断

README 可以承接：

- 稳定目录边界
- 分类依据
- 目录定位
- 长期有效的归属规则

README 不应承接：

- 一次性迁移过程记录
- 临时决策争论
- 任务批次编号
- 仅对当前一次重构有效的执行细节

这类内容优先保留在 `temp/`、反馈文档或必要的 `conflict.md` / `backlog.md` 中。

## 5. 分类轴与边界规则

### 5.1 同层级尽量只用一种主分类轴

同一层级应尽量只采用一种主分类轴，例如：

- 问题域
- 对象类型
- 机制类型
- 环境类型

应避免在同一层级并列混入不同分类轴的目录，例如：

- 抽象主题与具体项目并列
- 能力模块与对象实例并列

应避免将材料型目录与知识主题目录并列，例如：

- `papers/`、`images/`、`notes/` 与问题域目录同级展示

### 5.2 允许存在“有意识的复合轴”，但必须写明原因

如果某个目录内部同时存在：

- 类型轴
- 横切机制轴
- 用途轴

并且这种混合是当前最稳定、最可维护的方案，那么可以保留，但必须在 README 或 overview 中明确说明复合轴是**有意识保留**，不是误分类。

### 5.3 领域边界写在领域目录，不写成仓库通用规则

以下内容通常属于领域内部结构规则：

- 某个主题应放 `single-agent` 还是 `multi-agent`
- 某个协议在不同语境下的归属边界
- 某个对象目录应保留 object-centric 还是改成功能 taxonomy

这类规则应优先写入对应领域的 `README.md` / `overview.md`，不要直接上升为仓库通用规则，除非它已经明显对多个领域都成立。

## 6. 材料目录治理规则

### 6.1 材料目录不是知识分类节点

`papers/`、`images/`、`notes/` 等目录首先是材料容器，不是知识主题。

因此：

- 它们不应与问题域目录长期并列为主分类节点
- 不应在一级 README 目录树中当作核心知识骨架展示
- 不应因为“当前方便”而成为默认收纳一切内容的桶

### 6.2 是否迁移材料目录，取决于真实内容与引用风险

治理材料目录前，先确认：

- 目录中是否有真实文件，还是只有 `.gitkeep`
- 是否已有正文通过相对路径引用这些资源
- 迁移是否会引起大面积断链
- 内容是否已经足够明确归入某个主题目录

如果资源已被正文稳定引用，且统一迁移策略尚未确定，可以暂时保持原位，同时从 README 树中移除展示。

## 7. 什么时候把经验沉淀为规则

并非所有重构经验都值得立刻写成规则。只有满足以下条件时，才建议回流到 `docs/contributing/`：

- 该判断已在多个目录或多轮重构中重复出现
- 它不是某个单一主题的知识边界，而是组织方法规则
- 它能减少未来大量重复判断
- 它不依赖某次临时任务背景才能成立

否则，优先保留在：

- `temp/` 反馈文档
- 相关目录 `README.md`
- `backlog.md` / `conflict.md`

## 8. 推荐执行顺序

当需要做一轮目录结构重构时，推荐顺序如下：

1. 确认问题属于结构问题，而不是正文内容问题
2. 判断哪些规则是仓库通用，哪些只是领域内部边界
3. 先完成目录改名或移动
4. 同步更新路径、目录树和导航文字
5. 清理 README 中不应展示的材料型目录
6. 补充必要边界说明
7. 复核旧路径残留、断链风险与 README 规则符合性
8. 最后再决定哪些经验需要沉淀到 `docs/contributing/`

## 9. 与其他规则文件的关系

- 如果问题是 **README 深度、导航边界、目录树展示**，优先看 [`readme-rules.md`](./readme-rules.md)
- 如果问题是 **元信息文件职责**，优先看 [`metadata-files.md`](./metadata-files.md)
- 如果问题是 **材料进入正文前的判断与分流**，优先看 [`documentation-workflow.md`](./documentation-workflow.md)
- 如果问题是 **案例研究、论文材料、研究辅助材料组织**，优先看 [`research-artifacts.md`](./research-artifacts.md)
- 如果问题是 **已经决定要重构结构，怎样低风险执行并沉淀规则**，优先看本文件

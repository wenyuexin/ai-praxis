# 基础理论对象不应被应用层重复收编：`Lost in the Middle` 的落位案例

本案例用于说明：**当一个理论、机制或现象本身已经构成可独立维护的基础研究对象时，应用层应优先引用它，而不是因为“当前论证需要”就把它重新收编成自己的候选对象或局部对象位。**

## 1. 这个案例回答什么问题

它回答的不是 `Lost in the Middle` 这篇论文本身的技术结论，而是下面这些结构判断：

- 什么情况下一个理论对象应优先放在基础层，而不是挂在当前应用目录里
- 为什么“对当前应用有用”不等于“应由当前应用目录承接对象本身”
- AI 在只盯着当前任务上下文时，为什么容易把基础对象误写进应用层 `candidates.md`
- 应用层什么时候应直接引用基础层，什么时候只需局部背景承接，什么时候才值得单独起位

## 2. 案例背景

在 `agentic/06-frameworks-and-tools/05-comparisons/candidates.md` 中，`Lost in the Middle` 一度被保留为 `text-skill-governance.md` 的高优先级候选对象，理由是它可以支撑长上下文中的规则位置效应、context shaping 和规则重复策略。

但仓库里其实已经有更稳定的基础承接位：

- `llm/01-foundations/transformer-architecture/attention-mechanisms/position-bias/README.md`
- `llm/01-foundations/transformer-architecture/attention-mechanisms/position-bias/overview.md`
- `llm/01-foundations/transformer-architecture/attention-mechanisms/position-bias/papers/LostInTheMiddle_2307.03172.md`

这说明问题已经不是“这篇论文是否对 agentic 有帮助”，而是：**它是否本来就属于一个独立的基础问题空间。**

## 3. 最容易出现的误判

最常见的误判不是技术理解错误，而是落位时只看当前任务：

- 现在正在研究 `agentic`
- 这篇论文对当前段落有用
- 于是把它继续挂在 `agentic/.../candidates.md` 里

这种做法短期顺手，但长期会把基础对象重新吸进应用目录，使后续维护者越来越难区分：

- 哪些是应用层自己的研究对象
- 哪些只是应用层正在消费的基础理论

## 4. 为什么“当前有用”不是稳定落位标准

“当前有用”只能说明它适合被引用，不能直接说明它应由当前目录承接对象本身。

更稳的判断顺序应是：

1. 先看这个对象能否脱离当前应用语境独立成立。
2. 再看它是否会被多个上层主题反复复用。
3. 再看它是否已经形成自己的研究链条，例如现象、解释、方法、边界与开放问题。
4. 最后再检查仓库里是否已经有对应基础承接位。

如果后 2–4 点已经成立，应用层通常不应再把它当成“自己的候选对象”。

## 5. 这个案例里的稳妥判断标准

`Lost in the Middle` 之所以更适合放在基础层，不是因为“理论一定高于应用”，而是因为它同时满足了几类更可操作的信号：

- **可脱离当前应用独立成立**：即使不提 `agentic`，它仍然是一个自然的研究问题。
- **跨场景复用**：它不只服务 `text-skill-governance`，也会被 RAG、长文档理解、retrieval ordering、prompt layout 等方向反复引用。
- **已有稳定命名**：`position bias`、`Lost in the Middle` 已经是外部世界中可聚合材料的稳定对象名。
- **已有独立研究链条**：仓库中已经不是一句背景介绍，而是形成了专题目录、总览和论文笔记。
- **仓库已有承接位**：因此应用层更合理的动作是引用，而不是重复收编。

## 6. 本案例最终采用的分工

对这个对象更稳的分工是：

- 基础对象本身留在 `llm/.../position-bias/` 目录下继续承接
- 应用层文档（如 `text-skill-governance.md`）直接引用其结论、边界和相关论文
- 应用层 `candidates.md` 不再把它继续保留为“待研究对象”，除非当前问题真的还在研究对象本身，而不是消费已有研究

这里最关键的不是“谁先写”，而是：**对象归属与引用关系要分开。**

## 7. 这个案例不支持什么结论

本案例不支持下面这些过度推广：

- 不是所有理论材料都必须先建基础目录
- 不是所有应用层都不能写理论背景
- 不是只要论文看起来更抽象，就应自动上移到 foundations

如果某个理论材料：

- 只服务当前应用问题
- 不能脱离当前应用语境独立成立
- 不太可能被多个主题复用
- 也没有继续长出独立研究链条的迹象

那它完全可以只在应用层用一个小节、一段话或一篇专题文局部承接，而不必单独起位。

## 8. 这个案例给出的最小判断顺序

以后再遇到类似对象时，更稳的顺序不是直接问“它有没有研究价值”，而是先问：

1. 它现在是当前应用的支撑材料，还是可独立成立的对象？
2. 它是否会被多个主题复用？
3. 它是否已有稳定命名和独立研究链条？
4. 仓库里是否已经存在基础承接位？

如果 2–4 条明显成立，默认优先放基础层；应用层引用，不重复承接对象本身。

## 9. 什么时候才该回看这个案例

只有在以下情况才需要回看本案例：

- 已经读过主规则，但仍拿不准某个理论对象该不该留在当前应用目录
- AI 反复把“基础对象”和“应用层正在引用的对象”混写
- 需要判断一个理论材料究竟应独立起位，还是只做应用层局部背景
- 需要向其他 AI 说明：为什么“当前论证有用”不等于“当前目录拥有该对象”

如果只是日常判断元信息文件职责，优先回到 `docs/contributing/metadata-files.md`；如果只是想理解元信息文件为什么按问题触发，再优先回到 `docs/contributing/intent/metadata-files.md`。

# Agent System Modeling：`README.md` + `candidates.md` 先成立的案例

本案例用于说明：**当一个目录最先稳定出现的是候选研究对象队列，而不是内容缺口、结构支架或人类向主线判断时，为什么更合理的落位是 `README.md` + `candidates.md`，并暂不创建 `backlog.md`、`landscape.md` 或 `overview.md`。**

## 1. 这个案例回答什么问题

它回答的不是 Agent harness 本身的知识结论，而是下面这些规则判断：

- 什么情况下一个目录会先长出 `candidates.md`
- 为什么“当前没有 `overview.md`”在这里不是缺文件，而是时机未到
- 为什么对象队列可以先稳定下来，而问题缺口仍然留在上层 `backlog.md`
- 为什么 README 只做粗定位与最小入口分流，不直接承担对象清单

## 2. 案例背景

`agentic/01-foundations/agent-system-modeling/` 关注的是 foundations 层的系统建模问题。

在这个目录里，当前最先稳定下来的不是：

- 一套成熟的上位主线判断
- 一组需要单独列账的内容缺口
- 一篇解释为什么当前目录按某种结构切分的结构化研究文

而是：

- 一批值得持续跟踪的论文、survey 与候选研究对象
- 它们围绕同一个上位问题聚集，但还不足以直接压成正式总论
- 下一步最需要回答的是“先继续跟踪哪些对象”，而不是“先补哪一块空白”

也就是说，这里最先暴露出来的是**对象队列需求**。

## 3. 为什么这里不应急着写 `overview.md`

按主规则，`overview.md` 应在目录已经积累出足以支撑一篇相对独立、可单篇成立的人类向主文时再考虑创建。

而这个目录当前并不满足这个条件：

- 虽然已经围绕 `Agent Harness as System Model` 形成主题焦点，但上位判断仍在继续压实
- 现有材料更多是“哪些对象值得继续跟踪、哪些结论还要复核”，而不是已稳定主线
- 如果此时急着补 `overview.md`，很容易把候选对象管理、待复核判断与 foundations 层总论混进同一篇文档里

所以这里更稳的判断不是“补一个主文占位”，而是承认：**当前更早成熟的是 `candidates` 信号，不是 `overview` 信号。**

## 4. 为什么这里也不应单独补 `backlog.md`

这并不是因为没有问题缺口，而是因为：

- 当前最先稳定下来的不是一组待补主题，而是一组待跟踪对象
- foundations 层的一般问题缺口已经由上层 `01-foundations/backlog.md` 统一承接
- 如果在这里再单独补一份 `backlog.md`，很容易把“对象队列”和“问题缺口”重新混写

所以这里更稳的分工是：

- 本目录内用 `candidates.md` 跟踪下一步值得研究的对象
- 更上位的问题缺口继续留在上层 `backlog.md`

## 5. 本案例最终采用的拆分方式

对 `agent-system-modeling/` 更稳的拆分是：

- `README.md`：保留目录定位、边界说明与最小入口分流
- `candidates.md`：承接当前主题下的候选研究对象、阅读优先级与对象队列维护原则
- 暂不创建 `backlog.md`
- 暂不创建 `landscape.md`
- 暂不创建 `overview.md`

这里最关键的不是“文件越少越好”，而是承认：当前真正稳定出现的认知任务只有两类：

- 这个目录是什么、承接什么问题
- 下一步该跟踪哪些对象

## 6. 这个案例最容易诱发的三类误判

### 误判 A：既然已经有清晰主题，就应该马上写 `overview.md`

这种误判会忽略一个关键条件：主题聚焦，不等于已经形成相对独立、可单篇成立的人类向主文。

这里当前更像“围绕一个上位问题组织对象队列”，而不是“已经有一篇成熟总论可写”。

### 误判 B：只要有待研究对象，也应该同时补 `backlog.md`

这种误判会把“对象队列”和“问题缺口”混为一谈。

- `candidates.md` 回答的是“接下来值得研究哪些对象”
- `backlog.md` 回答的是“这个方向还有哪些内容缺口”

在这个目录里，当前先稳定下来的是前者，而后者仍由上层目录统一承接。

### 误判 C：README 可以顺手把对象列表也写进去

这种误判会让 README 再次变回万能入口。

README 在这里的职责只是：

- 说明这个目录属于什么问题空间
- 让读者先判断这里是不是自己要找的目录
- 把对象队列转交给 `candidates.md`

它不应直接承接候选对象表。

## 7. 这个案例给出的关键判断

这个案例最值得复用的判断有四条：

- `candidates.md` 可以先于 `overview.md` 出现
- “当前没有 `overview.md`”有时不是缺文件，而是对象队列先成熟、主线判断尚未成熟
- `backlog.md` 不必在每个子目录里机械补齐；如果问题缺口已由上层稳定承接，本层只保留 `candidates.md` 也可以成立
- README 只要能完成粗定位与最小分流，就不应吞掉对象队列职责

## 8. 什么情况下应重新触发 `overview.md` 或 `backlog.md` 判断

如果未来出现下面任一情况，应重新评估是否补更多元信息文件：

- 多个候选对象反复收敛到同一组稳定上位判断，已经足以支撑相对独立的人类向主文
- 当前对象队列之外，又出现一组本目录内部独有、且不适合继续放在上层的内容缺口
- 仅靠 README 与 candidates，已经不足以承接目录内的结构、判断与后续研究协作

也就是说，本案例不是在证明“这个目录永远只需要 README + candidates”，而是在说明：**当前时点这就是更稳的起手式。**

## 9. 这个案例对规则系统的直接启发

本案例的核心价值，不是重复主规则或 `intent/` 中已经给出的时机模型，而是提供一个可观察的真实样本，说明“对象队列先成熟”在仓库里会长成什么样。相关结论应回到 `docs/contributing/metadata-files.md` §2.2 / §5.2，以及 `docs/contributing/intent/metadata-files.md` §10 / §12 统一理解。

它对应的是另一种很典型的研究建设流程：

1. 先有目录定位
2. 然后候选对象队列先稳定下来
3. 问题缺口仍由上层统一承接
4. 最后才在上位判断成熟后考虑 `overview.md` 或其他文件

## 10. 队列条目什么时候应开始收缩

这个案例最初回答的是：为什么一个目录可以先长出 `README.md` + `candidates.md`。

但当对象队列继续演化时，还会出现第二个常见判断：**候选对象一旦已经在对象目录或本层形成稳定正式产出，`candidates.md` 条目就不应继续膨胀成压缩研究摘要。**

更稳的做法是把条目收回到队列层，只保留：

- 当前状态：它现在仍处于待开题、已建立 case-study，还是“已有正式产出、继续补 gap”
- 当前产出或落位：正式专题、论文笔记、对象目录或案例文档在哪里
- 剩余 gap：为什么它还暂时留在队列里，而不是完全移除

不宜继续堆在条目里的内容包括：

- 已经在正式案例文档中稳定展开的机制细节
- 长段源码核验摘要
- 本该由对象专题自己承接的结论列表

也就是说，`candidates.md` 可以很丰富，但它首先仍应让读者一眼看出：**这个对象为什么还在队列里，以及下一步该去哪里继续读。**

## 11. 什么时候才该回看这个案例

只有在以下情况才需要回看本案例：

- 已经读过主规则，但仍拿不准“为什么这里只有 `README.md` + `candidates.md`”
- 需要判断某个目录当前先成熟的是对象队列，而不是结构支架或主线主文
- 需要复盘为什么某些子目录不该同时补 `backlog.md` 与 `candidates.md`
- 需要判断候选对象已经形成正式产出后，队列条目是否该收缩成“状态 + 指针 + gap”
- 需要给其他 AI 一个真实样本，说明 `README + candidates` 也是合理的早期组合

如果只是日常判断元信息文件职责，优先回到 `docs/contributing/metadata-files.md`；如果只是想理解这些职责为什么这样定义，再优先回到 `docs/contributing/intent/metadata-files.md`。

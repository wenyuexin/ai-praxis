# Training Infrastructure：`landscape.md` 先于 `overview.md` 出现的案例

本案例用于说明：**当一个目录最先稳定出现的是结构切分、边界归属与研究下钻路径，而不是人类向主线判断时，为什么更合理的落位是 `README.md`、`index.md`、`landscape.md`、`roadmap.md`，并暂不创建 `overview.md`。**

## 1. 这个案例回答什么问题

它回答的不是训练基础设施本身的知识细节，而是下面这些规则判断：

- 什么情况下 `landscape.md` 会先于 `overview.md` 出现
- 为什么“当前没有 `overview.md`”有时不是缺文件，而是时机尚未成熟
- 为什么一个目录可以同时需要 `index.md`、`landscape.md`、`roadmap.md`，但这并不等于职责重叠
- 为什么 README 即使仍然很重要，也不该继续扛目录树、路径和结构治理

## 2. 案例背景

`training-infra/` 关注的是大规模模型训练过程中的工程底座。

这个目录天然会遇到几类很重的结构问题：

- 主题可以按系统层、瓶颈类型、生命周期阶段等多种轴切分
- 与 `llm/`、`deep-learning/`、`reinforce-learning/` 的边界并不天然自明
- 继续补材料时，维护者首先会问“这应落在哪一层、与哪个相邻目录分工”，而不是“这能否先写成一篇主文”

也就是说，这个目录最先暴露出来的不是主线综述需求，而是**结构支架需求**。

## 3. 为什么这里不应急着写 `overview.md`

按主规则，`overview.md` 只有在目录已经积累出足以支撑一篇相对独立、可单篇成立的人类向主文时才应出现。

而这个目录当前并不满足这个条件：

- 各主干子目录虽然已经稳定，但内容仍在继续建设
- 读者当前最先需要恢复的，不是单一主线叙事，而是“这个主题内部如何切、与相邻目录怎么分工”
- 如果此时急着写 `overview.md`，很容易把结构治理、边界裁剪和主线理解混进同一篇文档里

所以这里更稳的判断不是“补齐文件类型”，而是承认：**当前更早成熟的是 `landscape` 信号，不是 `overview` 信号。**

## 4. 本案例最终采用的拆分方式

对 `training-infra/` 更稳的拆分是：

- `README.md`：保留目录定位、轻量分类依据、边界说明与最小入口分流
- `index.md`：承接目录结构、按子目录定位、按问题找入口
- `landscape.md`：承接为什么这样切分、与相邻目录如何分工、后续该往哪里下钻
- `roadmap.md`：承接按读者背景推荐起点、学习顺序与推进路径
- 暂不创建 `overview.md`

这里最关键的不是“多建几个元信息文件”，而是把几类不同认知任务拆开：

- 目录说明
- 结构与查找
- 结构支架与边界治理
- 起点与路径

## 5. 这个案例最容易诱发的三类误判

### 误判 A：训练基础设施主题很大，所以应该先写 `overview.md`

这种误判会忽略一个关键条件：主题大，不等于已经形成单篇可成立的人类向主文。

如果结构切分、边界归属与下钻路径还在频繁影响材料落位，`overview.md` 往往还不是最先该出现的文件。

### 误判 B：既然已经有 `landscape.md`，那 `index.md` 可以不建

这种误判会把“结构支架”与“目录导航”混为一谈。

- `landscape.md` 解释为什么这样切、各部分如何分工
- `index.md` 解决我已经知道想找什么时该去哪

两者都可能需要，但它们回答的不是同一类问题。

### 误判 C：既然 README 里已经说明了分类依据，`landscape.md` 只是重复

这种误判会低估“轻量分类依据”和“结构化研究文”的差异。

README 只需要让读者先判断这个目录是否相关；`landscape.md` 则需要稳定回答：

- 为什么按这一轴切，而不是按另一轴切
- 与相邻目录如何分工
- 补新材料时应优先按什么结构继续下钻

## 6. 这个案例给出的关键判断

这个案例最值得复用的判断有四条：

- `landscape.md` 不一定是从 `overview.md` 里拆出来的后置文件，它也可能先作为研究支架出现
- 当结构问题先成熟时，“暂时没有 `overview.md`”是合理状态，不是缺文件
- `README.md`、`index.md`、`landscape.md`、`roadmap.md` 可以同时存在，但前提是它们回答的认知任务不同
- 目录越容易在 README 中混进目录树、路径和结构治理，越要尽早把这些职责拆出去

## 7. 什么情况下应重新触发 `overview.md` 判断

如果未来出现下面任一情况，应重新评估是否在 `training-infra/` 补 `overview.md`：

- 顶层已经形成稳定主线，且读者只读一篇就应先读它
- 多个子目录的问题开始反复回到同一组上位判断
- 当前仅靠 README、index、landscape、roadmap，读者仍无法恢复整体理解
- 目录已经积累出足以支撑相对独立人类向主文的材料与主线判断

也就是说，本案例不是在证明“`training-infra/` 永远不需要 `overview.md`”，而是在说明：**当前时点还不该机械创建它。**

## 8. 这个案例对规则系统的直接启发

本案例的核心价值，不是重复主规则或 `intent/` 中已经给出的时机模型，而是提供一个可观察的真实样本，说明“结构支架先成熟”在仓库里会长成什么样。相关结论应回到 `docs/contributing/metadata-files.md` §2.2 / §4.4，以及 `docs/contributing/intent/metadata-files.md` §10 / §12 统一理解。

它对应的是一种很典型的研究建设流程：

1. 先有目录定位
2. 然后结构问题先暴露
3. 再把目录导航与学习路径从 README 中拆出去
4. 最后才在材料成熟后考虑 `overview.md`

## 9. 什么时候才该回看这个案例

只有在以下情况才需要回看本案例：

- 已经读过主规则，但仍拿不准“为什么这里允许先有 `landscape.md` 而没有 `overview.md`”
- 需要判断某个目录是不是已经暴露出结构支架问题，而不是主线主文问题
- 需要复盘为什么某些目录不该急着补 `overview.md`
- 需要给其他 AI 一个真实样本，说明 `landscape-before-overview` 不是纸上规则

如果只是日常判断元信息文件职责，优先回到 `docs/contributing/metadata-files.md`；如果只是想理解这些职责为什么这样定义，再优先回到 `docs/contributing/intent/metadata-files.md`。

# Interdisciplinarity：复杂横向目录的 README / index / roadmap 拆分案例

本案例用于说明：**当一个目录既要解释自身为什么存在，又要承接大量结构映射与路径分流时，为什么 `README.md`、`index.md`、`roadmap.md` 不能继续混写，以及何时只出现 `overview` 信号而不机械创建 `overview.md`。**

## 1. 这个案例回答什么问题

它回答的不是 `interdisciplinarity/` 目录下每个子主题的知识细节，而是下面这些规则边界问题：

- 什么样的目录会稳定同时长出“目录定位”“结构导航”“阅读路径”三类内容
- 为什么复杂目录越不能把 `README.md` 写成大而全入口
- 为什么 `index.md` 与 `roadmap.md` 必须分开，而不能都算“导航”
- 什么叫已经出现 `overview` 信号，什么叫仍不应机械创建 `overview.md`

## 2. 案例背景

`interdisciplinarity/` 不是普通的纵向技术目录，而是一个横向交叉目录。

它既需要解释：

- 为什么这里按外部学科而不是按技术栈组织
- 为什么这些横向视角会映射到 `llm/`、`agentic/`、`rag/` 等纵向目录
- 为什么目录顺序按“个体 → 系统 → 群体 → 制度”递进

又天然容易继续长出：

- 子目录结构展示
- 按问题找入口
- 按阅读目标找入口
- 按读者背景推荐起点
- 建议先看哪个、再看哪个

这意味着：如果不主动拆分，`README.md` 很容易把三类职责全部吞进去。

## 3. 为什么不能继续让 README 一把抓

在这种目录里，`README.md` 最容易沿着下面的滑坡扩张：

- 先解释目录定位
- 接着顺手展示目录结构
- 展示结构后，再继续补“我该先读哪里”
- 最后把按问题查找、按读者背景分流、阅读路径推荐都写进去

短期看，这会让单篇文件很方便；但长期会稳定制造三类问题：

- `README.md` 重新变成万能入口，吞掉其他元信息文件的触发空间
- `index.md` 退化成可有可无的辅助页
- `roadmap.md` 被误写成“轻量导航补充”，无法稳定承接路径职责

## 4. 本案例最终采用的拆分方式

对 `interdisciplinarity/` 更稳的拆分是：

- `README.md`：只保留目录定位、排序逻辑、与纵向技术栈的关系，让读者先明白“这个目录为什么存在、它按什么轴组织”。
- `index.md`：承接目录结构、按子目录定位、按问题找入口、按阅读目标找入口，让读者在已经知道自己想找什么时能直接定位。
- `roadmap.md`：单独承接按读者背景推荐起点、按阅读目标推荐路径、阅读顺序与推进建议，回答“从哪开始、怎么走”。

这里最关键的不是“把内容平均分配到三个文件”，而是把三种不同认知任务拆开：

- 目录说明
- 结构与查找
- 起点与路径

## 5. 这个案例最容易诱发的两类误判

### 误判 A：只要是入口文件，就把结构和路径一起塞进 README

这种误判会导致：

- `README.md` 越写越长
- `index.md` 即使存在，也没有独立触发空间
- “目录说明”和“查找导航”重新缠在一起

### 误判 B：只要都带一点导航味道，index 和 roadmap 就可以混用

这种误判会导致：

- `index.md` 开始写“建议先看哪个、适合谁先看”
- `roadmap.md` 反过来开始补目录结构或子目录说明
- 最终又回到“多篇文件共同承担同一组模糊职责”的状态

## 6. 这个案例给出的关键判断

这个案例最值得复用的判断有两条：

- `index.md` 解决“去哪”，所以它负责结构展示与查找导航。
- `roadmap.md` 解决“从哪开始、怎么走”，所以它负责起点、顺序和路径分流。

只要一个目录同时稳定出现“结构映射”和“路径分流”两类需求，就不要再把它们压回同一个文件里。

## 7. 为什么这里没有顺手创建 overview.md

`interdisciplinarity/` 已经出现了一种很重要的信号：某些子目录内部开始长出主线性很强的“核心追问”，说明后续可能需要 `overview.md` 来承接更独立的人类向主文。

但这个信号本身，还不等于应该立刻创建 `overview.md`。

原因是：

- 顶层当前最迫切的问题，是把目录说明、结构导航、路径分流先拆稳
- 主线性强的问题簇目前仍主要散落在子目录中，还没有收敛成一篇顶层单篇可成立的人类向主文
- 如果这时为了“文件类型完整”而补 `overview.md`，很容易重新把结构解释或目录协调任务写进去

所以这里更稳的判断是：**承认已经出现 `overview` 信号，但暂不机械创建 `overview.md`。**

## 8. 什么情况下应重新触发 overview 判断

如果未来出现下面任一情况，应重新评估是否在该目录补 `overview.md`：

- 顶层已经形成一条稳定主线，且读者只读一篇就应先读它
- 多个子目录的核心问题开始反复回到同一组上位判断
- 目录中已经不只是“按结构找内容”，而是需要先恢复整体理解框架
- 现有 `README.md` / `index.md` / `roadmap.md` 都无法承接那篇应当单独成立的人类向主文

## 9. 这个案例给规则系统带来的直接启发

本案例不是为了证明 `interdisciplinarity/` 特殊，而是为了给元信息文件规则补两个很容易复发的判断：

- 复杂目录不应默认把 `README.md` 写成大而全入口
- 看到 `overview` 信号，不等于立刻补 `overview.md`；要先判断是否已经形成单篇可成立的人类向主文

也就是说：文件类型的出现顺序，不应由“看起来都需要”决定，而应由**哪类认知任务已经稳定出现**来决定。

## 10. 什么时候才该回看这个案例

只有在以下情况才需要回看本案例：

- 主规则已经读过，但仍拿不准复杂目录该怎么拆 `README.md`、`index.md`、`roadmap.md`
- 需要判断“为什么这里已经有 `overview` 信号，但仍不应马上创建 `overview.md`”
- 需要复盘某个横向目录为什么总把结构、导航、路径重新混写

如果只是日常判断元信息文件职责，优先回到 `docs/contributing/metadata-files.md`；如果只是想理解这些职责为什么这样定义，再优先回到 `docs/contributing/intent/metadata-files.md`。

# Learning Materials：`README.md` + `backlog.md` 先成立的案例

本案例用于说明：**当一个目录最先稳定出现的是待补主题与方向缺口，而不是结构支架、候选对象队列或人类向主线判断时，为什么更合理的落位是 `README.md` + `backlog.md`，并暂不创建 `index.md`、`landscape.md`、`candidates.md` 或 `overview.md`。**

## 1. 这个案例回答什么问题

它回答的不是学习资料本身的内容细节，而是下面这些规则判断：

- 什么情况下一个目录会先长出 `backlog.md`
- 为什么“当前没有 `overview.md`”在这里不是缺文件，而是时机未到
- 为什么待补主题可以先稳定下来，而候选对象队列不一定需要同步出现
- 为什么 README 只做目录说明与最小入口分流，不直接吞并缺口清单

## 2. 案例背景

`learning-materials/` 承接的是正式知识目录之外的辅助学习资源与零散沉淀入口。

在这个目录里，当前最先稳定下来的不是：

- 一篇可单篇成立的人类向主文
- 一份解释目录内部复杂切分逻辑的结构化研究文
- 一批需要长期维护的候选研究对象队列

而是：

- 一组已经识别出来、但尚未沉淀进正式知识目录的待补主题与方向缺口
- 它们可以被持续记录、分优先级维护，但还不适合压成正式总论
- 下一步最需要回答的是“还有哪些方向没补”，而不是“先跟踪哪些对象”

也就是说，这里最先暴露出来的是**内容缺口需求**。

## 3. 为什么这里不应急着写 `overview.md`

按主规则，`overview.md` 应在目录已经积累出足以支撑一篇相对独立、可单篇成立的人类向主文时再考虑创建。

而这个目录当前并不满足这个条件：

- 这里更像辅助资源入口，而不是已经形成稳定主线判断的主题目录
- 现有内容首先需要被管理的是“哪些方向还没补”，不是“这个主题最值得先理解的主线是什么”
- 如果此时急着补 `overview.md`，很容易把资源入口说明、待补方向和零散判断混进同一篇文档里

所以这里更稳的判断不是“先补一篇总览占位”，而是承认：**当前更早成熟的是 `backlog` 信号，不是 `overview` 信号。**

## 4. 为什么这里也不必先补 `candidates.md`

这并不是因为没有可研究对象，而是因为：

- 当前最先稳定下来的不是一批需要持续跟踪的论文、项目或产品对象
- 这里主要承接的是资源入口与方向补缺，而不是对象研究队列管理
- 如果在这里同步补一份 `candidates.md`，很容易把“待补主题”和“候选对象”重新混写

所以这里更稳的分工是：

- 用 `backlog.md` 记录待补方向、主题缺口与后续沉淀目标
- 暂不为并未稳定成队列的对象管理单独创建 `candidates.md`

## 5. 本案例最终采用的拆分方式

对 `learning-materials/` 更稳的拆分是：

- `README.md`：保留目录定位、边界说明与最小入口分流
- `backlog.md`：承接尚未沉淀进正式知识目录的待补主题与方向缺口
- 暂不创建 `index.md`
- 暂不创建 `candidates.md`
- 暂不创建 `landscape.md`
- 暂不创建 `overview.md`

这里最关键的不是“文件越少越好”，而是承认：当前真正稳定出现的认知任务只有两类：

- 这个目录是什么、为什么存在
- 还有哪些方向与主题缺口值得继续补

## 6. 这个案例最容易诱发的三类误判

### 误判 A：既然目录里已经有很多资源类型文件，就应该先补 `index.md`

这种误判会把“资源按单文件收纳”直接等同于“已经形成需要骨架型导航页的稳定子目录结构”。

这里当前主要是文件级资源入口，真正先稳定的问题不是目录树导航，而是方向缺口管理。

### 误判 B：只要有待补方向，也应该同时补 `candidates.md`

这种误判会把“内容缺口”和“对象队列”混为一谈。

- `backlog.md` 回答的是“这里还有哪些内容没覆盖”
- `candidates.md` 回答的是“接下来值得研究哪些对象”

在这个目录里，当前先稳定下来的是前者，而不是后者。

### 误判 C：README 可以顺手把待补清单也写进去

这种误判会让 README 再次变回万能入口。

README 在这里的职责只是：

- 说明这个目录承接什么类型的辅助材料
- 让读者先判断这里是不是自己要找的入口
- 把待补方向转交给 `backlog.md`

它不应直接承接长期维护的缺口清单。

## 7. 这个案例给出的关键判断

这个案例最值得复用的判断有四条：

- `backlog.md` 可以先于 `overview.md` 出现
- “当前没有 `overview.md`”有时不是缺文件，而是内容缺口先成熟、主线判断尚未成熟
- `candidates.md` 不必在每个早期目录里机械补齐；当稳定出现的是待补方向而不是对象队列时，只保留 `backlog.md` 就可以成立
- README 只要能完成目录说明与最小分流，就不应吞掉缺口清单职责

## 8. 什么情况下应重新触发 `index.md`、`candidates.md` 或 `overview.md` 判断

如果未来出现下面任一情况，应重新评估是否补更多元信息文件：

- 目录下形成多个稳定子目录，且读者需要先看结构再决定往哪走
- 待补方向开始收敛成一批需要持续跟踪的具体对象队列
- 当前资源入口之外，已经积累出足以支撑相对独立人类向主文的稳定主线判断
- 仅靠 README 与 backlog，已经不足以承接目录内的结构、对象与后续研究协作

也就是说，本案例不是在证明“这个目录永远只需要 README + backlog”，而是在说明：**当前时点这就是更稳的起手式。**

## 9. 这个案例对规则系统的直接启发

本案例的核心价值，不是重复主规则或 `intent/` 中已经给出的时机模型，而是提供一个可观察的真实样本，说明“内容缺口先成熟”在仓库里会长成什么样。相关结论应回到 `docs/contributing/metadata-files.md` §2.2 / §5.1，以及 `docs/contributing/intent/metadata-files.md` §10 / §12 统一理解。

它对应的是另一种很典型的研究建设流程：

1. 先有目录定位
2. 然后待补方向与主题缺口先稳定下来
3. 再按优先级持续维护缺口，而不是急着补对象队列或主线总览
4. 最后才在结构、对象或主线真正成熟后考虑其他文件

## 10. 什么时候才该回看这个案例

只有在以下情况才需要回看本案例：

- 已经读过主规则，但仍拿不准“为什么这里只有 `README.md` + `backlog.md`”
- 需要判断某个目录当前先成熟的是内容缺口，而不是对象队列、结构支架或主线主文
- 需要复盘为什么某些目录不该同时补 `backlog.md` 与 `candidates.md`
- 需要给其他 AI 一个真实样本，说明 `README + backlog` 也是合理的早期组合

如果只是日常判断元信息文件职责，优先回到 `docs/contributing/metadata-files.md`；如果只是想理解这些职责为什么这样定义，再优先回到 `docs/contributing/intent/metadata-files.md`。

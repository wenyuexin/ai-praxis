# 元信息文件模型

本文件定义仓库中元信息文件的职责边界、出现条件和使用方式。

相关入口：[`CONTRIBUTING.md`](../../CONTRIBUTING.md) / [`docs/contributing/README.md`](./README.md) / [`README 规则`](./readme-rules.md) / [`Evidence 规则`](./evidence-rules.md) / [`Traceability 规则`](./traceability-rules.md)。

元信息文件不是知识正文，而是帮助读者和维护者完成定向、发现、理解、推进、选题、规划、校验等认知任务的辅助文件。

## 快速判断

如果你只是想快速判断“这个目录现在该不该建某个元信息文件”，先按下面顺序判断，不必先通读全文：

- 先判断当前问题属于哪一类：**说明、导航、理解、结构、缺口、对象、路径、冲突**。
- 对应文件分别是：`README.md`、`index.md`、`overview.md`、`landscape.md`、`backlog.md`、`candidates.md`、`roadmap.md`、`conflict.md`。
- 不要机械补齐所有文件；文件按问题触发，不按层级补齐。
- `README.md` 负责目录说明，不再默认承担目录结构展示；`index.md` 负责目录结构与查找导航；`landscape.md` 不是默认创建文件。
- 拿不准是否该新建正文专题时，先不要急着写正文；先判断问题属于目录说明、查找导航、内容缺口还是口径冲突，再分别落到 `README.md`、`index.md`、`backlog.md` 或 `conflict.md`。
- 需要细看边界时：`README / overview / landscape` 看第 4 章，`backlog / candidates / roadmap` 看第 5 章，创建策略看第 7 章。

本节只用于快速分流；文件职责以第 3 章总览表为准，具体边界看第 4–6 章，创建与拆分规则看第 7 章。

## 1. 使用边界

本文件只回答三件事：哪些文件属于元信息文件、它们分别解决什么问题、在什么情况下出现。

- 用它判断 `README.md`、`overview.md`、`landscape.md`、`backlog.md`、`candidates.md`、`roadmap.md`、`conflict.md` 的职责边界和出现条件。
- 材料处理流程见 [`documentation-workflow.md`](./documentation-workflow.md)。
- 单篇写作表达不在本文。
- 项目研究辅助材料（`notes.md`、`notes/`、`source-notes.md`、`evidence-notes.md` 等）不在本文，见 [`research-artifacts.md`](./research-artifacts.md)。
- Evidence registry 不是默认层级；仅在证据对照本身成为稳定维护对象时启用，见 [`evidence-rules.md`](./evidence-rules.md)。

## 2. 核心原则

### 2.1 元信息文件服务认知任务，不服务形式对称

每个元信息文件对应一个主认知任务：说明、导航、理解、推进、选题、路径、校验。不需要为每个目录机械补齐所有任务的文件。一个目录是否有某个文件，取决于它当前是否有对应的认知需求，而不是模型中定义了七个层级就应该有七个文件。

### 2.2 文件按问题触发，不按层级补齐

元信息文件的创建信号是目录的实际问题，而不是层级对称欲望。当读者不知道这个目录是什么或为什么存在时，补 README；当需要目录结构展示或查找导航时，考虑 index；当主题主线不明时，考虑 overview；当结构分歧散落各处时，考虑 landscape。不要因为父目录有某份文件、子目录就必须同样有。

### 2.3 一个文件只承接一个主问题

在同一份文件中同时讲"主题是什么"和"本目录如何组织"，通常说明 overview 和 landscape 职责混杂，应考虑拆分。同理，把候选对象队列塞进 backlog，或者把学习路径写进 README 而非 roadmap，都会让文件和读者对不上。

### 2.4 先明确分工，再稳定命名

新目录出现时，先问"这个目录当前最需要哪个认知文件"，而不是先决定"我要创建哪些文件"。文件分工稳定后，再按约定命名锁定职责。命名服务于分工，不能反过来用命名推分工。

### 2.5 允许能力位存在，不等于默认要创建

`index.md`、`roadmap.md` 是仓库允许存在的预定义能力位，但它们在多数目录下并不需要。只有在稳定的实际需求出现时才创建，而不是因为模型中定义了 L2/L6。

## 3. 总览：文件与认知任务对照

下表是元信息文件的快速对照工具，不是所有目录都必须机械补齐的模板。

| 层级 | 认知任务 | 文件名 | 职责 | 出现条件 | 典型内容 |
|:---|:---|:---|:---|:---|:---|
| L1 说明 | Explain | `README.md` | "这是什么地方？为什么存在？和相邻目录如何区分？" | 目录需要被当作入口说明对象时启用；纯叶子目录（仅含文件，无子目录可能）不需要 | 一句话定位、入口说明、分类依据、边界说明、相邻目录关系 |
| L2 导航 | Navigate | `index.md` | "这个目录下面有什么？我已经知道想找什么时该去哪？" | 可选；当目录需要同时承接结构展示与查找导航时启用 | 目录结构、按问题找入口、按阅读目标找入口、按子目录或对象类型找入口；不包含阅读顺序 |
| L3 理解 | Understand | `overview.md` | "如果我只读一篇，这个主题最值得先理解的主线是什么？" | 该目录代表一个需要建立整体认知的主题 | 主线脉络、关键判断、整体理解、相对独立的主文 |
| — | 结构化研究 | `landscape.md` | "这个主题内部有哪些板块、边界和研究路径？各部分如何分工与衔接？" | 仅当 overview 与结构研究职责分离时；不是默认创建文件 | 板块划分、研究地图、下钻路径、文档协作关系 |
| L4 推进 | Track | `backlog.md` | "这个领域还有什么没被覆盖？" | 目录处于活跃建设期，内容有明显缺口 | 已知空白、待填补主题、关联主题或已有材料 |
| L5 选题 | Queue | `candidates.md` | "接下来值得研究哪些对象？" | 可选；需要持续跟踪论文、仓库、官方文档、产品、协议、benchmark 等候选研究对象，且对象列表不宜塞进 `backlog.md` 时创建 | 待研究对象、对象类型、关联问题、研究理由、当前状态、研究产出或落位 |
| L6 规划 | Plan | `roadmap.md` | "从哪开始、按什么顺序阅读/建设？" | 可选；有明确的内容组织顺序、学习路径或前置依赖时创建 | 阅读阶段、推荐顺序、前置知识或依赖 |
| L7 校验 | Reconcile | `conflict.md` | "这里是否存在定义冲突或口径不一致？" | 仅当发现冲突时可选创建 | 冲突点、涉及文档、冲突类型、待核验口径或边界 |

`landscape.md` 在表中没有 L 编号，因为它不是七层模型的强制层级——它是按需存在的结构化研究文，服务于 overview 已无法承载的结构说明需求。

## 4. 入口、发现与理解

### 4.1 `README.md`：目录说明

`README.md` 回答"这是什么地方？为什么存在？和我有什么关系？"

#### 基线职责

`README.md` 是目录说明文件，负责在入口处完成说明与最小分流。它通常承接：

- 一句话定位：本目录是什么、面向谁
- 入口说明：第一次进入时应先走哪条阅读路径
- 轻量边界说明：什么适合放这里、什么不适合放这里
- 轻量分类依据：本目录按什么逻辑组织
- 与相邻目录的关系：为什么放在这里、和相邻目录如何区分

`README.md` 不再默认承担目录结构展示，也不承担主要查找导航。

#### 触发信号

当一个目录需要被人类或 AI 当作说明入口使用，而仅靠目录名无法稳定判断其定位时，可考虑补 README，以解决目录说明问题。

以下现象通常提示 README 需要补强：

- 新读者进入目录后，无法判断这里的主题边界
- 上下级目录关系不清，材料容易放错位置
- 仅靠文件名或子目录名，无法判断这个目录为什么存在
- 同一目录内部已经形成稳定分类依据，但尚未在入口处说明

#### 不解决的问题

`README.md` 不应承接：

- 完整主题理解
- 系统性的研究结构图
- 目录结构展示（仅在明确不使用 `index.md` 时，README 才按需保留最小目录树）
- 主要查找导航
- 反向索引或关键词检索入口
- 长篇学科导论
- 详细论文列表

如需提供结构展示或查找导航，优先使用 `index.md`，不要把 `README.md` 写成扩展版导航页。

#### 与 index / overview / landscape 的边界

README 解决的是"这个目录是什么、为什么存在、和谁区分"，不解决"这个目录下面有什么"、"我已经知道想找什么时该去哪"、"这个主题最值得先理解什么"和"这个主题内部如何切分"。这些问题应分别由 `index.md`、`overview.md` 和 `landscape.md` 承接。README 很重要，但不是万能文件——它不应吞掉 index 的导航职责，也不应试图替代 overview 或 landscape。

### 4.2 `index.md`：目录结构与查找导航

`index.md` 回答"这个目录下面有什么？我已经知道想找什么时该去哪？"

#### 基线职责

`index.md` 负责目录结构展示与查找导航。它不是 README 的扩展版说明页，而是本目录的导航页。

它的最小稳定模型只有两部分：

- 目录结构：回答“这个目录下面有什么”
- 索引列表：回答“我在找某类内容时该去哪”

其中，索引列表可以很轻，只要能稳定把常见查找意图映射到对应子目录、文件或入口页即可；不要求每个目录都写成复杂的导航表。

在父子层级中，`index.md` 默认遵循“父层粗定位、子层细定位”的分工：

- 父层 `index.md` 主要把读者稳定路由到直接子目录
- 子层 `index.md` 再继续承接更细的文件级、对象级或更深子目录定位
- 父层跨层直链到更深层文件只应作为少量高频例外，不应成为主体

如果父层 `index.md` 的某一节已经主要由孙级或更深层链接构成，通常说明这一层开始替代子层 `index.md`，应优先收回到子目录级路由。

对于横向跨目录映射，同样不要把父层 `index.md` 写成高密度深链清单：默认优先链接到目标目录本身，或目标目录下一层足以稳定承接该问题的子目录；只有当更深层对象本身就是稳定公共锚点，且不直链会明显损失导航价值时，才保留为显式深链。若这些深链开始频繁失效、每次结构调整都需要同步回改，且读者已能通过目标目录的 `README.md` 或 `index.md` 稳定到达，就应考虑收缩为更粗粒度的目录级路由。

它通常承接：

- 目录结构展示
- 按问题找入口
- 按阅读目标找入口
- 不回答阅读顺序、学习阶段或推进路径
- 按子目录、主题类型或对象类型找入口
- 当读者不确定问题落在哪个板块时的最小判别语句

#### 触发信号

`index.md` 不是每个目录都必须启用的文件。只有当目录需要把"目录结构展示"与"查找导航"稳定交给同一文件时，才考虑启用 `index.md`。

以下现象通常提示应考虑 `index.md`：

- 读者需要先看目录结构，再决定往哪个板块走
- 读者知道自己要找什么，但 README 只能解释目录说明，不能直接完成查找
- 同一主题横跨多个子目录，难以靠 README 的说明文字稳定导航
- 需要按关键词、问题、阅读目标、对象类型，而不是只按层级浏览
- 目录说明、结构展示与导航职责已经开始在 README 中混杂

#### 不解决的问题

`index.md` 不应承接：

- 主题主线解释
- 技术判断或领域分析
- 目录存在理由
- 结构边界治理
- 长篇综述
- 阅读顺序、学习阶段或推进路径
- overview / landscape 式理解正文

与其他文件的区别：

- `README.md` 回答"这是什么地方、为什么存在"
- `index.md` 回答"这个目录下面有什么，以及我该去哪"
- `roadmap.md` 回答"我应按什么顺序推进"
- `overview.md` 回答"这个主题最值得先理解什么"

`index.md` 可以把不属于自己的问题转交给 `README.md`、`overview.md` 或 `roadmap.md`，但不应在本文件中直接代答这些问题。也就是说，它可以写“想先建立整体理解，请读 `overview.md`”或“不确定从哪开始，请读 `roadmap.md`”，但不应继续展开成整体主线说明、推荐起点、阅读顺序或路径建议。

### 4.3 `overview.md`：人类向主文

`overview.md` 回答"如果我只读一篇，这个主题最值得先理解的主线是什么？"

#### 基线职责

`overview.md` 是面向人类读者的主文，目标是让读者即使不继续阅读该目录下的其他文档，也能对该主题形成相对完整、可独立成立的整体理解。

它优先强调：

- 主题是什么
- 为什么重要
- 关键问题、主线脉络和阶段性判断
- 少量必要的代表路线、代表对象或关键分歧
- 适合对外分享或供维护者本人快速恢复认知的内容组织方式

#### 触发信号

当仅靠 README 和零散专题文档，读者仍无法稳定形成整体理解时，应考虑创建 `overview.md`。

以下现象通常提示应考虑 `overview.md`：

- 主题存在明确主线，但被分散在多个专题中
- 没有一篇文档能单独承接整体理解
- 读者必须逐篇拼装信息才能理解这个主题
- 目录已经不只是收纳材料，而是需要表达一个较稳定的主题判断

#### 不解决的问题

`overview.md` 可以与下层专题文档共享部分结论，但不应退化为目录结构说明，也不应承担查找导航或长期研究导航图的职责。

不适合放入 `overview.md` 的内容包括：

- 大量目录级分工说明
- 细致的板块边界治理
- 下层子专题如何协作的系统性说明
- 主要面向 AI / 维护者的研究路线图

如果一篇 `overview.md` 同时承担"人类向主文"和"结构化研究文"两种职责，应考虑拆分出 `landscape.md`。

### 4.4 `landscape.md`：结构化研究文

`landscape.md` 回答"这个主题内部有哪些板块、边界和研究路径？各部分如何分工与衔接？"

#### 基线职责

`landscape.md` 服务于结构化研究、文档体系构建、长期维护和下钻协作，帮助维护者与深入读者理解：

- 该主题内部的主要板块如何划分
- 相邻子主题、对象研究和辅助材料之间的边界
- 继续深入时应从哪些方向展开
- 下层文档、子目录和研究产物如何协作
- 为什么当前文档体系这样组织，而不是按另一条看似合理的轴组织

`landscape.md` 既服务 AI 和维护者，也应保持人类可读；它不是只给 AI 看的内部说明书。

适合放入 `landscape.md` 的内容包括：

- 板块划分与内部结构
- 研究地图与下钻路径
- 相邻主题边界
- 目录内各类文档的协作关系
- 为什么这样切分，而不是那样切分

#### 触发信号

`landscape.md` 不应作为 AI 的默认创建文件。通常只在目录已经出现稳定结构问题时，才考虑由维护者决定拆出它。

以下现象通常提示应考虑 `landscape.md`：

- `overview.md` 开始反复解释"本目录为什么这样切"
- 新材料的边界归属、板块划分和下钻路径反复引发犹豫
- 结构判断散落在 README、overview、对象研究和 notes 中
- 主题主线与研究结构开始互相干扰，无法稳定共存于一篇文档

#### 不解决的问题

`landscape.md` 不应承接：

- 面向外部读者的主线综述职责
- 另一篇与 `overview.md` 重复的独立长文
- README 式目录说明
- index 式目录结构展示或查找导航
- 纯任务清单或 backlog 替代物

`landscape.md` 的开头应优先回答"这个主题内部如何切分"。如果开头主要在回答"这个主题是什么、为什么重要"，那它更像 `overview.md`。

### 4.5 何时同时使用 `overview.md` 与 `landscape.md`

当一个目录同时存在以下两类需求时，应考虑同时保留 `overview.md` 与 `landscape.md`：

- 需要一篇相对独立、可单篇成立、可读性优先的人类向主文
- 同时又需要一篇更偏结构化研究的文档，用于说明板块、边界、下层分工和研究路径

以下现象通常提示应考虑拆分：

- 一篇文档同时在"讲主题本身"和"讲本目录如何组织"之间频繁切换
- 文档中出现大量目录级分工说明、边界治理或下钻路径说明
- 读者向主文与研究结构图明显开始互相干扰
- 为了兼顾可读性与结构说明，文档持续膨胀并出现职责混杂

拆分后应遵循：

- `overview.md` 保留人类向主线和相对独立的整体理解
- `landscape.md` 保留结构化研究、边界说明和协作关系

`landscape.md` 只应在出现实际需求时引入，不要因为目录层级多或内容多就机械创建。

## 5. 推进、选题与路径

### 5.1 `backlog.md`：内容缺口

`backlog.md` 记录该领域已被识别但尚未被覆盖的内容空白。

典型内容：

- 已知空白
- 待填补主题
- 关联主题或已有材料
- 现有相关资源
- 为何尚未覆盖
- 缺什么证据或研究对象

组织方式：

- 可按 P0 / P1 / P2 分组
- P0：已识别高价值
- P1：有价值但模糊
- P2：长期方向

它不是任务列表：不分配责任人、不设 deadline、不追踪完成状态。

与 `candidates.md` 的边界：

- `backlog.md` 记录问题缺口
- `candidates.md` 记录候选研究对象

### 5.2 `candidates.md`：候选研究对象队列

`candidates.md` 记录该领域下一步值得研究的候选对象。

对象包括：

- 论文
- GitHub 仓库
- 官方文档
- 产品
- 协议
- benchmark
- 高质量技术博客或社区讨论

每个条目建议包含：

- 对象名
- 对象类型
- 关联问题
- 为何值得研究
- 当前状态：待看 / 进行中 / 已完成 / 已转专题
- 研究产出或落位

按需创建：仅当对象队列已形成稳定维护需求，且不适合塞进 `backlog.md` 时使用。

不确定时的默认判断：先问"我现在掌握的是一个问题，还是一个对象？"问题进 `backlog.md`，对象进 `candidates.md`。

### 5.3 `roadmap.md`：路径指引

`roadmap.md` 回答"从哪开始、按什么顺序阅读或建设？"

`roadmap.md` 解决的是先后顺序与路径分流，不解决“我已经知道想找什么，该去哪”。它不仅承接线性的阅读顺序，也承接按读者背景、阅读目标或进入起点划分的路径指引。

`roadmap.md` 不是每个目录都需要启用的文件。只有当目录确实存在稳定的阅读顺序、学习路径或建设顺序时，才创建它。

适合：

- 学习路线图
- 阅读或建设顺序
- 阅读阶段
- 推荐顺序
- 前置知识或依赖
- 按读者背景推荐不同进入路径
- 按阅读目标推荐不同起点或分支路径

它不是项目管理工具：不设时间 deadline、不追踪完成状态。重点是"顺序"和"依赖"，而非"进度"。

一句话判断：`index.md` 解决“去哪”，`roadmap.md` 解决“从哪开始、怎么走”。

不要把“按读者背景推荐起点”或“按阅读目标分流路径”的内容误写进 `README.md` 或 `index.md`。

### 5.4 三者边界与常见混淆

backlog、candidates、roadmap 解决的是同一个方向性问题的不同侧面，但容易混淆。以下是它们的关键区别：

| 问题 | 归属 | 不应归入 |
|---|---|---|
| "这里还有哪些内容没覆盖？" | `backlog.md` | candidates（除非已明确需要跟踪具体对象） |
| "这个研究方向值得看哪些论文或项目？" | `candidates.md` | backlog（对象不是问题缺口） |
| "新来的人应该按什么顺序读？" | `roadmap.md` | README（README 负责目录说明，不是学习路径） |

**常见混淆处理：**

- **何时从 backlog 拆出 candidates**：当 backlog 中"待看某篇论文"或"待研究某个项目"这类对象条目明显多于"待填补的知识缺口"，并且对象队列已形成稳定维护需求时，拆出 `candidates.md`。
- **何时根本不需要 roadmap**：当目录没有稳定阅读顺序（例如对象研究目录、论文笔记集），或者目录只需要回答“这个目录下面有什么、我该去哪”而不需要回答“我应按什么顺序推进”时，不需要 roadmap。不要因为子目录多就创建一个 roadmap。
- **backlog 和 candidates 的条目可以互相引用**：一条 backlog 缺口可以列出关联 candidates 条目作为可能的填补方向；一条 candidates 条目可以说明它试图填补哪个 backlog 缺口。

不确定时的默认判断：先问"我现在掌握的是一个问题，还是一个对象，还是一个顺序？"问题进 backlog，对象进 candidates，顺序进 roadmap。

对象归属仍待判断时，优先记录为 `backlog.md` 中的待澄清问题。不要因为某个对象与某个上层方向相关，就直接写入该上层目录的 `candidates.md`——那会提前固化子归属。

## 6. 冲突与校验

### 6.1 `conflict.md`：冲突记录

`conflict.md` 用于记录当前所在层级目录范围内的冲突，以及其直接子目录之间（含跨子目录层级）的冲突。

仅在发现冲突时创建；无冲突则不创建该文件。

每条记录建议包含：

- 冲突描述
- 涉及文档路径
- 冲突类型：术语 / 事实 / 结论 / 版本 / 适用边界
- 当前证据
- 待核验口径或边界
- 建议处理方向

该文件是研究与修订输入，不是最终结论文档。

冲突解决后，应在相关文档修正，并同步更新或移除对应记录。

### 6.2 什么算冲突，什么不算

以下情况适合记录到 `conflict.md`：

- 同一概念在不同目录中被给出不同定义
- 同一实现在不同文档中结论相反
- 跨文档的术语、事实或统计数据存在矛盾

以下情况不适合记录到 `conflict.md`：

- 只是尚未覆盖或尚未验证——这类应进 `backlog.md`
- 只是未研究完的候选对象——这类应进 `candidates.md`
- 只是对某个判断存疑但尚无对立证据——这类应在正文中标记 `Unverified`

`conflict.md` 不替代正文修订。发现冲突后，应在正文中说明冲突双方，并在 `conflict.md` 中记录追踪状态，而不是直接选择一方作为定论。

## 7. 创建策略

### 7.1 默认入口

`README.md` 用于承接目录说明问题，但不是"所有知识目录都应优先存在"。

以下情况下不需要 README：

- **纯叶子目录**（仅含文件，无子目录可能）：不需要。由父目录 README 或文件名语义提供入口定位。
- **没有入口问题的目录**：如果目录名本身已足够自解释，没有读者走错或材料放错的历史，不需要为完整起见补一个 README。
- **不需要承担目录级语义接口的场景**：例如临时容器、过渡态目录、纯论文笔记集等。

README 的核心价值是解决目录说明问题，不是满足形式完整。如果某个目录不存在说明困惑，强制补 README 的意义很小。

按需出现的文件清单：

- `overview.md`、`landscape.md`、`backlog.md`、`candidates.md`、`roadmap.md`、`conflict.md`、`index.md`

### 7.2 按需能力位

以下文件有明确的触发条件，不应在无实际需求时创建：

- `index.md`、`roadmap.md`：预定义能力位，仓库允许存在，但不要为每个目录机械补齐
- `landscape.md`：不应作为 AI 的默认创建文件。仅当 overview 已无法承载结构研究职责时，由维护者决定是否拆分
- `candidates.md`：仅当对象队列已形成稳定维护需求，且不适合塞进 `backlog.md` 时使用

### 7.3 什么时候不该创建新文件

- 不要为每个目录机械创建所有七层文件
- 不要因为父目录有某个文件，子目录就跟着建
- 不要因为"未来可能有用"而提前创建
- 不要因为模型中有某个槽位就创建一份空文件占位
- 不要机械同时创建 `overview.md` 与 `landscape.md`——只有当人类向主文与结构化研究文确实分离时，才同时保留两者
- `evidence.md` / `evidence-registry.md` 不是默认元信息文件；只有证据对照本身成为稳定维护对象时，才按主题启用

### 7.4 什么时候该拆分已有文件

- 当一篇文档同时承担"人类向主文"和"结构化研究文"两种职责时，考虑拆分为 `overview.md` + `landscape.md`
- 当一篇 overview 开始大量解释"本目录为什么这样切"时，考虑拆出 `landscape.md`
- 当 `backlog.md` 中的条目类型已经明显区分为"问题"和"对象"时，考虑拆出 `candidates.md`

### 7.5 拿不准时的默认处理

- 拿不准是否该新建正文专题时，不要直接补一个元信息文件了事。必须先判断当前问题属于目录说明、查找导航、内容缺口还是口径冲突，再分别落到 `README.md`、`index.md`、`backlog.md` 或 `conflict.md`
- 拿不准是问题还是对象时，问题进 `backlog.md`，对象进 `candidates.md`
- 拿不准是否存在冲突时，先记录到 `conflict.md`，不要直接改写正文定论
- 拿不准是否需要 overview 时，先确认该目录是否已形成独立主线判断；如果没有，README 已足够

## 8. 命名补充

### 8.1 `overview.md`

当一个目录只需要一篇"理解型主文"时，统一使用 `overview.md`。

### 8.2 `landscape.md`

当一个目录除了 `overview.md` 之外，还需要一篇结构化研究文时，统一使用 `landscape.md`。

- `overview.md`：相对独立的人类向总览主文
- `landscape.md`：结构化文档体系构建文

### 8.3 `summary.md` 废弃

`summary.md` / `SUMMARY.md` 已废弃；目录级全貌文件统一使用 `overview.md`。

## 附录：单篇文档的头尾信息布局建议

本附录只回答：哪些信息前置，哪些信息保留文末。

### 前置到标题下方

前置读者一开始就需要知道的信息。

- 领域综述或一般研究文档：`适用范围`、`阶段状态`、`使用说明`、必要时的一句话定位
- 论文研究文档：标题、作者、机构、发布时间、论文链接，以及开篇就需要判断的对象信息
- 开源仓库、产品、框架、协议等对象研究文档：对象名称、对象类型、上游地址、研究范围、必要时的版本或观察时间

### 保留在文末

默认保留维护、追溯和证据判断信息，例如：`Evidence`、`Trace`、`Needs`、`最后更新`。

### 不要机械统一

- 不要把所有尾注信息都前置到标题下方
- 不要把所有头信息都堆到文末
- 不要不区分文档类型强行套同一模板

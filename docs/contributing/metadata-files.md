# 元信息文件模型

本文件定义仓库中元信息文件的职责边界、出现条件和使用方式。

相关入口：[`CONTRIBUTING.md`](../../CONTRIBUTING.md) / [`docs/contributing/README.md`](./README.md) / [`README 规则`](./readme-rules.md) / [`Evidence 规则`](./evidence-rules.md) / [`Traceability 规则`](./traceability-rules.md)。

元信息文件不是知识正文，而是帮助读者和维护者完成定向、发现、理解、推进、选题、规划、校验等认知任务的辅助文件。

## 1. 使用边界

本文件回答：仓库中哪些文件属于元信息文件、它们分别服务什么认知任务、应该在什么情况下出现。

- 它是参考手册，用于查清 `README.md`、`overview.md`、`backlog.md`、`candidates.md` 等文件的职责边界。
- 它不负责给出材料处理流程；拿到新材料后如何分级、分流和落地，应查看 [`documentation-workflow.md`](./documentation-workflow.md)。
- 它不讨论单篇正文如何写得更清楚；那属于写作表达规则。
- 它也不把项目研究、论文研究或源码核验记录定义为新的元信息层；这些属于正文或研究笔记的内部组织问题，应按 [`research-artifacts.md`](./research-artifacts.md) 中案例研究辅助材料的规则处理。

如果只是想判断“这个目录下应该有哪些元信息文件”，读本文件。

如果手上已经有一份材料，想知道下一步怎么处理，应查看 [`材料处理与文档增改流程`](./documentation-workflow.md)。

## 2. 七层模型

除笔记正文外，目录中的元信息文件按以下七层模型组织。每层对应一种读者认知任务，按需出现，不强制全部存在。

其中 `index.md`、`roadmap.md` 属于**预定义能力位**：仓库允许它们存在，也保留统一命名和职责边界；但只有当目录确实出现发现、路径指引需求时才创建。不要因为模型中定义了 L2/L6，就为每个目录机械补齐。

Evidence registry 不属于七层元信息文件模型的默认层级。证据优先写在被支撑内容所在文档或条目内部；只有当某个主题的证据对照本身成为稳定维护对象时，才按 [`Evidence 规则`](./evidence-rules.md) 启用主题级 registry。

同理，项目研究中的 `notes.md`、`notes/`、`source-notes.md`、`evidence-notes.md`、源码核验记录、论文细读笔记等也不是七层元信息文件。它们属于具体案例研究的辅助材料层，具体使用方式见 [`research-artifacts.md`](./research-artifacts.md) 第 5 节。

| 层级 | 认知任务 | 文件名 | 职责 | 出现条件 | 典型内容 |
|:---|:---|:---|:---|:---|:---|
| L1 定向 | Orient | `README.md` | “这是什么地方？跟我有什么关系？” | 目录存在或未来可能容纳子目录时必选；纯叶子目录（仅含文件，无子目录可能）不需要 | 一句话定位、快速入口、目录树 |
| L2 发现 | Discover | `index.md` | “我要的文件/主题在哪？” | 可选；文件/子目录 > ~10，或主题交叉关系复杂时才创建 | 关键词索引、反向映射、主题-文档对照表 |
| L3 理解 | Understand | `overview.md` | “这个领域的全貌和现状是什么？” | 该目录代表一个需要建立整体认知的领域 | 技术分类、代表工作、能力边界、关键 trade-off |
| L4 推进 | Track | `backlog.md` | “这个领域还有什么没被覆盖？” | 目录处于活跃建设期，内容有明显缺口 | 已知空白、待填补主题、关联主题或已有材料 |
| L5 选题 | Queue | `candidates.md` | “接下来值得研究哪些对象？” | 可选；需要持续跟踪论文、仓库、官方文档、产品、协议、benchmark 等候选研究对象，且对象列表不宜塞进 `backlog.md` 时创建 | 待研究对象、对象类型、关联问题、研究理由、当前状态、研究产出或落位 |
| L6 规划 | Plan | `roadmap.md` | “从哪开始、按什么顺序阅读/建设？” | 可选；有明确的内容组织顺序或学习路径时才创建 | 阅读阶段、推荐顺序、前置知识或依赖 |
| L7 校验 | Reconcile | `conflict.md` | “这里是否存在定义冲突或口径不一致？” | 仅当发现冲突时可选创建 | 冲突点、涉及文档、冲突类型、待核验口径或边界 |

## 3. 命名约定

L3 的标准文件名为 `overview.md`。

`summary.md` / `SUMMARY.md` 已废弃。

原因是 `summary` 语义过载，既可指内容摘要，也可指领域综述；`overview` 明确指向“领域全貌”。

若目录下已有内容摘要性质的简短文件，应扩展为领域全貌描述后再使用 `overview.md`。

## 4. L1 README：定向

`README.md` 回答“这是什么地方？跟我有什么关系？”

典型内容：

- 一句话定位：本目录是什么、面向谁
- 快速入口：按背景推荐路径、核心文档链接
- 目录结构：树状图，遵循 README 规则
- 同目录导航：优先列出正文主题、子目录入口或阅读路径

禁止放入：

- 详细技术分类
- 论文列表
- 长篇学科导论

如需提供导航，不要在 README 中主动列出 `overview.md`、`backlog.md`、`candidates.md`、`roadmap.md`、`conflict.md` 等元信息文件。

## 5. L2 index：发现

`index.md` 回答“我要的文件/主题在哪？”

`index.md` 是可选文件，不是当前每个目录都需要启用的默认配置。只有当 README 目录树不足以支持快速定位时，才创建它。

使用方式：

- 当层级目录树不足以支持快速定位时使用
- 按关键词、主题领域或反向映射扁平聚合，而非层级展开
- 跨学科主题可建立“技术领域 → 关联学科视角”的反向映射

与 `overview.md` 的区别：

- `index.md` 只回答“在哪”
- `overview.md` 回答“是什么”和“为什么重要”

`index.md` 禁止放入概念解释、技术判断、领域分析。

## 6. L3 overview：理解

`overview.md` 回答“这个领域的全貌和现状是什么？”

它是面向读者的综述或领域概览，目标是让不想逐篇阅读所有关联文档的人，也能对该领域或子领域形成相对完整的理解。虽然它属于目录级辅助文件，但内容重心应放在领域本身，而不是文档体系建设、目录维护过程或仓库治理说明。

典型内容：

- 技术分类
- 代表工作
- 能力边界
- 关键 trade-off
- 领域间关系与边界（可以讲概念层面的依赖和影响，不逐目录展开职责分工）
- 领域覆盖与成熟度判断
- 对关键结论的简要归纳（可与专项文档重复，但应保持更高层、更简练）

注意：`overview.md` 讲的是技术或领域本身的全貌，不是目录结构说明、目录体系分工说明或跨目录导航，也不是 backlog、候选对象队列或冲突登记。`overview.md` 可以讲领域间的概念依赖（如 foundations 的概念定义被下游领域引用），但不逐目录解释每个下游目录的职责分工；后者由 `README.md` 承担。

允许 `overview.md` 与其他正文文档对同一组结论做重复解释，只要这种重复服务于综述的可读性和相对完整性。区别不在于“是否能重复”，而在于详略层级：`overview.md` 负责用更简练的方式给出整体判断与核心脉络，专项文档则负责展开证据、机制细节、边界条件和例外情况。

相关但不属于 `overview.md` 的需求应按认知任务分流：定向和快速入口放 `README.md`；主题查找和反向映射按需放 `index.md`；内容缺口放 `backlog.md`；候选研究对象放 `candidates.md`；术语、事实或边界冲突放 `conflict.md`；稳定阅读或建设顺序按需放 `roadmap.md`。

读者在 `overview.md` 中获得领域认知后，再去 README、index 或目录树中定位具体文件。

## 7. L4 backlog：内容缺口

`backlog.md` 记录该领域已被识别但尚未被覆盖的内容空白。

典型内容：

- 已知空白
- 待填补主题
- 关联主题或已有材料
- 现有相关资源
- 为何尚未覆盖
- 缺什么证据或研究对象

组织方式：

- 可按 P0 / P1 / P2 分组
- P0：已识别高价值
- P1：有价值但模糊
- P2：长期方向

它不是任务列表：不分配责任人、不设 deadline、不追踪完成状态。

与 `candidates.md` 的边界：

- `backlog.md` 记录问题缺口
- `candidates.md` 记录候选研究对象

## 8. L5 candidates：候选研究对象队列

`candidates.md` 记录该领域下一步值得研究的候选对象。

对象包括：

- 论文
- GitHub 仓库
- 官方文档
- 产品
- 协议
- benchmark
- 高质量技术博客或社区讨论

每个条目建议包含：

- 对象名
- 对象类型
- 关联问题
- 为何值得研究
- 当前状态：待看 / 进行中 / 已完成 / 已转专题
- 研究产出或落位

按需创建：仅当对象队列已形成稳定维护需求，且不适合塞进 `backlog.md` 时使用。

不确定时的默认判断：先问“我现在掌握的是一个问题，还是一个对象？”问题进 `backlog.md`，对象进 `candidates.md`。

## 9. L6 roadmap：路径指引

`roadmap.md` 回答“从哪开始、按什么顺序阅读或建设？”

`roadmap.md` 是可选文件，不是当前每个目录都需要启用的默认配置。只有当目录确实存在稳定的阅读顺序、学习路径或建设顺序时，才创建它。

适合：

- 学习路线图
- 阅读或建设顺序
- 阅读阶段
- 推荐顺序
- 前置知识或依赖

它不是项目管理工具：不设时间 deadline、不追踪完成状态。重点是“顺序”和“依赖”，而非“进度”。

## 10. L7 conflict：冲突记录

`conflict.md` 用于记录当前所在层级目录范围内的冲突，以及其直接子目录之间（含跨子目录层级）的冲突。

仅在发现冲突时创建；无冲突则不创建该文件。

每条记录建议包含：

- 冲突描述
- 涉及文档路径
- 冲突类型：术语 / 事实 / 结论 / 版本 / 适用边界
- 当前证据
- 待核验口径或边界
- 建议处理方向

该文件是研究与修订输入，不是最终结论文档。

冲突解决后，应在相关文档修正，并同步更新或移除对应记录。

## 11. 默认策略

- 不要为每个目录机械创建所有 L1-L7 元信息文件。
- 元信息文件按需出现，服务具体认知任务。
- `index.md`、`roadmap.md` 是预定义能力位，不是默认必建文件。
- `evidence.md` / `evidence-registry.md` 不是默认元信息文件；只有证据对照本身成为稳定维护对象时，才按主题启用。
- 拿不准是否该新建正文专题时，优先补 README、backlog 或 conflict。
- 拿不准是问题还是对象时，问题进 `backlog.md`，对象进 `candidates.md`。
- 拿不准是否存在冲突时，先记录到 `conflict.md`，不要直接改写正文定论。

## 12. 附：单篇文档的头尾信息布局建议

本节不是七层元信息文件模型的一部分，而是与文档类型相关的补充布局建议。这里讨论的不是 `README.md`、`overview.md`、`backlog.md` 这类**元信息文件**本身承担什么职责，而是单篇文档内部的前置信息与文末尾注如何摆放。

并非所有文档都应采用相同的文档头信息与尾注信息布局。判断某条信息应当前置还是后置，取决于它是在帮助读者**识别研究对象/理解文档用途**，还是在帮助维护者**追溯证据、更新内容**。

### 12.1 适合前置到标题下方的文档头信息

适合前置的，是读者一开始就需要知道的信息。

**对于领域综述或一般性研究文档**（如子领域 `overview.md`），优先前置：

- 适用范围
- 阶段状态
- 使用说明
- 必要时的一句话定位

**对于论文研究文档**，优先前置：

- 论文标题
- 中文标题（如有）
- 作者
- 机构
- 发布时间
- 论文链接
- 页数、版本等读者开篇就需要判断的对象信息

**对于开源仓库、产品、框架、协议等对象研究文档**，优先前置：

- 对象名称
- 对象类型
- 上游地址（GitHub / 官网 / 官方文档）
- 研究范围
- 当前版本或观察时间（如确实重要）

原则：如果某条信息是在帮助读者回答“我正在读的对象是谁”“这篇文档是干什么的”，它更适合作为文档头信息前置。

### 12.2 适合保留在文末的尾注信息

适合后置的，是维护、追溯和证据判断信息，例如：

- `Evidence`
- `Trace`
- `Needs`
- `最后更新`

这些信息对“开始阅读正文”通常不是刚需，但对“判断这篇文档能否直接引用、是否需要继续补证”非常重要，因此默认保留文末。

### 12.3 不要机械统一所有文档的头尾信息布局

以下做法都不推荐：

- 把所有文档尾注信息都前置到标题下方
- 把所有文档头信息都堆到文末
- 不区分领域综述文档与对象研究文档，强行套同一模板

更稳妥的做法是：

- **领域综述型文档**：前置“怎么读”
- **论文 / 对象研究型文档**：前置“研究对象是谁”
- **证据与维护型元信息**：默认保留文末

# 元信息文件模型

本文件定义仓库中元信息文件的职责边界、出现条件和最小使用方式。

相关入口：[`CONTRIBUTING.md`](../../CONTRIBUTING.md) / [`docs/contributing/README.md`](./README.md) / [`README 规则`](./readme-rules.md) / [`Index 规则`](./index-rules.md) / [`Evidence 规则`](./evidence-rules.md) / [`Traceability 规则`](./traceability-rules.md)。如需理解这些边界为什么这样设计、最容易被误读成什么，读 [`intent/metadata-files.md`](./intent/metadata-files.md)。

元信息文件不是知识正文，而是帮助读者和维护者完成说明、导航、理解、结构、推进、选题、路径、校验等认知任务的辅助文件。

## 快速判断

如果你只是想快速判断“这个目录现在该不该建某个元信息文件”，先按下面顺序判断：

- 先判断当前问题属于哪一类：**说明、导航、理解、结构、缺口、对象、路径、冲突**。
- 对应文件分别是：`README.md`、`index.md`、`overview.md`、`landscape.md`、`backlog.md`、`candidates.md`、`roadmap.md`、`conflict.md`。
- 不要机械补齐所有文件；文件按问题触发，不按层级补齐。
- `README.md` 负责目录说明，不再默认承担目录结构展示；`index.md` 负责目录结构与查找导航；`landscape.md` 不是默认创建文件。
- 拿不准是否该新建正文专题时，先不要急着写正文；先判断问题属于目录说明、查找导航、内容缺口、对象队列、结构边界还是口径冲突，再分别落到 `README.md`、`index.md`、`backlog.md`、`candidates.md`、`landscape.md` 或 `conflict.md`。
- 需要细看边界时：`README / overview / landscape` 看第 4 章，`index.md` 的专项细则读 [`index-rules.md`](./index-rules.md)，`backlog / candidates / roadmap` 看第 5 章，创建策略看第 7 章。

## 1. 使用边界

本文件只回答三件事：哪些文件属于元信息文件、它们分别解决什么问题、在什么情况下出现。

- 用它判断 `README.md`、`index.md`、`overview.md`、`landscape.md`、`backlog.md`、`candidates.md`、`roadmap.md`、`conflict.md` 的职责边界和出现条件。
- 材料处理流程见 [`documentation-workflow.md`](./documentation-workflow.md)。
- 项目研究辅助材料（`notes.md`、`notes/`、`source-notes.md`、`evidence-notes.md` 等）不在本文，见 [`research-artifacts.md`](./research-artifacts.md)。
- Evidence registry 不是默认元信息文件；仅在证据对照本身成为稳定维护对象时启用，见 [`evidence-rules.md`](./evidence-rules.md)。
- 单篇文档表达、头尾字段布局不在本文。

## 2. 核心原则

- **元信息文件服务认知任务，不服务形式对称。** 不需要为每个目录机械补齐所有文件。
- **文件按问题触发，不按层级补齐。** 不要因为父目录有某份文件，子目录就必须同样有。
- **一个文件只承接一个主问题。** 同一文件里同时回答多个主问题时，优先考虑拆分。
- **先明确分工，再稳定命名。** 命名服务于分工，不能反过来用命名推分工。
- **`index.md` 与 `roadmap.md` 都是允许存在的能力位，但触发方式不同。** `index.md` 在目录下已有两个及以上语义稳定、预期会持续保留的子目录时应优先考虑创建；`roadmap.md` 仍只在存在稳定阅读顺序、学习路径或建设顺序时创建。`index.md` 的专项细则见 [`index-rules.md`](./index-rules.md)。

## 3. 总览：文件与认知任务对照

下表是元信息文件的快速对照工具，不是所有目录都必须机械补齐的模板。

| 层级 | 认知任务 | 文件名 | 职责 | 出现条件 | 典型内容 |
|:---|:---|:---|:---|:---|:---|
| L1 说明 | Explain | `README.md` | “这是什么地方？为什么存在？和相邻目录如何区分？” | 目录需要被当作入口说明对象时启用；纯叶子目录（仅含文件，无子目录可能）不需要 | 一句话定位、入口说明、分类依据、边界说明、相邻目录关系 |
| L2 导航 | Navigate | `index.md` | “这个目录下面有什么？我已经知道想找什么时该去哪？” | 默认在目录下已有两个及以上语义稳定、预期会持续保留的子目录时启用；未达到这条基线时按需创建 | 目录结构、直接子目录定位、按问题找入口、按阅读目标找入口、按子目录或对象类型找入口 |
| L3 理解 | Understand | `overview.md` | “如果我只读一篇，这个主题最值得先理解的主线是什么？” | 该目录已积累出足以支撑相对独立人类向主文的材料与主线判断 | 主线脉络、关键判断、整体理解、相对独立的主文 |
| — | 结构化研究 | `landscape.md` | “这个主题内部有哪些板块、边界和研究路径？各部分如何分工与衔接？” | 当结构边界、归属判断或下钻路径先稳定下来时，或 overview 与结构研究职责已需要分离时；不是默认创建文件 | 板块划分、研究地图、下钻路径、文档协作关系 |
| L4 推进 | Track | `backlog.md` | “这个领域还有什么没被覆盖？” | 目录处于活跃建设期，且已出现需要持续记录的方向缺口、待补主题或问题簇时 | 已知空白、待填补主题、关联主题或已有材料 |
| L5 选题 | Queue | `candidates.md` | “接下来值得研究哪些对象？” | 已有一批值得持续跟踪的候选研究对象，且对象队列不宜塞进 `backlog.md` 时创建 | 待研究对象、对象类型、关联问题、研究理由、当前状态、研究产出或落位 |
| L6 规划 | Plan | `roadmap.md` | “从哪开始、按什么顺序阅读/建设？” | 有明确的内容组织顺序、学习路径或前置依赖时创建 | 阅读阶段、推荐顺序、前置知识或依赖 |
| L7 校验 | Reconcile | `conflict.md` | “这里是否存在定义冲突或口径不一致？” | 仅当发现冲突时创建 | 冲突点、涉及文档、冲突类型、待核验口径或边界 |

`landscape.md` 不在 L1-L7 编号体系中；它是按需存在的结构化研究文。

## 4. 入口、发现与理解

### 4.1 `README.md`

- **职责**：回答“这是什么地方、为什么存在、和相邻目录如何区分”，承接目录说明与最小入口分流。
- **触发信号**：目录需要被人类或 AI 当作说明入口使用，而仅靠目录名无法稳定判断其定位。
- **不解决的问题**：完整主题理解、系统性结构图、主要查找导航、学习路径、长篇综述。
- **边界提醒**：`README.md` 不再默认承担目录结构展示。目录下已有两个及以上语义稳定、预期会持续保留的子目录时，结构展示应优先迁到骨架型 `index.md`。

### 4.2 `index.md`

- **职责**：回答“这个目录下面有什么？我已经知道想找什么时该去哪？”，承接目录结构展示与查找导航。
- **触发信号**：目录下已有两个及以上语义稳定、预期会持续保留的子目录时，应优先考虑创建一个骨架型 `index.md`；当读者知道自己要找什么，但 README 不能直接定位时，也应考虑创建。
- **不解决的问题**：主题主线解释、目录存在理由、结构边界治理、阅读顺序、学习阶段或推进路径。
- **边界提醒**：`index.md` 解决“去哪”，`roadmap.md` 解决“从哪开始、怎么走”。更细的结构模型、父子层分工与 stop-line，读 [`index-rules.md`](./index-rules.md)。

### 4.3 `overview.md`

- **职责**：回答“如果我只读一篇，这个主题最值得先理解的主线是什么？”，承接相对独立的人类向主文。
- **触发信号**：仅靠 README、现有研究材料与结构化支架，读者仍无法稳定形成整体理解，且该目录已经积累出足以支撑一篇相对独立主文的材料与主线判断。
- **不解决的问题**：目录结构说明、查找导航、板块切分治理、长期研究导航图。
- **边界提醒**：如果一篇 `overview.md` 开始大量解释“本目录为什么这样切”或“各板块如何协作”，应考虑拆出 `landscape.md`。

### 4.4 `landscape.md`

- **职责**：回答“这个主题内部有哪些板块、边界和研究路径？各部分如何分工与衔接？”，承接结构化研究与文档体系协作。
- **触发信号**：目录首先需要稳定板块切分、边界归属与研究下钻路径；或这些问题已经与 `overview.md` 的主线职责发生混杂。
- **不解决的问题**：面向外部读者的主线综述、README 式目录说明、index 式结构展示或查找导航、缺口账本、对象队列、明确的顺序型建设计划。
- **边界提醒**：`landscape.md` 不是默认创建文件；如果开头主要在回答“这个主题是什么、为什么重要”，那它更像 `overview.md`。

### 4.5 何时同时使用 `overview.md` 与 `landscape.md`

当一个目录同时需要：

- 一篇相对独立、可单篇成立的人类向主文
- 一篇说明板块、边界、下层分工与研究路径的结构化研究文

可同时保留 `overview.md` 与 `landscape.md`。

常见拆分信号：

- 一篇文档同时在“讲主题本身”和“讲本目录如何组织”之间频繁切换
- 文档中出现大量目录级分工说明、边界治理或下钻路径说明
- 为了兼顾可读性与结构说明，文档持续膨胀并出现职责混杂

## 5. 推进、选题与路径

### 5.1 `backlog.md`

- **职责**：记录该领域已被识别但尚未被覆盖的内容缺口。
- **适合内容**：已知空白、待填补主题、关联主题或已有材料、为何尚未覆盖、缺什么证据或研究对象。
- **不解决的问题**：对象队列、责任分配、deadline、完成状态。
- **边界提醒**：`backlog.md` 记录问题缺口，不记录候选研究对象。

### 5.2 `candidates.md`

- **职责**：记录该领域下一步值得研究的候选对象。
- **适合内容**：论文、仓库、官方文档、产品、协议、benchmark、值得持续跟踪的高质量外部对象。
- **创建条件**：对象队列已形成稳定维护需求，且不适合继续塞进 `backlog.md`。
- **边界提醒**：不确定时先问“我现在掌握的是一个问题，还是一个对象？”问题进 `backlog.md`，对象进 `candidates.md`。

### 5.3 `roadmap.md`

- **职责**：回答“从哪开始、按什么顺序阅读或建设？”，承接路径指引。
- **适合内容**：学习路线图、阅读或建设顺序、阶段划分、前置知识或依赖、按读者背景或阅读目标分流的进入路径。
- **不解决的问题**：目录结构展示、查找导航、项目进度追踪。
- **边界提醒**：`index.md` 解决“去哪”，`roadmap.md` 解决“从哪开始、怎么走”。

### 5.4 三者边界速查

| 问题 | 归属 | 不应归入 |
|---|---|---|
| “这里还有哪些内容没覆盖？” | `backlog.md` | `candidates.md`（除非已明确需要跟踪具体对象） |
| “这个研究方向值得看哪些论文或项目？” | `candidates.md` | `backlog.md` |
| “新来的人应该按什么顺序读？” | `roadmap.md` | `README.md` / `index.md` |

默认判断：问题进 `backlog.md`，对象进 `candidates.md`，顺序进 `roadmap.md`。

## 6. 冲突与校验

### 6.1 `conflict.md`

- **职责**：记录当前所在层级目录范围内的冲突，以及其直接子目录之间的冲突。
- **创建条件**：仅在发现冲突时创建；无冲突则不创建。
- **每条记录建议包含**：冲突描述、涉及文档路径、冲突类型、当前证据、待核验口径或边界、建议处理方向。
- **边界提醒**：`conflict.md` 是研究与修订输入，不是最终结论文档；冲突解决后，应在相关文档修正，并同步更新或移除对应记录。

### 6.2 什么算冲突，什么不算

适合记录到 `conflict.md`：

- 同一概念在不同目录中被给出不同定义
- 同一实现在不同文档中结论相反
- 跨文档的术语、事实或统计数据存在矛盾

不适合记录到 `conflict.md`：

- 只是尚未覆盖或尚未验证：进 `backlog.md`
- 只是未研究完的候选对象：进 `candidates.md`
- 只是对某个判断存疑但尚无对立证据：在正文中标记 `Unverified`

## 7. 创建策略

### 7.1 什么时候不需要新文件

- 不要为每个目录机械创建所有元信息文件。
- 不要因为父目录有某个文件，子目录就跟着建。
- 不要因为“未来可能有用”而提前创建。
- 不要因为模型中有某个槽位就创建一份空文件占位。
- `evidence.md` / `evidence-registry.md` 不是默认元信息文件；只有证据对照本身成为稳定维护对象时，才按主题启用。

### 7.2 什么时候应优先创建

- `README.md`：目录存在入口说明问题时创建。
- `index.md`：目录下已有两个及以上语义稳定、预期会持续保留的子目录时，优先考虑创建骨架型 `index.md`。
- `landscape.md`：当结构边界、归属判断或下钻路径先稳定下来时，可较早作为研究支架出现。
- `candidates.md`：对象队列已形成稳定维护需求，且不适合继续塞进 `backlog.md` 时创建。

### 7.3 什么时候该拆分已有文件

- 一篇文档同时承担“人类向主文”和“结构化研究文”两种职责时，考虑拆分为 `overview.md` + `landscape.md`。
- 一篇 `overview.md` 开始大量解释“本目录为什么这样切”时，考虑拆出 `landscape.md`。
- `backlog.md` 中的条目类型已经明显区分为“问题”和“对象”时，考虑拆出 `candidates.md`。

### 7.4 拿不准时的默认处理

- 拿不准是否该新建正文专题时，不要直接补一个元信息文件了事；先判断当前问题属于目录说明、查找导航、内容缺口、对象队列、结构边界还是口径冲突。
- 拿不准是问题还是对象时，问题进 `backlog.md`，对象进 `candidates.md`。
- 拿不准是否首先需要稳定结构时，优先考虑 `landscape.md` 作为研究支架，而不是急着写 `overview.md`。
- 拿不准是否存在冲突时，先记录到 `conflict.md`，不要直接改写正文定论。
- 拿不准是否需要 `overview.md` 时，先确认该目录是否已形成独立主线判断；如果没有，README、backlog、candidates 或 landscape 往往已经足够。

## 8. 命名补充

- 当一个目录只需要一篇理解型主文时，统一使用 `overview.md`。
- 当一个目录需要一篇结构化研究文时，统一使用 `landscape.md`。
- `summary.md` / `SUMMARY.md` 已废弃；目录级全貌文件统一使用 `overview.md`。

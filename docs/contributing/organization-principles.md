# 文档组织原则

本文件回答的问题不是“某篇笔记该写到哪里”，也不是“某次重构怎样执行”，而是：**这个仓库为什么要这样组织知识，什么样的目录结构才算长期合理，判断目录设计优劣时应优先看什么。**

它沉淀的是上位组织原则，用于指导：

- 新建目录前的结构判断
- 既有目录是否需要重构
- 子领域内部如何建立稳定分类轴
- 为什么某些目录应该保留，某些目录应该拆分、改名或降级为辅助材料

相关入口：[`CONTRIBUTING.md`](../../CONTRIBUTING.md) / [`结构重构规则`](./structure-refactoring-rules.md) / [`README 规则`](./readme-rules.md) / [`元信息文件模型`](./metadata-files.md) / [`研究产物组织规则`](./research-artifacts.md)

## 1. 长期主义优先于当前内容状态

目录结构首先服务于**未来内容增长后的可维护性**，而不是当前的文件数量、空目录比例或暂时没有冲突的状态。

因此：

- 不能用“当前内容还少”来证明结构合理
- 不能把“等以后内容多了再改”当默认策略
- 应优先判断某个命名、分类轴或目录边界在未来扩张数倍后是否仍然稳定

评价一个目录是否合理，优先问：

1. 如果未来内容增长数倍，这个目录名是否仍能承载？
2. 如果现在不改，未来是否会产生高成本二次迁移？
3. 这个目录是否会误导后续写作者把内容放错位置？

## 2. 目录结构首先服务写作者的正确落位判断

目录结构不是静态展示，而是持续影响写作者判断的“语义接口”。

一个好的知识树，应让后续维护者在不熟悉历史上下文的情况下，也能较高概率做出正确落位判断。

因此：

- 目录名应提供稳定、低歧义的语义暗示
- 边界不够直观时，应在 README 中明确“放在这里 / 不放在这里”
- 容易跨域的主题，应通过交叉引用和边界说明降低误放概率

目录设计的目标不是让当前维护者看懂，而是让未来维护者也能自然延续结构。后续各条原则，本质上都服务于这件事。

## 3. 分类轴一致性优先于局部便利

同一层级尽量只采用一种主分类轴。目录结构最忌讳的，不是“某个名字不够好看”，而是**同层级并列目录在回答不同问题**。

常见主分类轴包括：

- 问题域
- 机制类型
- 对象类型
- 环境类型
- 评估维度

当同一层级同时混入多种分类轴时，后续内容会越来越难落位，目录之间的边界也会越来越模糊。

如何判断和治理分类轴问题，见 [`structure-refactoring-rules.md`](./structure-refactoring-rules.md) 第 5 节。

## 4. 语义边界比当前内容数量更重要

一个目录是否成立，首先取决于它的**语义边界是否清楚、稳定、可持续**，而不是当前已经放了多少内容。

一个好的目录名应当：

- 足够明确，能让写作者知道什么应放这里
- 足够稳定，未来不因主题稍微扩展就必须改名
- 足够克制，不会无限吞并相邻目录

当目录名本身无法承载全部定义时，应通过 README 或 overview 补充语义界定。

如何区分“过窄”“过宽”与“目录名无法独立承载边界”的情况，见 [`structure-refactoring-rules.md`](./structure-refactoring-rules.md) 第 3 节。

## 5. 知识目录、对象目录、材料目录必须分离

仓库中的不同目录，承担的是不同类型的组织职责：

- **知识目录**：组织问题域、机制、方法、环境、评估维度等知识主题
- **对象目录**：组织框架、工具、产品、项目、协议等现实对象
- **材料目录**：保存论文笔记、图片、临时 notes、证据摘录、网页抓取等辅助材料

这三类目录不应在同一层级、以同一种语义身份长期并列。

如何区分知识目录、对象目录与材料目录，以及材料目录何时应从主分类树中降级，见 [`structure-refactoring-rules.md`](./structure-refactoring-rules.md) 第 5 节与第 6 节。

## 6. 高迁移成本问题要前置解决

结构设计时，应主动寻找那些**现在不改、以后代价会显著升高**的问题。

默认原则：**优先处理未来会导致大规模路径修改、导航重写和认知重建的问题。**

如何识别和处理高迁移成本点，见 [`structure-refactoring-rules.md`](./structure-refactoring-rules.md) 第 3 节。

## 7. 对象中心与能力中心要分清

有些领域应按**知识问题**组织，有些领域应按**现实对象**组织。两者都合理，但不能混淆。

例如：

- `02-single-agent/`、`03-multi-agent/`、`05-environments/` 更接近问题域或机制组织
- `06-frameworks-and-tools/` 更接近 object-centric 组织

当一个目录被定义为 object-centric 时，它的主骨架应围绕框架、工具、产品、项目、系统类型等对象来建，而不应逐渐退化为 planning / memory / tool-use / reflection 这类能力 taxonomy。

相关能力分析可以进入对象内部，但不应反过来重建一套能力主骨架。更具体的边界落实，应写入对应领域 README / overview，而不是全部上升为仓库通用规则。

## 8. README 负责表达骨架，不负责承载全部历史

README 的职责是定向与导航，不是保存一次次重构过程的记忆。

README 只承接已经被确认值得长期保留的结构性判断，而不承接任务批次、迁移过程、临时判断或一次性争议。

如何判断 README 应承接什么、不应承接什么，见 [`structure-refactoring-rules.md`](./structure-refactoring-rules.md) 第 4 节。

## 9. 规则沉淀也要分层

不是所有结构经验都应直接写进仓库级规则。

建议按三层落位：

- **仓库级通用原则**：适合写入 `docs/contributing/` 下的规则文件
- **领域级组织边界**：适合写入对应领域的 `README.md` / `overview.md`
- **一次性任务判断**：优先保留在 `temp/`、`backlog.md`、`conflict.md`

如何判断一条经验应沉淀到哪一层，见 [`structure-refactoring-rules.md`](./structure-refactoring-rules.md) 第 7 节。

## 10. 判断结构是否合理的推荐问题清单

当你怀疑某个目录结构是否合理时，优先先问：

1. 这个结构是否在帮助未来写作者做正确落位判断？
2. 这个层级当前采用的主分类轴是什么？是否一致？
3. 有没有把知识主题、对象类型、材料容器混在一起？
4. 目录名是在帮助理解边界，还是在制造歧义？
5. 如果未来内容增长数倍，这个结构还能稳定吗？
6. 现在不改，未来的迁移成本会不会更高？
7. 这个判断是仓库级原则、领域级边界，还是一次性任务经验？

具体的执行前检查清单，见 [`structure-refactoring-rules.md`](./structure-refactoring-rules.md) 第 2 节。

## 11. 与其他规则文件的关系

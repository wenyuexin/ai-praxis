# 规则的规则

本文件只回答一件事：**什么时候改规则，以及规则应该怎么改。**

本文件按需使用，不是 AI 协作者日常任务路由的一部分。只在人类维护者准备改规则、并需要 AI 协助时才读。

它不定义知识正文写法，也不替代 `evidence-rules.md`、`traceability-rules.md`、`documentation-workflow.md` 等专题规则。

## 1. 什么时候改规则

只有在少数情况才改规则：

- 同类错误被重复指出，说明现有规则没有拦住。
- 两篇或多篇规则文件之间职责不清，导致协作者反复误判。
- 新问题会长期重复出现，不能只靠这一次口头提醒解决。
- 现有规则已经明显过时、冲突，或会误导协作者。

以下情况通常**不改规则**：

- 只是一次性任务偏差。
- 只需修正文档内容，不需改治理规则。
- 通过补一个局部说明、删一段歧义表述就能解决。
- 只是想把规则写得更完整，但没有真实复发问题支撑。

## 2. 改规则前先判断

改规则前，先按这个顺序判断：

1. 这是**正文规则问题**，还是**规则如何维护的问题**？
2. 这是要改**现有专题规则**，还是要补一条**meta-rule**？
3. 问题是出在**缺规则**、**规则冲突**、还是**规则太长太难执行**？
4. 最小修复是加一句、删一句、挪位置，还是需要从长期结构出发重组规则？

如果只是规则维护方式的问题，优先写在 `meta-rules.md`，不要塞回正文规则文件。

## 3. 规则应该怎么改

默认先追求**长期最优**，不是局部最省事：

- **先判断长期结构是否合理**，不要只看这次修改是否方便。
- **如果小改足以解决且不破坏长期结构，可以小改。**
- **如果现有规则结构本身不合理，应允许重组、拆分、迁移或新建文件。**
- **先收紧职责边界**，再考虑补新内容。
- **先删歧义**，再加新条文。
- **先补短顺序提示或最小骨架**，不要先加长解释。

但有一个前提：**涉及明显重组、大改或多文件迁移时，必须先获得人类维护者的同意与协助，确认方案可行后再执行。**

只有当问题稳定属于“规则如何维护”这一层时，才写入本文件。

## 4. 写规则时的约束

- 规则文档优先服务 AI 快速执行，不追求面面俱到。
- 默认标准是：AI 在下次会话中不误执行、不走错路径；只保留少量必要解释，避免人类读者完全看不懂。
- 规则和“指导写规则的规则”要分开；后者只放本文件。
- `meta-rules.md` 按需使用；因为改规则本来就是低频动作，所以不要把大量低频维护原则铺进日常任务上下文。
- 示例只在很短且作用明显时才写；否则不写。
- 对能力较弱的模型，优先补顺序提示、字段骨架和 stop-line。
- 不要把一次局部修正膨胀成新的通用制度。

### 4.1 规则正文、制定理由、规则细则要分开

- **规则正文**：只写协作者需要直接执行的要求，优先使用“要 / 不要 / 先 / 仅在…时”这类可操作表达；保持短、稳、可扫读。
- **制定理由**：回答“为什么有这条规则、在防什么问题、权衡是什么”；默认不要塞进规则正文，优先放到 `intent/`、文件开头定位说明，或单独的解释层。
- **规则细则**：回答“遇到边界情况时具体怎么做”；只在同类误判反复出现时补，优先写进对应专题规则文件，而不是把正文规则越写越长。
- **Stop-line / 字段骨架 / 顺序提示** 属于执行支架，可以留在规则正文；但解释它们为什么存在的段落，不要和正文要求混写。

## 5. 放到哪里

- **专题内容规则**：放回对应规则文件，如 `evidence-rules.md`、`traceability-rules.md`、`documentation-workflow.md`。
- **规则职责边界**：优先改 `README.md` 或相关专题文件。
- **规则如何维护、何时改规则**：放在 `meta-rules.md`。

## 6. AI 忽略重要维护规则时怎么办

如果 AI 在执行中忽略了**重要的维护规则**，不要只做静默修正。

- **重复出现**：同类忽略被多次发现时，必须明确告诉人类维护者，并一起排查是规则缺失、职责边界不清、规则太长难执行，还是 AI 没有按路由读取规则。
- **即使次数不多，但已被人类提醒**：也不能只当作一次性偏差；至少要向人类说明这次为什么会忽略该规则，并协同判断是否需要补路由、补 stop-line、收紧职责边界，或补一条更短的维护规则。
- **排查目标**不是追责，而是确认问题属于哪一类：缺规则、规则冲突、规则过长、路由错误，还是执行时漏掉了已有要求。
- **越熟悉规则，越要反查遗漏**：如果协作者已经熟悉相关规则，却仍然没有遵守某条要求，不要先把它当作单纯执行失误；要优先怀疑这条规则本身仍有缺陷，例如表达不够直接、位置不够显眼、路由不够稳定，或与相邻规则职责边界不清。
- **默认动作**：先向人类暴露问题，再决定是修正文档内容、补专题规则，还是补 `meta-rules.md`。

不要因为问题这次还能手工补救，就跳过“为什么 AI 会忽略规则”的排查；否则同类错误很容易继续复发。

## 7. 最小工作流

当你准备改规则时，最稳妥的顺序是：

1. 先确认这是复发问题，不是一次性偏差。
2. 先确认该问题属于哪一层规则。
3. 先判断小改是否已经足够；如果会把长期结构继续做坏，不要为了局部省事而停在小改。
4. 若需要明显重组或多文件迁移，先请人类维护者确认方案并协助执行。
5. 只有现有文件不适合承接“规则如何维护”这一层问题时，才补到 `meta-rules.md`。

## 7. 自检

改完规则后，至少检查：

- 有没有把正文规则和 meta-rule 混在一起。
- 有没有为了少数情况新增过重规则。
- 有没有让 AI 需要多读一篇文件才能完成普通任务。
- 有没有新增长示例、长解释，反而分散注意力。

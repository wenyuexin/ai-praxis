# 规则的规则

本文件只回答一件事：**什么时候改规则，以及规则应该怎么改。**

本文件按需使用，不是 AI 协作者日常任务路由的一部分。只在人类维护者准备改规则、并需要 AI 协助时才读。

它不定义知识正文写法，也不替代 `evidence-rules.md`、`traceability-rules.md`、`documentation-workflow.md` 等专题规则。

## 1. 什么时候改规则

只有在少数情况才改规则：

- 同类错误被重复指出，说明现有规则没有拦住。
- 两篇或多篇规则文件之间职责不清，导致协作者反复误判。
- 新问题会长期重复出现，不能只靠这一次口头提醒解决。
- 现有规则已经明显过时、冲突，或会误导协作者。

以下情况通常**不改规则**：

- 只是一次性任务偏差。
- 只需修正文档内容，不需改治理规则。
- 通过补一个局部说明、删一段歧义表述就能解决。
- 只是想把规则写得更完整，但没有真实复发问题支撑。

## 2. 改规则前先判断

改规则前，先按这个顺序判断：

1. 这是**正文规则问题**，还是**规则如何维护的问题**？
2. 这是要改**现有专题规则**，还是要补一条**meta-rule**？
3. 问题是出在**缺规则**、**规则冲突**、还是**规则太长太难执行**？
4. 最小修复是加一句、删一句、挪位置，还是需要从长期结构出发重组规则？

如果只是规则维护方式的问题，优先写在 `meta-rules.md`，不要塞回正文规则文件。

## 3. 规则应该怎么改

默认先追求**长期最优**，不是局部最省事：

- **先判断长期结构是否合理**，不要只看这次修改是否方便。
- **如果小改足以解决且不破坏长期结构，可以小改。**
- **如果现有规则结构本身不合理，应允许重组、拆分、迁移或新建文件。**
- **先收紧职责边界**，再考虑补新内容。
- **先删歧义**，再加新条文。
- **先补短顺序提示或最小骨架**，不要先加长解释。

但有一个前提：**涉及明显重组、大改或多文件迁移时，必须先获得人类维护者的同意与协助，确认方案可行后再执行。**

只有当问题稳定属于“规则如何维护”这一层时，才写入本文件。

## 4. 写规则时的约束

- 规则文档优先服务 AI 快速执行，不追求面面俱到。
- 规则和“指导写规则的规则”要分开；后者只放本文件。
- 示例只在很短且作用明显时才写；否则不写。
- 对能力较弱的模型，优先补顺序提示、字段骨架和 stop-line。
- 不要把一次局部修正膨胀成新的通用制度。

## 5. 放到哪里

- **专题内容规则**：放回对应规则文件，如 `evidence-rules.md`、`traceability-rules.md`、`documentation-workflow.md`。
- **规则职责边界**：优先改 `README.md` 或相关专题文件。
- **规则如何维护、何时改规则**：放在 `meta-rules.md`。

## 6. 最小工作流

当你准备改规则时，最稳妥的顺序是：

1. 先确认这是复发问题，不是一次性偏差。
2. 先确认该问题属于哪一层规则。
3. 先判断小改是否已经足够；如果会把长期结构继续做坏，不要为了局部省事而停在小改。
4. 若需要明显重组或多文件迁移，先请人类维护者确认方案并协助执行。
5. 只有现有文件不适合承接“规则如何维护”这一层问题时，才补到 `meta-rules.md`。

## 7. 自检

改完规则后，至少检查：

- 有没有把正文规则和 meta-rule 混在一起。
- 有没有为了少数情况新增过重规则。
- 有没有让 AI 需要多读一篇文件才能完成普通任务。
- 有没有新增长示例、长解释，反而分散注意力。

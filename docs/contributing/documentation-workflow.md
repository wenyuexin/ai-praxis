# 材料处理与文档增改流程

本文回答：**当你已经准备实际修改仓库文档时，哪些判断顺序、最小门槛与 stop-line 不能跳过。**

- 它是 `docs/contributing/` 中的**规则化执行入口**，不是完整的材料吸收能力说明。
- 它引用 [`metadata-files.md`](./metadata-files.md) 的元信息文件模型，但不重新展开每个文件的完整定义。
- 它也不替代 [`docs/capabilities/ingest.md`](../capabilities/ingest.md) 对“为什么输入要先分流、降级、暂存与回流”的能力说明。
- 如果你在处理具体案例研究目录中的 `notes/` / 正文 / 机制专题分工，继续读 [`research-artifacts.md`](./research-artifacts.md)。

本文的核心原则是：

> **正文负责表达经过整理和判断后的知识；规则层只负责守住处理顺序、回流门槛与 stop-line。**

## 1. 最小执行门槛

真正动手新增、迁移、修改文档前，至少先完成下面四个判断：

1. **材料类型**：它是整体认知、主题缺口、待研究对象、冲突问题，还是正文专题材料？
2. **Evidence 状态**：它属于 `Verified`、`Observed`、`Inferred`、`Unverified`、`Conflicting` 还是 `Deprecated`？具体状态与标注方式见 [`evidence-rules.md`](./evidence-rules.md)，内容链路记录见 [`traceability-rules.md`](./traceability-rules.md)。
3. **目标落位**：它更适合进入 `overview.md`、`backlog.md`、`candidates.md`、`conflict.md`，还是某篇正文专题？
4. **临时落点**：如果今天不写正文，最合适的暂存位置是什么？

如果以上问题没有想清楚，默认不要直接新建正文专题。

## 2. 外部材料的最低处理要求

新材料常见来源包括：`temp/` 文档、外部 AI 调研报告、联网检索 / 网页抓取结果、博客摘录、issue / PR 讨论、论文阅读笔记、开源仓库观察记录。

这些材料默认都不是正文，也不默认等于主线结论。AI 协作者使用 `web_search` / `web_fetch` 获取资料时，也应视为正在调研；不能因为资料来自工具实时获取，就跳过分级、分流和回流判断。

如果材料来自本机上另行下载的外部公开仓库（例如从 GitHub 克隆到本地的开源项目），在仓库文档中的正式引用应优先使用上游公开仓库 URL、`owner/repo`，或基准 URL + 相对路径，而不是机器私有的本地文件系统路径。本地路径只可作为临时核验上下文，不应作为面向读者和协作者的标准来源标识。

如果引用对象本身就是当前仓库内的文件，则应使用仓库内相对路径作为正式引用，而不是绝对本地路径。

如果材料来自带分支、tag、release 和持续迭代历史的开源代码库，还必须先判断这次观察基于哪个版本截面：`branch`、`tag/release`、`commit`，以及观察日期。没有版本基线的源码观察，不应直接回流成主线定论。

## 3. 外部材料先做证据分级

从 `temp/`、联网检索 / 网页抓取、外部 AI 调研或临时笔记回流主线时，先判断 Evidence 状态。完整定义见 [`evidence-rules.md`](./evidence-rules.md)，本节只保留处理摘要：

- **Verified**：可追溯到论文原文、官方文档、GitHub 源码、release notes、维护者正式说明等一手材料，且关键结论能被直接支撑。
- **Observed**：在具体系统、项目、案例或多个二手来源中观察到，但尚不足以成为稳定通用结论。
- **Inferred**：基于已有材料、仓库上下文或相邻主题推导出的结论，但没有直接证据完全支撑。
- **Unverified**：作为工作假设、待验证观点或临时结构占位存在。
- **Conflicting**：不同来源之间存在术语、事实、版本、适用边界或结论力度冲突。
- **Deprecated**：曾经成立，但因版本变化、来源失效或新证据出现而不再适用。

处理方式：

- `Verified`：可作为进入 `overview.md` 或正文专题的候选材料。
- `Observed`：可进入正文，但必须保留观察对象、适用范围和限制；如仍不稳定，优先进入 `backlog.md`、`candidates.md` 或 `conflict.md`。
- `Inferred`：可作为主线草稿或结构化归纳进入正文，但必须显式标注推断依据和待补证方向。
- `Unverified`：只保留为线索，留在 `temp/`、阅读笔记、`backlog.md` 或 `candidates.md`；不要写成主线定论。
- `Conflicting`：优先进入 `conflict.md`，不要直接改写正文定论。
- `Deprecated`：从正文定论中移除或改为历史说明，并指向替代结论。

> **外部 AI 调研结果、联网检索结果和网页抓取结果默认只是上游输入，不是可直接发布的主线正文。**
>
> 不要因为它写得完整、结构漂亮、像综述，或来自实时联网工具，就跳过核验和分流步骤。

## 4. 新增、迁移与修改时的最低顺序

### 4.1 新增笔记

1. 先完成第 1 节的四个前置判断；不要把“还未整理的材料”直接写成正文。
2. 确定所属领域，找到对应的一级目录（顶层领域目录）。
3. 根据主题找到最匹配的二级或更深层子目录；如果该目录组有稳定顺序，优先使用已有编号目录。
4. 如果材料是对开源项目、产品或系统的系统性研究，优先集中放入该领域的项目对象目录；案例研究目录内部结构见 [`research-artifacts.md`](./research-artifacts.md)。
5. 如果判断依赖源码实现，先记录版本基线与观察日期；拿不准版本基线时，不要先写正文定论。
6. 按根目录 [`CONTRIBUTING.md`](../../CONTRIBUTING.md) 第二章规则命名文件。
7. 如无法归入现有子目录，可新建语义自解释的英文目录；只有该目录所在组存在明确阅读顺序、阶段顺序、流程顺序或稳定分类顺序时，才使用 `编号-英文短名` 格式，并同步更新该领域的 README。
8. 如果材料主要来自外部调研、联网检索 / 网页抓取或临时笔记，优先考虑先写入 `backlog.md`、`candidates.md` 或 `conflict.md`，而不是立即形成正文。

### 4.2 新增子目录

1. 确定在层级中的位置，并判断同组目录是否存在稳定阅读顺序、阶段顺序、流程顺序或分类顺序；只有存在稳定顺序时才接续编号。
2. 判断该目录主要承载正文专题，还是先承接元信息文件；目录定位先于内容填充。
3. 如果该目录存在或未来可能容纳子目录，创建目录并添加 `README.md`，内部用一句话说明该主题的定位和研究计划（无需展开）；纯文件目录（无子目录可能）不需要 README。
4. 更新对应一级目录（顶层领域目录）的 README 目录结构。
5. 如新增子目录使该领域的层级深度超出根 README 的展示限制，同步更新根 README。

### 4.3 新增一级目录（顶层领域目录）

1. 评估该目录属于哪个层级（基础方法 / 核心领域 / 应用与集成 / 基础设施 / 跨学科 / 支撑资源）。
2. 创建目录、`README.md`（一句话定位）。
3. 按层级顺序更新根目录 [`CONTRIBUTING.md`](../../CONTRIBUTING.md) 的层级设计表。
4. 更新根 README 的层级关系、知识体系（Mermaid 图）、目录结构三个章节。
5. 若该目录短期内主要处于调研积累阶段，优先先建立 `README.md` 与必要的元信息文件，而不是预先创建大量正文空壳。

### 4.4 修改已有正文

1. 先判断新增材料是在**强化现有结论**，还是在**引入新的冲突 / 限制条件**。
2. 如果只是强化已稳定结论，可直接补入正文，并视需要更新 `overview.md`。
3. 如果会改变术语、事实、版本状态、适用边界或结论力度，先更新 `conflict.md` 或相关元信息文件，再决定是否修改正文。
4. 如果只是发现“这里还值得继续写”，优先更新 `backlog.md`，不要用正文占位。
5. 修改正文后，如目录导航、边界说明或阅读路径受到影响，再同步更新 README。

## 5. 处理外部调研结果时的硬约束

拿到一份外部 AI 调研、联网检索 / 网页抓取结果、博客整理或阅读笔记后，至少守住下面几条：

1. 先标记其中哪些结论已经能追到一手来源，哪些只是间接观察。
2. 把“值得继续展开的主题缺口”写入 `backlog.md`。
3. 把“值得继续跟踪的对象”写入 `candidates.md`。
4. 如果对象是开源项目、产品或系统，且材料已经超出单一专题证据，应集中沉淀到该领域的项目对象目录。
5. 若来源是 GitHub 等公开社区仓库，即使你在本地完成了核验，正式文档中的来源引用仍应写上游仓库 URL，不写本地路径。
6. 把“定义 / 事实 / 结论冲突”写入 `conflict.md`。
7. 只有剩余的稳定部分，才考虑回流 `overview.md` 或正文专题；回流时只保留与该专题直接相关的证据和结论。
8. **不要把 `temp/` 路径本身写成主线正文或正式元信息文件中的最终 `Sources`。** `temp/` 只承接输入材料与临时整理；回流后，正式来源表达应优先改写为论文原文、官方文档、上游公开仓库 URL，或摘要性 Trace，而不是继续把 `temp/` 文件当成正式来源终点。

## 6. 外部临时材料回流的 stop-line

外部 AI 特别容易在以下场景中产生“看起来可信、实际上不稳”的内容：

- 把第三方博客中的实现细节写成官方事实。
- 把趋势判断写成行业共识。
- 把单篇论文的表述写成稳定结论。
- 把量化数字、性能比较或安全结论写得过满。

回流 stop-line：

- **不要把临时调研稿当成正文草稿直接润色。**
- **不要把 `temp/`、`web_search`、`web_fetch` 结果路径写成正式 `Sources`。**
- **不要用“某轮外部调研”替代本可恢复的真实来源。**
- **不要因为调研稿结构完整，就跳过 Claim 拆分、Evidence 分级和落位判断。**
- **不要让主干文档在删除临时文件后失去证据链。**

外部临时材料的完整吸收逻辑、回流工序与中间稳定层判断，继续读 [`../capabilities/ingest.md`](../capabilities/ingest.md)；对象目录中的 `notes/` / 正文 / 机制专题分工，继续读 [`research-artifacts.md`](./research-artifacts.md)。

## 7. 默认策略

- **拿不准写哪里**：先写元信息文件，不抢写正文。
- **拿不准是否可靠**：先降级处理，不抢写定论。
- **拿不准是 backlog 还是 candidates**：问题进 `backlog.md`，对象进 `candidates.md`；两者都有时允许互相引用。
- **拿不准是否该新建专题**：先补 `README.md`、`backlog.md` 或 `conflict.md`。
- **拿不准是否该迁移出 `temp/`**：先保留在 `temp/`，等落位、Evidence 状态和 Trace 字段明确后再迁移；回流条件见 [`evidence-and-traceability.md`](./evidence-and-traceability.md)。

## 8. AI 协作者执行前检查清单

AI 协作者在创建新文件、修改正文或迁移 `temp/` 内容前，必须先检查：

- 是否已经判断材料类型，而不是看到内容完整就直接写正文？
- 是否已经判断 Evidence 状态，区分 `Verified`、`Observed`、`Inferred`、`Unverified`、`Conflicting`、`Deprecated`？
- 如果材料来自外部 AI、联网检索 / 网页抓取或博客，是否已经把它当作上游输入，而不是主线结论？
- 是否存在更合适的元信息落点，例如 `backlog.md`、`candidates.md` 或 `conflict.md`？
- 是否把“问题缺口”和“研究对象”混在同一个文件里？如混合出现，是否做了互相引用？
- 如果正在研究具体论文、代码库或产品，是否同时保留了对象内聚性和对子领域共性的回流边界？案例研究组织方式见 [`research-artifacts.md`](./research-artifacts.md)。
- 如果某篇案例文档已经像源码流水或证据表，是否判断过应继续保留为 evidence notes，还是需要另写读者友好的机制总结？
- 是否正在为某个目录机械创建所有 L1-L7 元信息文件？若是，应停止；元信息文件按需出现。
- 是否因为拿不准而新建了正文专题？若拿不准，应先降级到元信息文件或保留在 `temp/`。

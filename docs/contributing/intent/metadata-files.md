# Metadata Files 的设计意图

本文件不重复 `../metadata-files.md` 的条文，而是解释：**为什么这个仓库要把元信息文件理解成一组按认知任务触发的设计位，而不是一组可以按文件名机械补齐的对称槽位。**

从结构上看，本文件可以按三组问题来读：

- 第 1–2 节：说明这份 intent 文件在回答什么，以及元信息文件层到底在解决什么问题
- 第 3–4 节：解释元信息文件为什么按问题触发，而不是按固定顺序或抽象分类机械补齐
- 第 5–6 节：说明主规则与 intent 的分工，以及这份文件应该在什么场景下使用

更细的专项设计意图请按问题继续进入：

- `README.md` / `index.md` / `roadmap.md` 的导航入口分工：读 [`navigation-files.md`](./navigation-files.md)
- `overview.md` / `landscape.md` 的原意与常见误读：读 [`overview-landscape.md`](./overview-landscape.md)

## 一、问题背景

## 1. 这个文件回答什么问题

`metadata-files.md` 负责回答：

- 哪些文件属于元信息文件
- 它们各自解决什么问题
- 在什么情况下出现

本文件回答的是更上游的问题：

- 为什么这些文件不能被当成一组对称槽位来理解
- 为什么元信息文件要按问题触发，而不是按固定顺序或抽象分类补齐
- 为什么主规则只保留最短、可执行边界，而把长解释留在 `intent/`

## 2. 元信息文件为什么存在

这个仓库不是把所有内容都写成知识正文，也不是把所有辅助信息都塞进 README。

这里先澄清一个容易被误读的点：`metadata-files.md` 里的“不要机械补齐”，主要是在约束**元信息文件层**，不是在给所有前瞻性结构设计下一个普遍禁令。它真正反对的是：在还没有对应认知任务支撑时，就把 `overview.md`、`landscape.md`、`backlog.md`、`candidates.md`、`roadmap.md`、`conflict.md` 一次性建好，或把大部分元信息文件预摆成一套对称槽位。

也就是说：

- 它反对的是**元信息文件的成套预铺**，不是一切长期结构前置判断
- 它强调的是**每个元信息文件都应有自己支撑的功能**，而不是因为模型里定义过这种文件就默认应该出现
- 它不否定按长期主义提前显式某个结构层；是否需要前置设计，应回到 `organization-principles.md` 讨论长期职责、边界稳定性与迁移成本

元信息文件存在，是因为仓库长期维护中会反复出现不同类型的问题：

- 入口定向问题
- 主线理解问题
- 结构切分问题
- 内容缺口问题
- 候选对象队列问题
- 阅读或建设顺序问题
- 口径冲突问题

这些问题如果混写在同一份文件里，读者和 AI 都会失去稳定预期：不知道打开某个文件后，看到的是“主题本身”、还是“目录怎么组织”、还是“接下来缺什么”。

所以元信息文件模型的核心，不是文件名管理，而是**认知任务分离**。

## 二、创建时机为什么按问题触发

## 3. 为什么“不要机械补齐”不能被读成固定顺序或普遍禁建

元信息文件并不是一组必须成套出现的槽位，它们的出现顺序也不应被读成固定顺序。不同目录在不同阶段，最先稳定下来的问题往往不同：

- 有时先出现的是目录粗定位，因此只有 `README.md`
- 有时先出现的是待补方向，因此先有 `backlog.md`
- 有时先出现的是候选对象队列，因此先有 `candidates.md`
- 有时先出现的是结构切分与边界归属，因此先有 `landscape.md`
- 只有当材料、结构与主线判断足够成熟时，`overview.md` 才更合理

所以，“不要机械补齐”的真正含义不是“现在先别建”，也不是“所有目录都应按同一顺序慢慢长文档”，而是：**不要在没有对应问题支撑时，把大部分元信息文件预先摆出来。**

换句话说，元信息文件的创建逻辑是：

- 按问题触发
- 按研究驱动分叉
- 按功能出现，而不是按文件名补全

## 4. 为什么创建时机不能只靠抽象分类决定

如果只按“主题理解 vs 结构说明”这种抽象二分来判断，AI 很容易把 `overview.md` 和 `landscape.md` 强行排成一条固定先后顺序，或者反过来在很多目录里机械创建 `landscape.md`，因为几乎任何复杂主题都可以说自己“也有结构”。

这个仓库不这样做。

更稳的判断方式不是先排文件顺序，而是先看**哪个问题先稳定出现**：

1. 只有一个初步想法时，通常先有 `README.md` 做粗定位。
2. 如果先稳定下来的是待补方向或问题簇，优先补 `backlog.md`。
3. 如果先稳定下来的是一批值得持续跟踪的研究对象，优先补 `candidates.md`。
4. 如果先稳定下来的是板块切分、边界归属或下钻路径，优先补 `landscape.md` 作为研究支架。
5. 只有当材料、结构与主线判断已经积累到足以支撑一篇相对独立的人类向主文时，才考虑 `overview.md`。
6. 当人类向主文与结构化研究职责同时成立，且已经不适合继续混写时，才同时保留 `overview.md` 与 `landscape.md`。

所以 `landscape.md` 不是默认 companion file，也不是目录成熟后自然会长出来的文件；但它也不必被理解成总是晚于 `overview.md` 的拆分产物。在很多真实研究流程里，`landscape.md` 会先作为结构支架出现，而 `overview.md` 更晚才成熟。

## 三、规则分层与使用方式

## 5. 主规则应该保留什么，`intent/` 再补什么

主规则文件里，应该保留足够短、足够稳的内容，例如：

- 元信息文件按认知任务分工
- 元信息文件按问题触发，不按层级补齐
- 不要把多个主问题重新混回同一文件
- 不同目录最先出现的元信息文件可以不同

这些信息应该足以让 AI 在多数场景下先执行对。

而本文件补的是：

- 为什么元信息文件的创建时机要按研究驱动分叉，而不是排成一条固定顺序
- 为什么元信息文件不能被理解成一组对称槽位
- 为什么主规则需要保持短、稳，而把长解释放进 `intent/`

更细的文件组专项原理，继续分流到：

- [`navigation-files.md`](./navigation-files.md)
- [`overview-landscape.md`](./overview-landscape.md)

## 6. 本文件的使用方式

本文件不是默认入口。

只有在以下情况才读：

- 需要修改 `metadata-files.md`，并确认元信息文件模型本身的设计依据
- 发现 AI 把元信息文件理解成一组对称槽位
- 需要解释为什么元信息文件应按问题触发，而不是按顺序或文件名补齐

平时执行任务，仍优先读 `metadata-files.md` 本身。

# Metadata Files 的设计意图

本文件不重复 `../metadata-files.md` 的条文，而是解释：**为什么这个仓库要把某些元信息文件分开，以及为什么 `overview.md` / `landscape.md` 特别容易被 AI 误读。**

## 1. 这个文件回答什么问题

`metadata-files.md` 负责回答：

- 哪些文件属于元信息文件
- 它们各自解决什么问题
- 在什么情况下出现

本文件回答的是更上游的问题：

- 为什么这些文件不能被当成一组对称槽位来理解
- 为什么 `overview.md` 与 `landscape.md` 不能被重新抽象成通用文件类型
- 为什么仓库需要同时保留“面向人类读者的主文”和“结构化文档体系构建文”两种不同职责

## 2. 元信息文件为什么存在

这个仓库不是把所有内容都写成知识正文，也不是把所有辅助信息都塞进 README。

元信息文件存在，是因为仓库长期维护中会反复出现不同类型的问题：

- 入口定向问题
- 主线理解问题
- 结构切分问题
- 内容缺口问题
- 候选对象队列问题
- 阅读或建设顺序问题
- 口径冲突问题

这些问题如果混写在同一份文件里，读者和 AI 都会失去稳定预期：不知道打开某个文件后，看到的是“主题本身”、还是“目录怎么组织”、还是“接下来缺什么”。

所以元信息文件模型的核心，不是文件名管理，而是**认知任务分离**。

## 3. 为什么 `README` / `index` / `overview` / `landscape` 必须分开

这三类文件最容易互相吞并，因为它们都看起来像“总览文件”。

但它们解决的问题并不一样：

- `README.md`：目录说明。回答“这是什么地方，为什么存在，和相邻目录如何区分”。
- `index.md`：目录结构与查找导航。回答“这个目录下面有什么，我已经知道想找什么时该去哪”。
- `overview.md`：人类向主文。回答“如果我只读一篇，这个主题最值得先理解的主线是什么”。
- `landscape.md`：结构化研究文。回答“这个主题内部怎么切分，文档体系如何协作，继续研究时往哪里下钻”。

如果这几者不分开，常见结果是：

- README 重新吞回目录结构展示与主要导航
- index 退化成不被触发的抽象辅助文件，或反过来变成第二篇 README
- overview 变成目录结构说明
- landscape 退化成一般性的 topic map

看起来都还“能用”，但长期会把仓库的写作和维护边界全部拖乱。

## 4. 为什么 `README.md` 与 `index.md` 也必须分开

在旧模型里，README 很容易同时承担四件事：

- 说明这个目录是什么
- 展示目录结构
- 提供粗粒度导航
- 偶尔还承担一点学习路径或阅读建议

这种写法短期看起来省事，因为读者只打开一个文件就能看到很多东西；但长期会稳定出现两个问题：

- `README.md` 会不断吞并结构展示与查找导航，逐渐变成“什么都写一点”的大入口文件
- `index.md` 即使存在，也会因为 README 已经足够好用而失去触发空间，最终退化成语义模糊的辅助文件

这次把目录结构与查找导航迁到 `index.md`，不是为了制造新的形式分工，而是为了解决一个真实复发问题：**当 README 同时承担说明、结构和导航时，AI 很难稳定形成 `index.md` 的文件类型心智。**

更具体地说：

- 如果 README 继续展示目录结构，AI 往往会自然停在 README，不再进入 index
- 如果 index 只负责抽象的“反向发现”，又缺少结构展示，它就很容易被理解成技术性索引页或可有可无的辅助文件

因此，新模型把两者重新拉开：

- `README.md` 只保留目录说明与最小入口分流
- `index.md` 承担目录结构与查找导航

这样做的目的不是让 index 成为第一跳，而是让它至少稳定成为**被 README 转交的第二跳导航页**，并逐步形成更清晰的文件类型心智。

## 5. 为什么 `index.md` 要从抽象索引升级成显式导航页

如果 `index.md` 只被定义为“反向发现入口”，它仍然过于依赖规则解释，而不容易靠文件本身的内容被理解。

AI 常见的自然判断是：

- `README.md` 是入口文件
- `overview.md` 是总览文件
- `index.md` 可能只是技术性索引页或目录列表

这意味着，仅靠“它叫 index，所以应该负责查找”是不够的。为了让 `index.md` 真正可触发，它必须同时具备两种信号：

- **结构信号**：让读者一打开就知道“这个目录下面有什么”
- **导航信号**：让读者继续知道“我已经知道想找什么时该去哪”

也因此，新模型不再把 `index.md` 理解成一个轻量附属文件，而是把它提升成显式导航页：既承接目录结构，也承接按问题、按阅读目标、按子目录的查找入口。

更准确地说，`index.md` 的最小稳定模型并不复杂，它至少要有两部分：

- **目录树或结构列表**：让读者一眼知道“这个目录下面有什么”
- **索引列表**：让读者继续知道“我在找某类东西时，应该跳去哪个子目录、文件或入口页”

这里“索引列表”的核心，不在于形式上一定要做成复杂表格，而在于它像一个轻量的检索目录：先把常见查找意图压缩成若干索引项，再把读者导向更具体的落点。像 `MEMORY.md` 这样的文件之所以有启发性，不是因为它记录了哪些内容，而是因为它把“先看索引，再决定读哪一篇本体”的机制做得非常明确。

所以，按问题找入口、按阅读目标找入口、按对象类型找入口，这些都不是另一套职责，而只是“索引列表”在复杂目录中的展开版本。小目录只要有目录结构和足够轻的索引列表，`index.md` 就已经成立；复杂目录才需要进一步展开成表格、交叉映射或多组入口。

这不是让 `index.md` 变成第二篇 README，而是把“结构展示 + 查找导航”这一整组职责从 README 中剥离出来，集中给一个真正面向导航的文件承接。

## 6. 复杂目录为什么更需要把 `README.md` / `index.md` / `roadmap.md` 拆开

越是复杂目录，越容易同时长出三类内容：

- 目录为什么存在、按什么轴组织
- 这个目录下面有什么、已经知道想找什么时该去哪
- 不同读者应该从哪开始、按什么路径推进

如果这三类内容继续混写，最常见的结果就是：

- `README.md` 重新变成大而全入口
- `index.md` 退化成语义模糊的辅助文件
- `roadmap.md` 与 `index.md` 再次混成同一类“泛导航”文档

所以真正稳的切法不是“把内容平均分给三个文件”，而是按认知任务拆开：

- `README.md` 保留目录说明
- `index.md` 承接结构展示与查找导航
- `roadmap.md` 承接起点、顺序与路径分流

这也是为什么主规则里需要把 `index.md` 与 `roadmap.md` 的边界写得更硬：前者解决“去哪”，后者解决“从哪开始、怎么走”。

一个对应的反向判断是：复杂目录里即使已经出现一些主线性很强的问题簇，也不等于应该立刻补 `overview.md`；只有当顶层已经形成一篇相对独立、可单篇成立的人类向主文时，`overview.md` 才应该出现。

如果需要看这个判断在真实复杂目录中是如何落地的，可参考 [`cases/interdisciplinarity-readme-index-roadmap.md`](../cases/interdisciplinarity-readme-index-roadmap.md)。

## 7. `overview.md` 的原意

`overview.md` 不是“一个主题的主入口文件”，也不是“和 landscape 对称的一种理解层文件”。

它在这个仓库里的原意更强：**它是一篇相对独立、给人类读者看的全景式总览文档。**

这意味着：

- 即使脱离同目录下的其他文档，它仍然应该具备可读性。
- 读者不继续阅读 sibling docs，也应该能建立相对完整的整体理解。
- 它的优先目标是帮助人类恢复或建立主题主线，而不是辅助构件其他文档。

因此 `overview.md` 不应承担：

- 目录级分工说明
- 子专题协作关系说明
- 结构切分的治理理由
- 面向 AI / 维护者的研究组织任务

一旦一篇文档开始大量承担这些事，它就已经不再只是 overview。

## 8. `landscape.md` 的原意

`landscape.md` 也不是普通意义上的“研究地图”或“主题内部板块图”。

它在这个仓库里的职责更具体：**承接结构化文档体系的构建。**

这里的“结构化”不是泛指“有结构”，而是指这套仓库规则写出来的文档体系本身，包括：

- 板块如何切分
- 相邻主题边界如何稳定下来
- 对象研究、专题文档、辅助材料如何协作
- 文档构建与领域覆盖如何平衡
- 哪些地方需要更强的 evidence 写法
- 为什么这里这样切，而不是按另一条看似合理的轴切

所以 `landscape.md` 承接的，不只是主题内容本身，还包括**这套文档系统如何长期成立**。

这就是为什么把它简化成“topic map”会稳定出错：它丢掉了仓库特有的文档构建职责。

## 9. 为什么 AI 特别容易误读这两个文件

AI 最容易犯的错，不是完全读不到条文，而是：

- 先把 `overview` 理解成“主线解释文”
- 再把 `landscape` 理解成“结构地图文”
- 最后把两者重写成一组抽象、对称、通用的文件类型

这一步看似合理，实际上已经丢失了维护者原意。

因为在这个仓库里：

- `overview` 不是一般性的主线文，而是**相对独立的人类向总览主文**
- `landscape` 不是一般性的结构地图，而是**结构化文档体系构建文**

一旦这个区别被抹平，AI 后续判断文件职责、创建时机、文档内容边界时都会开始漂移。

## 10. 创建时机为什么不能只靠抽象分类决定

如果只按“主题理解 vs 结构说明”这种抽象二分来判断，AI 很容易在很多目录里机械创建 `landscape.md`，因为几乎任何复杂主题都可以说自己“也有结构”。

这个仓库不这样做。

更稳的判断顺序是：

1. 先问：这个目录是否已经需要一篇相对独立、可单篇成立的人类向总览主文？如果需要，考虑 `overview.md`。
2. 再问：这篇主文是否已经开始承担结构化文档体系构建职责？如果是，才考虑拆出 `landscape.md`。
3. 只有当“人类向主文”与“结构化研究文”确实分离时，才同时保留两者。

所以 `landscape.md` 不是默认 companion file，也不是目录成熟后自然会长出来的文件。它只在**文档体系构建问题已经足够重**时才出现。

## 11. 一个典型误读信号

当 AI 开始说出下面这类话时，通常说明它已经偏离了这组文件的仓库内原意：

- “`overview` 和 `landscape` 是两种并列的抽象文件类型……”
- “`overview` 负责主线，`landscape` 负责结构，所以谁先都可以……”
- “`landscape` 就是 topic map / research map ……”

这些说法的问题不在于它们完全错误，而在于它们会系统性丢失：

- `overview` 的独立可读性要求
- `landscape` 的结构化文档体系构建职责

## 12. 主规则应该保留什么，`intent/` 再补什么

主规则文件里，应该保留足够短、足够稳的内容，例如：

- `overview.md` 是人类向主文
- `landscape.md` 是结构化研究文
- overview 不能退化成目录结构说明
- landscape 不应作为默认创建文件

这些信息应该足以让 AI 在多数场景下先执行对。

而 `intent/metadata-files.md` 补的是：

- 为什么 `overview` 的“人类向主文”不是泛泛而谈，而是独立可读性要求
- 为什么 `landscape` 不只是 topic map，而是文档体系构建文
- 为什么这两个文件一旦被抽象成一组通用类型，后续文档就会稳定写偏

## 13. 本文件的使用方式

本文件不是默认入口。

只有在以下情况才读：

- 需要解释 `overview.md` / `landscape.md` 的原意
- 发现 AI 在这两个文件上反复误读
- 需要修改 `metadata-files.md`，并确认主规则条文背后的设计依据

平时执行任务，仍优先读 `metadata-files.md` 本身。

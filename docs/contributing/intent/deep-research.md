# 深度研究脚手架的设计意图

本文件不重复 `../research-artifacts.md` 的条文，而是解释：**为什么当总览已经够用、但机制理解还不够时，不能继续只靠补 overview 或依赖用户反复追问，而应切到主线驱动的深度研究脚手架。**

## 1. 这个文件回答什么问题

`research-artifacts.md` 负责回答研究产物如何组织，包括正文、机制专题和辅助材料怎么分层。

本文件回答的是更细的一层：

- 为什么有些研究不能继续靠 overview 深挖
- 为什么深度研究需要“主线问题 + 问题树 + 已知/待确认/下一步”这种脚手架
- 为什么不应该要求用户每轮都手动提示下一问
- 为什么脚手架不应直接原样升格为正式正文

## 2. 为什么 overview 足够做入门，却不够做深度研究

overview 的职责是让读者快速建立对象定位：

- 它是什么
- 为什么值得看
- 有哪些核心能力
- 系统边界大致在哪里

但当研究进入机制级理解时，overview 会稳定失效。

因为这时真正的问题不再是“它有什么”，而是：

- 关键机制到底怎么工作
- 这个机制由哪些上游子问题组成
- 哪些部分已经压实，哪些还只是线索
- 下一轮研究应该优先补哪里

如果继续只补 overview，常见结果是：

- 文档越来越长，但层级越来越混
- 已回答与未回答问题缠在一起
- 读者看到了很多信息，却仍然不知道主线机制如何闭环

## 3. 为什么不能只靠用户一轮一轮追问

当研究已经进入主线递进阶段，如果还要求用户不断手动发问，说明研究方法本身没有搭起脚手架。

这会产生两个稳定问题：

- 研究推进依赖用户临场记忆，而不是当前知识边界自然长出下一问
- AI 每轮都在局部响应，容易丢主线、重复探索、或者跳到低价值分支

所以深度研究不是“回答更多问题”，而是要把问题组织成一条能持续推进的主线。

## 4. 为什么要用“主线问题 + 问题树”

深度研究最需要的不是更多结论，而是一个能带着结论继续往下走的结构。

“主线问题 + 问题树”有三个作用：

- 把研究对象从“很多零散问题”收束成一条主线
- 让下一层问题从当前知识边界自然长出来
- 让已回答、未回答、待补证的状态显式可见

这也是为什么脚手架至少要保留：

- 主线问题
- 分层子问题
- 当前已知
- 待确认
- 下一步研究动作

没有这些层，研究就会重新退化成“想到哪问到哪”。

## 5. 为什么脚手架应放在研究辅助材料层

深度研究脚手架不是给普通读者看的完成态正文，而是给研究者和 AI 持续推进主线用的。

它天然包含：

- 未闭环问题
- 证据缺口
- 优先级判断
- 下一轮动作

这些内容如果直接写进正式正文，会让正文读起来像研究过程记录，而不是稳定知识。

所以脚手架最合适的落位是 `notes/`：

- 它保留推进结构
- 它允许暂时不完整
- 它服务于后续回流，而不是替代回流结果

## 6. 为什么不能把脚手架原样升格为正文

当某条主线已经压实，正确做法是：

- 从脚手架里提炼一篇正式机制文
- 让脚手架继续保留未闭环问题和后续推进路线

而不是：

- 直接把脚手架改成正文

因为脚手架和正文回答的是不同问题：

- 脚手架回答“接下来怎么继续研究”
- 正文回答“目前已经可以稳定告诉读者什么”

两者混在一起，会让正式知识带上过强的过程态痕迹。

## 7. 一个典型误读信号

当 AI 开始说出下面这类话时，通常说明它已经偏离了这条规则的原意：

- “overview 再补几段就够深了……”
- “用户继续追问时再展开就行……”
- “把 question tree 直接整理一下就是正文……”

这些说法的问题分别是：

- 把入门总览误当机制研究载体
- 把研究推进外包给用户手动提示
- 把研究驾驶舱误当完成态知识文档

## 8. 什么样的对象值得启用深度研究脚手架

这类判断不适合只靠人脑临时记住，因为它往往发生在“overview 已经写过，但还没确定要不要继续深挖”的边界时刻。

可以优先启用深度研究脚手架的对象，通常同时满足下面几条：

- 已经有 `overview` 或同层总览，足以支撑初步理解
- 但继续阅读后，仍然说不清某条关键机制到底怎么工作
- 问题能够自然沿一条主线递进，而不是只会横向扩展功能点
- 这条主线能继续拆成上游子问题，例如“如何定义 → 如何构建 → 为什么这样设计 → 如何参与调用或检索”
- 该对象值得继续深入，而不是一次性补完就结束的小点

暂时不必启用这套方法的对象，通常有这些特征：

- 还停留在“它是什么、有什么功能”都没站稳的阶段
- 没有清晰主线，只是一串并列问题或功能清单
- 研究价值不足以支撑持续递进，补几段正文就能收口
- 真正缺的不是机制研究，而是证据补齐、落位判断或命名收紧

这份清单不是硬性打分表，而是帮助维护者快速判断：当前问题更像“继续补正文”，还是“切换到问题树驱动的深度研究”。

## 9. 主规则应该保留什么，`intent/` 再补什么

主规则文件里，应该保留足够短、足够稳的内容，例如：

- 什么时候从总览切到深度研究脚手架
- 脚手架最小结构是什么
- 什么时候把脚手架回流成正式正文
- 不要把脚手架原样升格

这些信息应该足以让 AI 在多数场景下先执行对。

而 `intent/deep-research.md` 补的是：

- 为什么 overview 会在机制级研究中失效
- 为什么深度研究不能只靠用户反复追问
- 为什么脚手架应保留在辅助材料层
- 为什么回流应是“提炼正文”，而不是“脚手架改名”

## 9. 本文件的使用方式

本文件不是默认入口。

只有在以下情况才读：

- 需要修改 `research-artifacts.md` 中关于深度研究脚手架的条文
- 发现 AI 一直在补 overview，却没有切到机制级研究方法
- 需要解释为什么某个对象研究必须先建问题树，再回流正文
- 需要复盘“为什么用户不应每轮都手动提示下一问”

平时执行研究组织任务，仍优先读 `research-artifacts.md` 本身。

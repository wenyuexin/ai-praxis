# Traceability 规则的设计意图

本文件不重复 `../traceability-rules.md` 的条文，而是解释：**为什么这个仓库的 Traceability 既不能做成重型 lineage 系统，也不能退化成一句模糊备注。**

## 1. 这个文件回答什么问题

`traceability-rules.md` 负责回答“Trace 要记什么、最少记到什么程度、记在哪里”。

本文件回答的是更上游的问题：

- 为什么仓库需要 Traceability，而不只需要 Evidence
- 为什么当前采用的是轻量 Traceability，而不是全量谱系系统
- 为什么 `Source / Decision / Placement / Gap` 这四个字段不能被随意压扁成一句“来自某处”
- 为什么代码库研究里的版本链路字段不是可有可无的装饰

## 2. 为什么有了 Evidence 还不够

Evidence 解决的是：**这条说法凭什么成立。**

但仓库治理里还有另一类问题，Evidence 本身回答不了：

- 这段内容是从哪里来的
- 为什么它进入正文，而不是继续留在 `temp/`
- 为什么它落在 `overview.md`，而不是 `backlog.md` 或 `conflict.md`
- 当前到底是“结论已稳定”，还是“只是先被放到了一个临时合理的位置”

如果没有 Traceability，AI 很容易做出一种表面上“有证据”的错误：

- 给出一个 `Status`
- 列出几个 `Sources`
- 但完全不解释内容是怎么进入当前文件的
- 也不解释为什么现在应该放在这里

这会导致后续维护者只能看到“这句话可能是真的”，却看不到“这句话为什么会出现在这里”。

## 3. 为什么当前必须是轻量 Traceability

这个仓库不是在建设一套企业级内容资产管理系统。

如果把 Traceability 直接设计成重型 lineage 系统，常见结果是：

- AI 为了补字段开始机械造表
- 维护成本迅速高过实际收益
- 旧文档几乎无法补齐
- 协作者把精力花在追踪格式，而不是知识判断

所以当前仓库故意只保留**对知识治理真正有用的最小链路**：

- 输入从哪里来
- 为什么这样处理
- 最终落到哪里
- 还缺什么

也就是说，当前 Traceability 不是为了记录“所有编辑历史”，而是为了记录**为什么这份内容现在以这种方式存在于仓库中**。

## 4. 为什么又不能把它缩成一句模糊备注

另一种常见误读，是把轻量 Traceability 理解成“随便写一句 Trace 就行”。

这同样会稳定出错。

因为如果只写成：

- “来自调研”
- “综合整理”
- “从临时材料回流”

那实际上丢掉了最关键的治理信息：

- 到底是哪个输入材料
- 为什么这次决策是进正文、进 backlog，还是继续留 temp
- 当前落位是主题主文、对象研究，还是冲突记录
- 还缺的是哪一类证据，而不是泛泛一句“待补证”

所以轻量不等于含糊，最小不等于随便。

## 5. `Source / Decision / Placement / Gap` 为什么是最稳的骨架

这四个字段不是为了形式完整，而是因为它们刚好覆盖了一个治理动作最容易丢失的四类信息：

- `Source`：输入边界
- `Decision`：处理理由
- `Placement`：落位结果
- `Gap`：未完成部分

如果缺其中任何一个，后果都很具体：

- 缺 `Source`：读者无法回到输入材料
- 缺 `Decision`：读者不知道为什么这么处理
- 缺 `Placement`：读者不知道当前记录对应哪个真实落位
- 缺 `Gap`：读者会误以为这条内容已经闭环

这也是为什么主规则里允许正文用 `Sources / Trace / Needs` 这种更人类可读的写法，但底层仍然要能映射回这四类信息。

## 6. 为什么 Trace Unit 不是每句话

如果把 Trace Unit 定到“每一句话”，AI 会很容易把 Traceability 误做成逐句注释系统。

这不符合当前仓库的治理目标。

仓库真正关心的不是“这一句是几点几分改的”，而是：

- 这篇专题为什么被新建
- 这个 overview 判断为什么现在可以进入主线
- 这个 backlog 条目为什么只是缺口，而不是正文结论
- 这次 `temp/` 回流为什么落在这里

所以当前最小单位是**一次有治理意义的内容落位**，不是句子级 provenance。

## 7. 为什么代码库研究一定要补版本链路

在一般主题文档里，很多来源只要能稳定还原即可。

但代码库研究不同：源码实现天然带时间性。

如果不补 `Version Basis` 和 `Observed At`，AI 很容易把一次静态观察写成长期事实。例如：

- 只看了默认分支的当前代码，却写成“该项目支持 X”
- 混用了 release notes、issue 和当前源码，却没有说明它们是否属于同一版本窗口
- 把讨论中的未来能力写成当前能力

所以版本链路字段不是为了“文档更正式”，而是为了阻止**源码截面观察伪装成长期定论**。

这也是为什么 `Version Basis` 和 `Observed At` 是最低字段，而不是可选美化项。

## 8. 为什么正式 Trace 优先写共享来源，而不是本机路径

本机绝对路径对当前协作者也许方便，但对仓库读者和未来协作者不可共享。

Traceability 的目标之一是让后来者能够还原输入材料，而不是证明“我本地看过”。

所以正式 Trace 要优先写：

- 仓库内相对路径
- 上游 GitHub URL
- 官方文档 URL
- 可以被他人重复访问的公开来源

这条设计意图很重要，因为 AI 很容易把“本地验证动作”误写成“正式来源表达”。

## 9. 为什么 `temp/` 回流特别需要 Trace

`temp/`、外部 AI 调研、网页抓取结果最容易引发一种错觉：

- 内容看起来已经很像正文
- 于是直接被抄进主线
- 但没人知道这是稳定结论，还是只是整理过的输入

所以这类材料回流时，Trace 的价值特别高。

它不是在补格式，而是在告诉后续维护者：

- 这条内容原本是输入材料
- 为什么这次选择回流
- 回流后放进了哪个治理槽位
- 还缺什么验证，不能把它误读成彻底定稿

## 10. 一个典型误读信号

当 AI 开始说出下面这类话时，通常说明它已经偏离了 Traceability 的原意：

- “Trace 就是简单记一下来源……”
- “既然有 Git，就不用写 Trace 了……”
- “Traceability 应该给每条 Claim 建完整 lineage……”
- “写一句 from temp 就够了……”

这些说法的问题分别是：

- 把 Trace 缩得太轻，失去治理价值
- 把 Git 历史和治理链路混为一谈
- 把当前仓库误推向重型系统
- 把最小骨架字段压缩成无法复原的模糊备注

## 11. 主规则应该保留什么，`intent/` 再补什么

主规则文件里，应该保留足够短、足够稳的内容，例如：

- Traceability 回答“这条内容从哪里来、为什么落在这里”
- 当前采用轻量 Traceability
- 最小骨架是 `Source / Decision / Placement / Gap`
- 基于源码观察的结论不要省略 `Version Basis` 和 `Observed At`

这些信息应该足以让 AI 在多数场景下先执行对。

而 `intent/traceability-rules.md` 补的是：

- 为什么轻量不等于模糊备注
- 为什么不能把 Trace 做成重型 lineage 系统
- 为什么代码库研究里的版本链路字段是防错机制，不是装饰字段
- 为什么正式来源表达要优先可共享、可复现的引用，而不是本机路径

## 12. 本文件的使用方式

本文件不是默认入口。

只有在以下情况才读：

- 需要修改 `traceability-rules.md`
- 发现 AI 对 Traceability 的理解在“过重”和“过轻”之间反复摆动
- 需要解释为什么版本链路字段必须出现
- 需要解释为什么 Git 不能替代当前仓库的治理链路记录

平时执行文档任务，仍优先读 `traceability-rules.md` 和 `evidence-and-traceability.md`。

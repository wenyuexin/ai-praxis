# 导航入口文件的设计意图

本文件不重复 `../metadata-files.md`、`../readme-rules.md` 或 `../index-rules.md` 的条文，而是解释：**为什么这个仓库要把 `README.md`、`index.md`、`roadmap.md` 这组三个入口/导航文件拆开，以及为什么 `index.md` 不能继续退回 README 的附属物。**

## 1. 这个文件回答什么问题

主规则文件分别回答：

- `metadata-files.md`：这些元信息文件各自解决什么问题、在什么情况下出现
- `readme-rules.md`：README 本身怎么写、出现条件和导航边界
- `index-rules.md`：`index.md` 的触发条件、结构模型与导航细则

本文件回答的是更上游的问题：

- 为什么 `README.md`、`index.md`、`roadmap.md` 不能重新混回一个“大入口文件”
- 为什么 `index.md` 应比旧模型更早触发
- 为什么 `index.md` 不是抽象附属文件，而应成为显式导航页
- 为什么复杂目录里这三者更需要拆开，而不是继续靠 README 吞并

## 2. 为什么目录说明、结构导航、路径指引必须拆开

这三类内容看起来都像“入口信息”，但它们解决的问题并不一样：

- `README.md`：回答“这是什么地方、为什么存在、和相邻目录如何区分”
- `index.md`：回答“这个目录下面有什么、我已经知道想找什么时该去哪”
- `roadmap.md`：回答“从哪开始、按什么顺序推进”

如果继续混写，最常见的结果通常是：

- `README.md` 不断吞回目录结构展示与查找导航
- `index.md` 即使存在，也因为 README 已经“够用”而失去触发空间
- `roadmap.md` 与 `index.md` 再次混成同一类“泛导航”文档

看起来都还“能用”，但长期会稳定拖乱入口层分工。

## 3. 为什么 `README.md` 与 `index.md` 也必须分开

在旧模型里，README 很容易同时承担四件事：

- 说明目录是什么
- 展示目录结构
- 提供粗粒度导航
- 偶尔再承担一点学习路径或阅读建议

这种写法短期省事，因为读者只打开一个文件就能看到很多东西；但长期会稳定出现两个问题：

- `README.md` 变成“什么都写一点”的大入口文件
- `index.md` 即使存在，也会因为 README 已经足够好用而退化成语义模糊的辅助文件

这次把目录结构与查找导航迁到 `index.md`，不是为了制造新的形式分工，而是在修复一个真实复发问题：**当 README 同时承担说明、结构和导航时，AI 很难稳定形成 `index.md` 的文件类型心智。**

更具体地说：

- 如果 README 继续展示目录结构，AI 往往会自然停在 README，不再进入 index
- 如果 index 只负责抽象“反向发现”，又缺少结构展示，它就很容易被理解成技术性索引页或可有可无的附属文件

因此，新模型把两者重新拉开：

- `README.md` 只保留目录说明与最小入口分流
- `index.md` 承担目录结构与查找导航

这样做的目的不是让 index 成为第一跳，而是让它至少稳定成为**被 README 转交的第二跳导航页**。

## 4. 为什么 `index.md` 要更早触发

不要等到目录已经“足够复杂”才补 `index.md`。

只要稳定子结构已经出现，就应该允许先创建一个很薄的骨架型 `index.md`，把目录结构从 README 中尽早剥离出去。

如果继续把“内容还不够多”当作一律推迟 `index.md` 的理由，常见后果会是：

- 目录树继续留在 README 里
- README 越写越重
- index 的职责只能靠后补救，而不是从一开始就建立稳定心智

所以，`index.md` 应更早触发，不是因为模型里多了一个槽位，而是因为结构展示一旦长期留在 README，就会稳定吞回查找导航职责。

## 5. 为什么 `index.md` 要从抽象索引升级成显式导航页

如果 `index.md` 只是一个模糊的“索引页”概念，它很容易再次失去触发空间。

它至少要同时给出两种信号：

- **结构信号**：让读者一打开就知道“这个目录下面有什么”
- **导航信号**：让读者继续知道“我已经知道想找什么时该去哪”

所以当前仓库不再把 `index.md` 理解成轻量附属文件，而是把它提升成显式导航页：既承接目录结构，也承接按问题、按阅读目标、按子目录的查找入口。

更准确地说，它的最小稳定模型应拆成两档：

- **骨架型 `index.md`**：目录结构或结构列表 + 直接子目录的一句话定位
- **展开型 `index.md`**：在骨架型基础上，再补索引列表，让读者继续知道“我在找某类东西时，应该跳去哪个子目录、文件或入口页”

这不是让 `index.md` 变成第二篇 README，而是把“结构展示 + 查找导航”从 README 中剥离出来，集中给一个真正面向导航的文件承接。

## 6. 为什么复杂目录更需要把 `README.md` / `index.md` / `roadmap.md` 拆开

越是复杂目录，越容易同时长出三类内容：

- 目录为什么存在、按什么轴组织
- 这个目录下面有什么、已经知道想找什么时该去哪
- 不同读者应该从哪开始、按什么路径推进

如果继续混写，最常见的结果就是：

- `README.md` 重新变成大而全入口
- `index.md` 退化成语义模糊的辅助文件
- `roadmap.md` 与 `index.md` 再次混成同一类“泛导航”文档

真正稳的切法不是“把内容平均分给三个文件”，而是按认知任务拆开：

- `README.md` 保留目录说明
- `index.md` 承接结构展示与查找导航
- `roadmap.md` 承接起点、顺序与路径分流

如果需要看这个判断在真实复杂目录中如何落地，可参考 [`../cases/interdisciplinarity-readme-index-roadmap.md`](../cases/interdisciplinarity-readme-index-roadmap.md)。

## 7. 一个典型误读信号

当 AI 开始说出下面这类话时，通常说明它已经偏离了这组文件的仓库内原意：

- “README 里顺便放目录树也没关系……”
- “index 只是个附属索引页，可有可无……”
- “roadmap 也算导航，所以和 index 差不多……”

这些说法的问题分别是：

- 把目录说明重新扩写成结构导航
- 让 `index.md` 再次失去稳定触发空间
- 把“去哪”和“怎么走”混成同一类问题

## 8. 主规则应该保留什么，`intent/` 再补什么

主规则文件里，应该保留足够短、足够稳的内容，例如：

- `README.md` 负责目录说明
- `index.md` 负责结构展示与查找导航
- `roadmap.md` 负责起点、顺序与路径分流
- `index.md` 应比旧模型更早触发
- 不要让 README 长期代行目录树

这些信息应该足以让 AI 在多数场景下先执行对。

而本文件补的是：

- 为什么入口文件不能重新混成一个大入口
- 为什么 `index.md` 要更早触发
- 为什么 `index.md` 不是附属索引，而是显式导航页
- 为什么复杂目录里这三者更需要拆开

## 9. 本文件的使用方式

本文件不是默认入口。

只有在以下情况才读：

- 需要修改 `metadata-files.md`、`readme-rules.md`、`index-rules.md` 中与导航入口分工相关的条文
- 发现 AI 反复把 README、index、roadmap 重新混写
- 需要解释为什么 `index.md` 应更早触发
- 需要复盘“为什么不能继续让 README 代行目录树”

平时执行文档治理任务，仍优先读 `metadata-files.md`、`readme-rules.md`、`index-rules.md` 本身。

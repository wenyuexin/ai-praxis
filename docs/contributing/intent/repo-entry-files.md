# 仓库级入口文件的分工意图

本文件解释仓库级入口文件的分工，重点回答：为什么同时存在 `AGENTS.md` 与 `CONTRIBUTING.md`、两者分别承接什么，以及在修改这类文件时应如何判断改动应落在哪一层。它不是新的公共入口，也不新增执行条文；主规则负责让协作者默认执行对，本文件只负责保存这些入口分工背后的设计意图与边界。

**Stop-line：本文件只解释仓库级入口文件的分工与落位，不新增执行规则，也不扩展为所有顶层文件说明。**

## 1. 这个文件回答什么问题

当协作者准备修改 `AGENTS.md`、`CONTRIBUTING.md`，或讨论某条提示究竟该放在哪个顶层入口文件时，问题通常已经不只是“这句怎么写”，而是：

- 为什么仓库需要两个入口文件，而不是一个总说明文件
- 哪些内容更适合进入 `AGENTS.md`
- 哪些内容更适合进入 `CONTRIBUTING.md`
- 哪些低频说明不应留在任何公共入口，而应转去 `docs/contributing/` 或 `intent/`

本文件只解释这层分工原意，不替代具体规则：

- 改规则前的工作流，见 [`../meta-rules.md`](../meta-rules.md)
- 规则为什么需要 `intent/` 解释层，见 [`meta-rules.md`](./meta-rules.md)

## 2. 为什么 `AGENTS.md` 与 `CONTRIBUTING.md` 不合并

`AGENTS.md` 与 `CONTRIBUTING.md` 都是仓库级入口文件，但它们解决的不是同一个问题。

- **`AGENTS.md`** 负责 AI 协作者在进入仓库时必须先遵守的最小强约束：先读哪些文件、按什么任务路由去读下层规则、哪些 stop-line 不能越过。
- **`CONTRIBUTING.md`** 负责仓库级贡献、组织与写作规范：目录层级、命名规则、README 规范、元信息文件模型、材料处理默认策略等。

因此，两者不是“一个给 AI、一个给人类”的简单二分，也不是内容互相复制的双份入口，而是**执行前拦截层**与**仓库级规范层**的分工。

如果把两者合并，常见结果通常是：

- 顶层入口同时承担执行拦截、仓库规范、低频治理说明，导致入口过重
- 低频说明与高频约束混在一起，协作者难以区分什么必须立刻执行、什么只是后续查阅
- 每次补一个低频提示，都容易进一步膨胀公共入口上下文

保留两层入口的目的，不是让结构更复杂，而是让协作者在进入仓库时先被送到正确层级，再按需要下钻。

## 3. 两个文件各自该承接什么

### 3.1 `AGENTS.md` 该承接什么

`AGENTS.md` 适合承接以下内容：

- 进入仓库后必须先做的动作
- 日常任务最容易错过的强约束
- 面向 AI 的任务路由：做哪类任务时先读哪份下层规则
- 极短、必须优先执行的 stop-line

`AGENTS.md` 不适合承接：

- 长篇仓库组织说明
- 低频规则维护理由
- 需要背景解释才能理解的制度设计
- 只在修改规则时才会触发的长说明

### 3.2 `CONTRIBUTING.md` 该承接什么

`CONTRIBUTING.md` 适合承接以下内容：

- 仓库级组织与写作规范
- 目录层级、命名、README、元信息文件等长期稳定规则
- 带着具体任务进入仓库时的入口导读
- 适合作为默认策略保留的短提示

`CONTRIBUTING.md` 不适合承接：

- 只在少数场景触发的长维护说明
- 某条规则为什么这样制定的完整解释
- 规则系统如何维护的详细流程
- 本可下沉到 `docs/contributing/` 的低频专题说明

### 3.3 什么不该放进任何公共入口

以下内容即使重要，也不应优先塞进 `AGENTS.md` 或 `CONTRIBUTING.md`：

- 规则如何维护、何时改规则的完整工作流
- 规则原意为什么容易被误读
- 顶层入口分工背后的长解释
- 只在人类维护者准备改规则时才需要的低频判断

这些内容更适合放在：

- `docs/contributing/meta-rules.md`
- `docs/contributing/intent/*.md`
- 必要时的 `cases/` 或临时交接材料

## 4. 什么时候该改顶层入口文件

顶层入口文件不应因为每次出现新提醒就扩写。只有当问题已经稳定表现为**入口层误导**时，才考虑修改 `AGENTS.md` 或 `CONTRIBUTING.md`。

较稳的触发信号包括：

- 协作者反复在进入仓库时走错路，由入口层缺少最小路由导致
- 某条高频约束只有补到顶层入口，才能稳定被想起
- 不改顶层入口，就会持续漏掉某个必须先做的分流动作

以下情况通常不应先改顶层入口：

- 只是一次性解释成本较高
- 只是某份下层规则条文还不够短或不够清楚
- 通过修改 `docs/contributing/` 下的专题规则即可解决
- 只是想让顶层入口“更完整”

默认顺序应是：

1. 先判断问题是否只是下层规则或正文分工问题
2. 若涉及修改规则或判断是否改规则，先读 [`../meta-rules.md`](../meta-rules.md)；若准备做重大规则变更，先重新通读一遍，不要依赖长上下文中的记忆
3. 只有当问题稳定属于入口层误导时，才回到 `AGENTS.md` 或 `CONTRIBUTING.md`

## 5. 为什么低频维护路由只应最小进入顶层

仓库顶层入口既要足够强，能把协作者送到正确路径；又不能承载太多低频维护说明，否则会持续占用日常任务上下文。

因此，更稳的做法通常是：

- 在顶层入口只保留**最短路由**
- 把完整工作流放在 `meta-rules.md`
- 把设计意图与误读来源放在 `intent/`

例如，“如果任务涉及修改规则文件或判断是否改规则，先读 `meta-rules.md`”适合保留在公共入口；但“为什么这句话放在 `CONTRIBUTING.md` 而不是 `AGENTS.md`”，则更适合保留在本文件，而不是反过来把顶层入口继续写重。

## 6. 与 `meta-rules` / `intent` 的关系

本文件不替代 `meta-rules.md`，也不替代 `intent/meta-rules.md`。

三者的关系应理解为：

- [`../meta-rules.md`](../meta-rules.md)：什么时候改规则、规则应该怎么改
- [`meta-rules.md`](./meta-rules.md)：为什么需要一层专门处理“规则如何维护”的规则与解释
- `repo-entry-files.md`：为什么仓库顶层入口文件要分层存在、各自承接什么、修改时应优先判断哪一层

如果问题已经不只是“这句提示该放哪”，而是“为什么这些顶层文件这样分工、为什么低频维护说明不能直接塞进公共入口”，优先读本文件。

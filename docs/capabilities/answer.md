# Answer

本文回答：**当系统已经积累出一定知识之后，如何基于这些内容回答问题，同时守住 Evidence 边界，不把观察、推断、候选输入和稳定结论混成一句话。**

它面向普通读者、人类维护者与 AI 协作者，关注的是一种公共能力：**基于已有知识进行回答与解释的能力**。

它不替代 `docs/contributing/evidence-and-traceability.md` 的组合工作流，也不替代 AI 协作者在 `AGENTS.md` 与 `CONTRIBUTING.md` 下必须遵守的行为约束。本文只把 Answer 这项能力单独显式化，避免它长期只停留在行为提醒层，而没有形成公共说明。

## 1. Answer 不是写作，也不是搜索本身

在这个仓库里，`Answer` 很容易被混成另外两件事：

- 写一篇新文档
- 在目录树里继续搜索或定位

但它们都还不是 `Answer` 本身。

`Answer` 处理的是更靠后的问题：

- 现有知识里，哪些已经足够支撑一个回答
- 哪些内容只能作为观察、推断或待验证线索出现
- 回答一个问题时，什么时候应该直接解释，什么时候应该回到仓库继续查找
- 如何把“不知道”也回答得清楚，而不是硬补成定论

所以，Answer 的起点不是“我要新写什么”，而是“基于当前已经有的内容，我现在最多能回答到哪里”。

## 2. Answer 的三个核心判断

做回答时，最稳妥的是依次回答三个问题：

1. **这个问题现在能不能直接回答？**
2. **能回答的部分，证据状态分别是什么？**
3. **不能直接回答的部分，应明确成缺口、推断还是冲突？**

如果第一个问题都还没想清楚，就不要急着组织漂亮表述。

## 3. 先判断当前问题是否已被系统承接

不是所有问题都应该被立刻回答成一段完整解释。

更稳的第一步是先判断：这个问题现在是否已经被仓库里的现有内容稳定承接。

常见情况包括：

- **已被正文或 `overview.md` 稳定承接**：可以直接回答
- **只在 `backlog.md`、`candidates.md`、`conflict.md` 中出现**：只能回答当前状态，不能冒充已有定论
- **只存在零散输入材料或临时观察**：通常还不能直接形成完整回答
- **系统里还没有这个问题的稳定承接位**：应先承认缺口，而不是假装仓库已经知道答案

这一步的重点不是“我能不能凭常识说几句”，而是“仓库当前到底已经稳定知道了什么”。

## 4. 回答前先区分三类内容

真正进入回答阶段前，最重要的不是文风，而是先把内容拆成三类：

- **稳定结论**：已经有足够承接，且证据状态足以支持直接回答
- **受限判断**：只能在对象、版本、场景或观察范围内成立
- **未稳定内容**：仍属于推断、待验证缺口或冲突状态

如果不先拆这三类，最容易发生的就是：

- 把 `Observed` 写成通用结论
- 把 `Inferred` 写成直接事实
- 把 `Unverified` 写成仓库已有共识
- 把 `Conflicting` 写成单边答案

## 5. Evidence 状态决定回答力度

`Answer` 最核心的不是“说得多完整”，而是“回答力度和 Evidence 状态是否匹配”。

一个最小对应关系可以先这样理解：

- `Verified`：可以作为相对稳定的直接回答基础
- `Observed`：可以回答，但要保留观察对象、范围和限制
- `Inferred`：可以解释推断路径，但不能伪装成直接证据
- `Unverified`：只能作为待验证线索，不应写成结论
- `Conflicting`：应优先说明存在冲突，而不是直接替用户选边
- `Deprecated`：应说明它曾经成立，但当前不再适用

回答问题时，真正危险的往往不是“答少了”，而是**回答力度超过了证据力度**。

## 6. 一个最小的 Answer 顺序

拿不准时，可以先按下面顺序判断：

```text
收到一个问题
  → 仓库里是否已有稳定承接？
  → 当前能直接回答的是哪些部分？
  → 这些部分各自对应什么 Evidence 状态？
  → 还有哪些部分只能写成观察、推断、冲突或缺口？
  → 是否需要回到仓库继续查找，而不是直接补全？
```

最后一步很重要。

`Answer` 不是“先说出来再看要不要补证”，而是“先判断当前知识边界，再决定要不要继续检索”。

## 7. 三种最常见的回答形态

### 7.1 直接回答

当问题已经被正文、`overview.md` 或其他稳定知识位较好承接时，可以直接给出主回答。

这类回答的要求是：

- 结论边界清楚
- 不额外放大已有内容的确定性
- 必要时指出适用范围或前提

### 7.2 受限回答

当系统里有部分答案，但它只在特定对象、版本、实验或上下文中成立时，应给出受限回答。

这时重点不是追求一句话说满，而是明确：

- 它对谁成立
- 在什么范围内成立
- 为什么现在只能回答到这里

### 7.3 缺口回答

当系统当前并没有稳定答案，或只有冲突、候选对象、临时输入时，应直接回答当前缺口状态。

也就是说，可以回答：

- 仓库现在还没有稳定结论
- 目前只有哪些观察、推断或冲突
- 若要继续回答，下一步应回到哪里补查

这不是“没回答”，而是把当前知识边界回答清楚了。

## 8. 三个最常见的 Answer 误判

### 8.1 误判一：只要模型能组织出完整段落，就说明系统已经有答案

不是。

模型能把话说顺，不等于仓库已经有稳定结论。表达流畅只是语言能力，不是知识状态证明。

### 8.2 误判二：回答时最好把不确定部分都藏起来，免得显得不完整

不是。

在这个仓库里，真正有害的不是承认不知道，而是把未稳定内容伪装成已知结论。

### 8.3 误判三：Answer 只是 AI 协作者的临场习惯，不值得单独显式化

也不是。

因为这项能力不只属于 AI。普通读者、人类维护者和 AI 协作者都会面临同一个问题：**系统现在到底知道什么，不知道什么，能回答到什么力度。**

## 9. Answer 与其他文档的分工

### 9.1 与 `navigate.md` 的区别

- `navigate.md`：回答“当我只有一个问题时，如何找到最可能的入口或目标位置”
- `answer.md`：回答“找到相关内容后，如何基于已有知识真正作答”

前者偏找位置，后者偏用知识作答。

### 9.2 与 `ingest.md` 的区别

- `ingest.md`：回答“新材料进入系统时，如何先分流和降级处理”
- `answer.md`：回答“已有材料进入知识系统后，如何被拿来回答问题”

前者更早，后者更晚。

### 9.3 与 `evidence-and-traceability.md` 的区别

- `evidence-and-traceability.md`：回答新增或修改内容时，Evidence 与 Trace 应如何组合执行
- `answer.md`：回答在不新增内容的前提下，如何根据现有 Evidence 边界控制回答力度

前者偏写入治理，后者偏读取与解释治理。

## 10. 当前最值得优先警惕的回答场景

按当前仓库状态，最值得优先警惕的通常是：

- 只依据外部 AI 调研或联网结果做直接回答
- 把 `overview.md` 之外的零散输入拼成“仿佛已稳定”的主线解释
- 在对象级观察基础上，直接推广成跨对象结论
- 面对冲突材料时，跳过冲突状态直接给单边答案

这些场景最容易让系统看起来很会回答，但其实已经越过了自己的知识边界。

## 11. 使用边界

如果你当前要做的是：

- 想先找到问题应去哪个目录：优先读 [`navigate.md`](./navigate.md)
- 想处理一份新材料该先放哪里：优先读 [`ingest.md`](./ingest.md)
- 想判断找到区域后该写什么文件：读 [`place.md`](./place.md)
- 想理解回答时为什么不能把观察、推断和定论混写：从本文开始

## 12. 本文的边界

本文不负责：

- 代替 Evidence 规则定义所有状态细节
- 代替 Traceability 规则定义完整来源链路
- 代替 `documentation-workflow.md` 给出新增内容的操作步骤
- 对某个具体问题直接给出最终知识结论

它只负责把一个长期容易被默认成“模型自己会处理”的能力单独说清楚：**系统如何基于已有知识回答问题，同时不越过自己的证据边界。**

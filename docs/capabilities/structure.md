# Structure

本文回答：**如何让目录树、README、index 与各类元信息文件长期保持分工清楚、边界稳定、可导航、可维护，而不是随着内容增长慢慢重新混成一团。**

它面向普通读者、人类维护者与 AI 协作者，关注的是一种公共能力：**让知识系统持续保持结构清晰的能力**。

它不替代 `docs/contributing/readme-rules.md`、`docs/contributing/metadata-files.md` 或 `docs/contributing/organization-principles.md` 的具体规则细则。本文只把 Structure 这项能力单独显式化，避免它长期只散落在 README 规则、元信息文件规则和结构原则里。

## 1. Structure 不是一次重构，也不是单篇文档写法

在这个仓库里，`Structure` 很容易被误当成两类更具体的事情：

- 某次目录迁移、改名或链接修复
- 某篇 README、overview 或 landscape 该怎么写

但它们都还不是 `Structure` 本身。

`Structure` 关注的是更上层的问题：

- 这套目录树现在是否还能稳定承接后续内容增长
- README、index、overview、landscape 之间是否仍然各答各的问题
- 读者与写作者进入一个目录时，是否还能自然知道该往哪走、该把内容放哪
- 哪些职责已经开始重新混写，哪些边界已经开始漂移

所以，Structure 的起点不是“我要改哪一个文件”，而是“这套系统现在是否还保持结构可用”。

## 2. Structure 的四个核心对象

长期保持结构清晰时，最常要同时看的通常是四类对象：

1. **目录树本身**：父子层级、分类轴、相邻目录边界
2. **入口文件**：`README.md` 与 `index.md`
3. **理解与结构文件**：`overview.md` 与 `landscape.md`
4. **其余元信息文件**：`backlog.md`、`candidates.md`、`roadmap.md`、`conflict.md`

Structure 的难点不在于单个文件怎么定义，而在于这些对象会彼此牵动：

- 目录边界不稳，README 就会越写越重
- README 承担了目录树，index 就会被推迟甚至消失
- overview 和 landscape 不拆，主线理解与结构研究就会互相打架
- backlog、candidates、roadmap、conflict 分不清，未稳定问题就会重新挤回正文或 README

## 3. Structure 的三个核心判断

做结构判断时，最稳妥的是依次回答三个问题：

1. **当前这一层主要在回答什么问题？**
2. **这个问题现在由哪个目录或文件在承接？**
3. **它是否已经开始吞并相邻职责？**

只要第三个问题的答案开始变成“是”，通常就说明结构能力已经在衰减。

## 4. 先看目录树是否仍在表达稳定分类轴

Structure 的第一步，不是数文件多少，而是先看目录树本身是否还在表达稳定分类轴。

更稳的判断包括：

- 同一层级的并列目录是否在回答同一种问题
- 目录名是否仍能稳定承接未来增长后的内容
- 知识目录、对象目录、材料目录是否开始混层并列
- 当前目录是否正在靠历史上下文勉强维持，而不是靠名字和边界自然成立

如果目录树本身已经开始误导落位，再精细地修 README 或 index，通常也只是表面修补。

## 5. README 与 index 是结构接口，不是互相兜底的万能页

在这套系统里，`README.md` 与 `index.md` 不是可互换的两个入口，而是两个不同接口：

- `README.md`：回答“这是什么地方，为什么存在，和相邻目录如何区分”
- `index.md`：回答“这个目录下面有什么，我已经知道想找什么时该去哪”

因此，Structure 能力的一个核心任务，就是长期阻止两种退化：

- README 重新吞回目录结构展示与主要查找导航
- index 反过来承担主题解释、主线理解或阅读路径

一句话说：**README 负责定向，index 负责路由。**

只要目录下已有两个及以上语义稳定、预期会持续保留的子目录，Structure 就应倾向于尽早把结构展示交给骨架型 `index.md`，而不是继续扩写 README。

当搜索体验变差时，也不要只把问题理解成“搜索不够强”。很多时候，真正退化的是结构接口本身：

- 读者无法先通过 README 判断最高稳定归属
- 进入目录后没有 index 可继续缩小范围
- 本应由子层 index 承接的定位，被迫回退成父层 README + 人工猜测

因此，更稳的搜索体验通常不是在目录树之外另造一套方法，而是先守住一种混合顺序：**先用 README / index 缩小范围，再在候选支路内做定向搜索。** 这样做的重点不是追求一次命中最深文件，而是先保留结构感，再补速度与跨支路命中能力。

## 6. overview 与 landscape 解决的是两类不同的结构压力

`overview.md` 与 `landscape.md` 经常被当成“都算总览”，但从 Structure 视角看，它们承受的是不同压力：

- `overview.md` 承接的是人类向主线理解压力
- `landscape.md` 承接的是结构切分、边界稳定与下钻协作压力

所以，Structure 不是简单问“要不要有一篇总文”，而是先问：

- 当前最先稳定的是主线理解，还是结构边界？
- 一篇文档是否已经同时在讲“主题本身”和“本目录为什么这样组织”？
- 如果继续混在一起，会不会同时损害可读性和结构判断？

当这两个压力已经开始互相干扰时，Structure 的答案通常不是继续硬写一篇更长的文档，而是把 `overview.md` 与 `landscape.md` 拆开。

## 7. 其余元信息文件是在保护结构，不是在补形式完整

从 Structure 视角看，`backlog.md`、`candidates.md`、`roadmap.md`、`conflict.md` 的意义，不只是“多几个辅助文件”，而是防止未稳定问题重新挤进不该承接它们的地方。

它们各自保护的结构压力是：

- `backlog.md`：保护正文不被内容缺口占位
- `candidates.md`：保护 backlog 不被对象队列挤满
- `roadmap.md`：保护 README / index 不去承担顺序路径
- `conflict.md`：保护正文不被未解决冲突伪装成单一结论

所以，这些文件是否出现，不取决于“模型里有没有这个槽位”，而取决于结构当前首先暴露出了哪类压力。

## 8. 一个最小的 Structure 顺序

拿不准时，可以先按下面顺序判断：

```text
发现结构开始变重或变乱
  → 目录树本身的分类轴还稳不稳？
  → 当前最先混掉的是目录边界、README/index，还是 overview/landscape？
  → 这是说明问题、导航问题、理解问题、结构问题、缺口问题、对象问题、路径问题，还是冲突问题？
  → 哪个文件现在正在吞并不属于自己的职责？
  → 最小修复是补分工、补入口、补骨架 index，还是拆出新的元信息文件？
```

最后一步很重要。

Structure 的目标不是一次把所有文档都补齐，而是把当前已经混写的主问题重新分开承接。

## 9. 四个最常见的 Structure 误判

### 9.1 误判一：当前内容还少，所以结构问题可以以后再说

不是。

很多结构问题最贵的地方，恰恰在于早期不处理，后面会把迁移、重写和认知重建成本一起放大。

### 9.2 误判二：Structure 就是把目录树整理得更整齐

不是。

视觉整齐不是重点。真正的重点是：后续写作者是否更容易做对落位判断，读者是否更容易知道该往哪走。

### 9.3 误判三：只要 README 足够详细，就不需要 index 或其他元信息文件

不是。

README 变详细，常常只是说明它已经开始吞并导航、结构或路径职责，而不是说明结构已经更合理。

### 9.4 误判四：有了元信息文件模型，就应该按层级把文件配齐

也不是。

这些文件是按问题触发的能力位，不是按层级补齐的套装。Structure 追求的是职责分离，不是形式对称。

## 10. Structure 与其他文档的分工

### 10.1 与 `readme-rules.md` 的区别

- `readme-rules.md`：回答 README 什么时候出现、该写什么、不该写什么
- `structure.md`：回答为什么 README 不应重新吞回结构展示与导航职责，以及它在整套系统里的接口位置

前者偏 README 细则，后者偏结构能力。

### 10.2 与 `metadata-files.md` 的区别

- `metadata-files.md`：回答各类元信息文件分别承接什么认知任务、何时出现
- `structure.md`：回答这些文件为什么需要长期分开承接，以及一旦混写会造成什么结构退化

前者偏文件边界，后者偏系统分工。

### 10.3 与 `organization-principles.md` 的区别

- `organization-principles.md`：回答什么样的目录结构才算长期合理
- `structure.md`：回答这些长期原则如何在 README、index、overview、landscape 与其他元信息文件的分工上落地

前者偏上位原则，后者偏运行中的结构能力。

### 10.4 与 `maintain.md` 的区别

- `maintain.md`：回答系统长期运行后，如何识别并处理规则、案例、结构和入口层的老化与漂移
- `structure.md`：回答结构本身需要怎样分工，才不至于持续老化

前者偏长期维护调度，后者偏结构能力本身。

## 11. 当前最值得优先警惕的结构场景

按当前仓库状态，最值得优先警惕的通常是：

- README 又开始代行目录树或查找导航
- index 开始替代 overview、landscape 或 roadmap
- 一篇 overview 同时在讲主题主线和目录切分逻辑
- backlog、candidates、roadmap、conflict 的问题重新挤回 README 或正文

这些场景看起来只是局部写重了一点，但本质上都是结构能力在退化。

## 12. 使用边界

如果你当前要做的是：

- 想判断目录为什么这样组织、长期是否合理：优先读 [`../contributing/organization-principles.md`](../contributing/organization-principles.md)
- 想看 README 的具体写法和触发条件：优先读 [`../contributing/readme-rules.md`](../contributing/readme-rules.md)
- 想看各类元信息文件的完整职责边界：读 [`../contributing/metadata-files.md`](../contributing/metadata-files.md)
- 想理解为什么这些文件不能重新混写成几个万能入口：从本文开始

## 13. 本文的边界

本文不负责：

- 代替 README 规则定义所有写法细节
- 代替元信息文件模型定义每类文件的完整触发条件
- 代替结构重构规则给出具体迁移步骤
- 对某个具体目录直接给出最终重构方案

它只负责把一个长期容易被拆碎在多个规则文件里的能力单独说清楚：**这套知识系统如何持续保持结构清晰，而不是慢慢让不同职责重新混成少数几个万能文件。**

# Ingest

本文回答：**当一份新材料刚进入这套仓库时，系统如何先识别它是什么、证据状态如何、今天先落到哪里，并在条件成熟后继续吸收进稳定知识层。**

它面向普通读者、人类维护者与 AI 协作者，关注的是一种公共能力：**输入进入系统时的分流、吸收与回流能力**。

它不替代 [`docs/contributing/documentation-workflow.md`](../contributing/documentation-workflow.md) 的规则化执行入口，也不替代 Evidence / Traceability 规则的原子定义。本文只把 `Ingest` 这项能力单独显式化，避免它长期分散在流程规则、案例结构规则和证据规则里。

## 1. Ingest 不是写作，而是写作前后的吸收动作

在这个仓库里，很多返工并不是因为不会写，而是因为**输入一进来就被过早写成正文**，或者在已经进入 `temp/` 后又被直接润色回正文。

`Ingest` 处理的不是“这一段怎么表达更好”，而是更前面和更中间的问题：

- 这份材料到底是什么。
- 它现在能支撑多强的结论。
- 今天不写正文时，最合适的暂存位置是什么。
- 它应该先被当作缺口、对象、冲突，还是正文候选。
- 当它已经不该继续留在 `temp/` 时，应先进入哪个稳定中间层。

所以，Ingest 的起点不是“我要写一篇什么文档”，而是“我手上现在拿到了一份什么输入，它接下来该如何被系统吸收”。

## 2. Ingest 的四个核心判断

拿到新材料后，最稳的顺序通常是先回答四个问题：

1. **材料类型是什么？**
2. **Evidence 状态是什么？**
3. **目标落位最可能是什么？**
4. **如果今天不写正文，临时落点是什么？**

这四个问题里，只要前两个没想清楚，就不要急着回答后两个。

## 3. 先判断材料类型

`Ingest` 的第一步，不是看材料写得像不像综述，而是先判断它属于哪类输入。

这个仓库里最常见的输入包括：

- 新论文阅读笔记
- 外部 AI 调研结果
- 联网检索 / 网页抓取结果
- 博客摘录、issue / PR 讨论
- 对开源仓库、产品或 benchmark 的观察记录
- 已有仓库内容上的补充笔记或临时草稿

这些输入默认都还不是正文，也不默认等于主线结论。

更实用的最小判断是：

- **整体认知型输入**：更可能通向 `overview.md`
- **缺口型输入**：更可能通向 `backlog.md`
- **对象型输入**：更可能通向 `candidates.md`
- **冲突型输入**：更可能通向 `conflict.md`
- **正文候选输入**：只有在问题边界和证据状态都相对稳定时，才可能继续进入正文

## 4. 再判断 Evidence 状态

类型判断之后，下一步是判断这份输入现在能支撑多强的结论。

最常用的六种状态是：

- `Verified`
- `Observed`
- `Inferred`
- `Unverified`
- `Conflicting`
- `Deprecated`

在 Ingest 视角下，这六种状态最重要的作用不是做标签装饰，而是决定**这份输入暂时能走到系统里的哪一步**。

一个最小理解可以是：

- `Verified`：已经可能进入正文或 `overview.md` 的候选区
- `Observed`：可能进入正文，但通常还需要保留对象、范围和限制
- `Inferred`：可以进入结构化草稿，但不能伪装成直接证据
- `Unverified`：先保留在线索层，不写成正文定论
- `Conflicting`：优先进入 `conflict.md`
- `Deprecated`：不再直接作为当前主线结论使用

如果拿不准，默认降级，而不是默认升格。

## 5. Ingest 的默认目标不是正文

这点很重要。

很多输入看起来已经很完整，尤其是：

- 外部 AI 调研报告
- 联网检索整理
- 写得很像综述的博客或笔记
- 从源码观察中归纳出的暂时判断

但“看起来完整”不等于“已经适合正文”。

在这个仓库里，Ingest 的默认目标更接近：

- 先进入元信息文件
- 先保留在 `temp/`
- 先显式记录 Evidence / Trace / Needs
- 先把问题分流清楚

只有当输入已经同时满足：

- 问题边界相对稳定
- Evidence 状态足够支撑
- 目标归属已经相对明确

才适合继续进入正文。

## 6. 四类最常见的临时落点

如果今天不直接写正文，最常见的临时落点通常是这四类：

- `temp/`：还只是输入材料，归属或状态都未清
- `backlog.md`：当前首先稳定的是内容缺口
- `candidates.md`：当前首先稳定的是待研究对象
- `conflict.md`：当前首先稳定的是定义、事实、版本或口径冲突

这四类落点不是“次优垃圾桶”，而是系统防止过早写定论的重要缓冲层。

## 7. 何时从临时层进入稳定中间层

有些输入虽然还不足以进入正文，但也已经不适合继续只留在 `temp/`。这时，`Ingest` 不只回答“先别写正文”，还要回答“那先进入哪个稳定中间层”。

最常见的中间层有两类：

- **元信息文件层**：`backlog.md`、`candidates.md`、`conflict.md`
- **对象目录内的辅助材料层**：具体案例研究目录下的 `notes/`

最小判断是：

- 如果当前首先稳定的是**缺口 / 对象 / 冲突**，优先进入对应元信息文件。
- 如果材料已经**明确归属于某个具体对象目录**，但还不足以直接形成对象正文或机制专题，可先进入该对象目录的 `notes/`。
- 如果连对象归属都还不稳，继续留在 `temp/`，不要为了离正文更近而强行塞进某个对象目录。

## 8. 外部临时材料何时先回流到对象目录 `notes/`

当输入材料来自 `temp/`、外部 AI 调研、联网检索 / 网页抓取结果、博客整理或临时笔记时，不要直接“把材料改写进正文”。

如果它已经同时满足下面两个条件：

1. **对象归属已经稳定**：已经明确知道它属于哪个具体论文、代码库、产品、协议或 benchmark 目录。
2. **正文条件仍未满足**：虽然已经不该只留在 `temp/`，但还不足以直接进入 `overview.md`、`architecture.md` 或 `<mechanism>.md`。

那么它就可以先进入该对象目录的 `notes/`，作为对象内的稳定中间层。

典型情形包括：

- 材料已经和某个具体对象强绑定，但仍包含较多未压实的源码线索、长引用、对照表或待核验问题。
- 当前价值主要在于为后续正文提供 claim-source 对照、来源恢复、局部核验或研究脚手架，而不是直接形成给普通读者的稳定叙述。
- 结论已经不适合继续只留在 `temp/`，但还不足以直接写进对象正文。

在对象目录中，最常见的进一步分流是：

- `notes/source.md`：上游来源摘录、源码核验路径、长引用、失败搜索和版本语义。
- `notes/evidence.md`：Claim-Source 对照、Evidence 状态、局部结论和待补证点。
- `notes/general.md`：尚未结构化为 source / evidence 的对象内研究过程材料。

这一步的关键不是“给临时材料找个地方放”，而是：**让已经完成对象归属判断的材料，从输入层进入对象内可维护的稳定中间层。**

## 9. 外部临时材料回流正文时的标准能力链

从外部临时材料进入正文时，最稳的吸收链不是“先润色再补来源”，而是：

```text
外部输入 / temp 材料
  → 拆 Claim
  → 区分证据类型
  → 恢复真实来源
  → 判断 Evidence 状态
  → 决定落位（正文 / 元信息 / notes）
  → 只回流当前主题直接相关的部分
  → 改写 Trace，摆脱对临时入口的依赖
```

这条链条里最容易被跳过的是中间三步：

- 把完整调研稿当成可直接搬运的段落，而不是先拆 Claim
- 不恢复真实来源，直接让 `temp/` 或“某轮外部调研”充当最终来源
- 不先判断应该进 `backlog.md` / `candidates.md` / `conflict.md` 还是对象目录 `notes/`，就急着写正文

所以，Ingest 并不止是“新材料进来时的第一步”，它也包含“外部输入如何被系统吸收到稳定层”的后半程。

## 10. 一个最小 Ingest 顺序

拿不准时，可以先按下面这个顺序做最小判断：

```text
拿到新材料
  → 这是什么类型的输入？
  → 现在最多能标成哪种 Evidence 状态？
  → 当前更像整体认知、缺口、对象、冲突，还是正文候选？
  → 今天不写正文时，先留在 temp、backlog、candidates、conflict，还是对象目录 notes？
  → 如果后续回流正文，还缺什么验证、来源或落位判断？
```

这里最后一步很关键。Ingest 的目标不是一次性把材料处理完，而是把后续路径保留下来。

## 11. 三个最常见的 Ingest 误判

### 11.1 误判一：只要来源是实时获取的，就比仓库里的旧内容更可靠

不是。

联网检索、网页抓取或外部 AI 调研结果，默认仍只是上游输入。它们是否可靠，取决于来源质量、证据状态和是否能追溯，而不是取决于“这是刚查到的”。

### 11.2 误判二：只要材料写得很完整，就应该先整理成正文

不是。

很多完整材料其实只是把未验证内容写得很像定论。对这类输入，更稳的做法通常是先分流，再决定是否回流正文。

### 11.3 误判三：只要已经不留在 `temp/`，就应直接进入正文

也不是。

在很多对象研究场景里，更稳的中间层是对象目录下的 `notes/`，而不是直接把外部材料润色进 `overview.md` 或机制专题。

## 12. Ingest 与其他文档的分工

### 12.1 与 `documentation-workflow.md` 的区别

- `documentation-workflow.md`：回答“真正动手时，哪些判断顺序、门槛和 stop-line 不能跳过”
- `ingest.md`：回答“为什么输入进入系统时，必须先分型、分级、分流，并在条件成熟后再回流稳定层”

前者偏规则护栏，后者偏输入吸收能力。

### 12.2 与 `evidence-and-traceability.md` 的区别

- `evidence-and-traceability.md`：回答 Evidence 与 Trace 应如何组合执行
- `ingest.md`：回答为什么新输入在进入正文前，必须先接受这套判断

前者更像组合工作流，后者更像系统入口能力。

### 12.3 与 `place.md` 的区别

- `ingest.md`：回答新输入进来以后，先如何分流、暂存、吸收
- `place.md`：回答已经找到稳定区域后，应停在哪一层、写成什么文件

两者是前后相邻的能力，但不相同。Ingest 更早，也覆盖从临时层进入稳定中间层的过渡；Place 更偏已经找到区域后的落位决策。

### 12.4 与 `research-artifacts.md` 的区别

- `ingest.md`：回答为什么某些对象相关输入应先回流到对象目录 `notes/`
- `research-artifacts.md`：回答对象目录里的 `notes/`、正文、机制专题和辅助材料如何分工

一个负责“为什么先进这里”，一个负责“进来以后内部怎么摆”。

## 13. 当前最值得优先警惕的输入类型

按当前仓库状态，最值得优先警惕的通常是：

- 外部 AI 调研结果
- 联网检索 / 网页抓取结果
- 博客摘录或社区讨论
- 没有版本基线的源码观察结论

这些输入最容易出现的共同问题是：**写得很像知识，但其实仍是候选证据或待分流材料。**

## 14. 使用边界

如果你当前要做的是：

- 实际执行新增、迁移、修改流程：优先读 [`../contributing/documentation-workflow.md`](../contributing/documentation-workflow.md)
- 判断具体的 Evidence / Trace 字段怎么写：优先读 [`../contributing/evidence-and-traceability.md`](../contributing/evidence-and-traceability.md)
- 已经找到稳定区域，想判断该写什么文件：读 [`place.md`](./place.md)
- 想理解对象目录里的 `notes/` / 正文 / 机制专题怎么分工：读 [`../contributing/research-artifacts.md`](../contributing/research-artifacts.md)
- 想理解为什么新输入不能直接跳过分流写成正文：从本文开始

## 15. 本文的边界

本文不负责：

- 代替 Evidence 规则定义所有状态细节
- 代替 Traceability 规则定义完整链路字段
- 代替 `documentation-workflow.md` 给出具体操作清单
- 代替 `research-artifacts.md` 规定案例目录内部的全部产物结构
- 对某份具体输入直接给出最终落位裁决

它只负责把一个长期容易被流程细节掩盖的能力单独说清楚：**新材料进入系统时，如何先被正确吸收，并在条件成熟后再进入稳定知识层，而不是被过早写成知识定论。**

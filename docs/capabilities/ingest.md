# Ingest

本文回答：**当一份新材料刚进入这套仓库时，如何先判断它是什么、证据状态如何、今天先落到哪里，而不是一上来就抢写正文。**

它面向普通读者、人类维护者与 AI 协作者，关注的是一种公共能力：**输入进入系统时的分流与吸收能力**。

它不替代 `docs/contributing/documentation-workflow.md` 的完整操作流程，也不替代 Evidence / Traceability 规则的原子定义。本文只把 Ingest 这项能力单独显式化，避免它长期分散在流程规则和证据规则里。

## 1. Ingest 不是写作，而是写作前的吸收动作

在这个仓库里，很多返工并不是因为不会写，而是因为**输入一进来就被过早写成正文**。

`Ingest` 处理的不是“这一段怎么表达更好”，而是更前面的问题：

- 这份材料到底是什么
- 它现在能支撑多强的结论
- 今天不写正文时，最合适的暂存位置是什么
- 它应该先被当作缺口、对象、冲突，还是正文候选

所以，Ingest 的起点不是“我要写一篇什么文档”，而是“我手上现在拿到了一份什么输入”。

## 2. Ingest 的四个核心判断

拿到新材料后，最稳的顺序通常是先回答四个问题：

1. **材料类型是什么？**
2. **Evidence 状态是什么？**
3. **目标落位最可能是什么？**
4. **如果今天不写正文，临时落点是什么？**

这四个问题里，只要前两个没想清楚，就不要急着回答后两个。

## 3. 先判断材料类型

`Ingest` 的第一步，不是看材料写得像不像综述，而是先判断它属于哪类输入。

这个仓库里最常见的输入包括：

- 新论文阅读笔记
- 外部 AI 调研结果
- 联网检索 / 网页抓取结果
- 博客摘录、issue / PR 讨论
- 对开源仓库、产品或 benchmark 的观察记录
- 已有仓库内容上的补充笔记或临时草稿

这些输入默认都还不是正文，也不默认等于主线结论。

更实用的最小判断是：

- **整体认知型输入**：更可能通向 `overview.md`
- **缺口型输入**：更可能通向 `backlog.md`
- **对象型输入**：更可能通向 `candidates.md`
- **冲突型输入**：更可能通向 `conflict.md`
- **正文候选输入**：只有在问题边界和证据状态都相对稳定时，才可能继续进入正文

## 4. 再判断 Evidence 状态

类型判断之后，下一步是判断这份输入现在能支撑多强的结论。

最常用的六种状态是：

- `Verified`
- `Observed`
- `Inferred`
- `Unverified`
- `Conflicting`
- `Deprecated`

在 Ingest 视角下，这六种状态最重要的作用不是做标签装饰，而是决定**这份输入暂时能走到系统里的哪一步**。

一个最小理解可以是：

- `Verified`：已经可能进入正文或 `overview.md` 的候选区
- `Observed`：可能进入正文，但通常还需要保留对象、范围和限制
- `Inferred`：可以进入结构化草稿，但不能伪装成直接证据
- `Unverified`：先保留在线索层，不写成正文定论
- `Conflicting`：优先进入 `conflict.md`
- `Deprecated`：不再直接作为当前主线结论使用

如果拿不准，默认降级，而不是默认升格。

## 5. Ingest 的默认目标不是正文

这点很重要。

很多输入看起来已经很完整，尤其是：

- 外部 AI 调研报告
- 联网检索整理
- 写得很像综述的博客或笔记
- 从源码观察中归纳出的暂时判断

但“看起来完整”不等于“已经适合正文”。

在这个仓库里，Ingest 的默认目标更接近：

- 先进入元信息文件
- 先保留在 `temp/`
- 先显式记录 Evidence / Trace / Needs
- 先把问题分流清楚

只有当输入已经同时满足：

- 问题边界相对稳定
- Evidence 状态足够支撑
- 目标归属已经相对明确

才适合继续进入正文。

## 6. 四类最常见的临时落点

如果今天不直接写正文，最常见的临时落点通常是这四类：

- `temp/`：还只是输入材料，归属或状态都未清
- `backlog.md`：当前首先稳定的是内容缺口
- `candidates.md`：当前首先稳定的是待研究对象
- `conflict.md`：当前首先稳定的是定义、事实、版本或口径冲突

这四类落点不是“次优垃圾桶”，而是系统防止过早写定论的重要缓冲层。

## 7. 一个最小 Ingest 顺序

拿不准时，可以先按下面这个顺序做最小判断：

```text
拿到新材料
  → 这是什么类型的输入？
  → 现在最多能标成哪种 Evidence 状态？
  → 当前更像整体认知、缺口、对象、冲突，还是正文候选？
  → 今天不写正文时，先留在 temp、backlog、candidates 还是 conflict？
  → 如果后续回流正文，还缺什么验证、来源或落位判断？
```

这里最后一步很关键。Ingest 的目标不是一次性把材料处理完，而是把后续路径保留下来。

## 8. 三个最常见的 Ingest 误判

### 8.1 误判一：只要来源是实时获取的，就比仓库里的旧内容更可靠

不是。

联网检索、网页抓取或外部 AI 调研结果，默认仍只是上游输入。它们是否可靠，取决于来源质量、证据状态和是否能追溯，而不是取决于“这是刚查到的”。

### 8.2 误判二：只要材料写得很完整，就应该先整理成正文

不是。

很多完整材料其实只是把未验证内容写得很像定论。对这类输入，更稳的做法通常是先分流，再决定是否回流正文。

### 8.3 误判三：Ingest 只是 documentation workflow 的第一步，不值得单独理解

也不是。

因为它不只服务维护动作，也服务系统长期稳定吸收外部输入的能力。如果这一步长期隐形，系统就会不断把上游输入误写成已整理知识。

## 9. Ingest 与其他文档的分工

### 9.1 与 `documentation-workflow.md` 的区别

- `documentation-workflow.md`：回答“拿到材料后，实际按什么步骤动手”
- `ingest.md`：回答“为什么材料进入系统时，必须先做类型判断、证据判断和临时分流”

前者偏执行流程，后者偏输入吸收能力。

### 9.2 与 `evidence-and-traceability.md` 的区别

- `evidence-and-traceability.md`：回答 Evidence 与 Trace 应如何组合执行
- `ingest.md`：回答为什么新输入在进入正文前，必须先接受这套判断

前者更像组合工作流，后者更像系统入口能力。

### 9.3 与 `place.md` 的区别

- `ingest.md`：回答新输入进来以后，先如何分流
- `place.md`：回答已经找到稳定区域后，应停在哪一层、写成什么文件

两者是前后相邻的能力，但不相同。Ingest 更早，Place 更后。

## 10. 当前最值得优先警惕的输入类型

按当前仓库状态，最值得优先警惕的通常是：

- 外部 AI 调研结果
- 联网检索 / 网页抓取结果
- 博客摘录或社区讨论
- 没有版本基线的源码观察结论

这些输入最容易出现的共同问题是：**写得很像知识，但其实仍是候选证据或待分流材料。**

## 11. 使用边界

如果你当前要做的是：

- 实际执行新增、迁移、修改流程：优先读 [`../contributing/documentation-workflow.md`](../contributing/documentation-workflow.md)
- 判断具体的 Evidence / Trace 字段怎么写：优先读 [`../contributing/evidence-and-traceability.md`](../contributing/evidence-and-traceability.md)
- 已经找到稳定区域，想判断该写什么文件：读 [`place.md`](./place.md)
- 想理解为什么新输入不能直接跳过分流写成正文：从本文开始

## 12. 本文的边界

本文不负责：

- 代替 Evidence 规则定义所有状态细节
- 代替 Traceability 规则定义完整链路字段
- 代替 `documentation-workflow.md` 给出具体操作清单
- 对某份具体输入直接给出最终落位裁决

它只负责把一个长期容易被流程细节掩盖的能力单独说清楚：**新材料进入系统时，如何先被正确吸收，而不是被过早写成知识定论。**

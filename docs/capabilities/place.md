# Place

本文回答：**当你已经大致找到目标区域之后，如何决定内容应停在哪一层、应写成正文还是哪类元信息文件。**

它面向普通读者、人类维护者与 AI 协作者，关注的是一种公共能力：**定位完成后的落位决策**。

它不替代 `docs/contributing/documentation-workflow.md` 的操作流程，也不替代 `docs/contributing/metadata-files.md` 的完整职责定义。本文只把 Place 这项能力单独显式化，避免它长期混在 Navigate 或规则细则里。

## 1. Place 不是 Navigate

在这个仓库里，`Navigate` 与 `Place` 很容易被混成一步，但它们其实回答的是两个不同问题：

- `Navigate`：我该先去哪个目录、哪个入口、哪一层继续下钻
- `Place`：我已经找到一片稳定区域后，这份内容应停在哪一层、写成什么文件

一个简化的顺序是：

```text
问题 / 新材料
  → Navigate：先找到最可能的目标区域
  → Place：再决定应停在哪一层、写什么文件
```

所以，Place 的起点不是“什么都不知道”，而是你已经大致知道内容应该落在仓库的哪一片区域里。

## 2. Place 的三个核心判断

做落位决策时，最稳妥的是依次回答三个问题：

1. **最高稳定归属在哪一层？**
2. **该停在这一层，还是继续下沉？**
3. **停下来以后，应写正文还是元信息文件？**

这三个问题里，只要前一个没想清楚，就不要急着回答后一个。

## 3. 先判断最高稳定归属

Place 的第一步，不是选文件名，而是先找当前内容的**最高稳定归属**。

这里的意思是：在哪个目录层级下，你已经可以相对确定地说“它属于这里”，而不需要依赖尚未稳定的更细分类轴。

默认判断：

- 一级目录归属稳定后，再判断二级、三级是否稳定
- 下层归属不稳时，先停在更高一层
- 如果连较高层级都不稳，说明当前主要问题不是“写什么文件”，而是“归属未定”

遇到这种情况，不要强行下沉到猜测中的子目录；优先把归属未定本身保留为待澄清问题。

## 4. 再判断是否继续下沉

已经找到最高稳定归属后，下一步不是自动继续下钻，而是判断：**当前内容是否已经足够支撑更细一级的稳定归属。**

可继续下沉的常见信号：

- 下一级目录的边界已经清楚
- 这份内容主要回答的是一个更细子问题，而不是上层横跨性问题
- 把它放进更细目录，不会让该目录的分类轴被不稳定判断带偏

不应继续下沉的常见信号：

- 当前内容本身就在帮助判断子目录边界
- 它同时触及多个相邻子目录，主问题仍停留在上层
- 下一级目录的归属还依赖更多调研或后续结构调整

一句话说：**只有当更细一层的归属也已经稳定时，才继续下沉。**

## 5. 停下来以后，先问“这是正文还是元信息”

确定层级以后，才开始判断文件类型。

最重要的分界不是文件名，而是：

- 这是已经相对稳定、适合写成正文的内容
- 还是仍在帮助系统承接缺口、对象、边界、冲突、顺序等未稳定问题

### 5.1 适合正文的信号

更适合直接进入正文的情况通常包括：

- 问题边界已经清楚
- 证据相对充分
- 内容可以围绕一个主题、方法、对象或机制系统展开
- 它主要是在表达已经整理后的知识，而不是在管理未稳定材料

### 5.2 适合元信息文件的信号

更适合先进入元信息文件的情况通常包括：

- 你首先面对的是内容缺口，而不是稳定结论
- 你首先面对的是候选研究对象队列，而不是主题主文
- 你首先面对的是结构边界、板块切分与下钻路径
- 你首先面对的是冲突、待核验或顺序分流问题

这时不要为了“写出点东西”就抢写正文。先把未稳定问题承接好，反而更稳。

## 6. 元信息文件的最小判断顺序

如果当前更适合写元信息文件，可先按下面顺序粗分：

- **目录说明问题**：`README.md`
- **结构导航问题**：`index.md`
- **内容缺口问题**：`backlog.md`
- **对象队列问题**：`candidates.md`
- **结构边界问题**：`landscape.md`
- **顺序路径问题**：`roadmap.md`
- **冲突校验问题**：`conflict.md`
- **整体理解已成熟**：`overview.md`

这不是固定顺序，而是最小识别表。真正决定先出现哪一类文件，取决于先稳定下来的是什么问题。

## 7. 几种最常见的 Place 误判

### 7.1 误判一：找到目录就等于知道该写什么文件

不是。

找到目录只是完成了 Navigate，距离真正落位还差两步：

- 是否继续下沉
- 写正文还是写元信息文件

### 7.2 误判二：一旦拿不准，就先补正文占位

这通常是最不稳的做法。

如果当前最突出的其实是缺口、对象、边界或冲突，优先写元信息文件，通常比抢写正文更稳。

### 7.3 误判三：元信息文件应该按固定顺序出现

不是。

在真实研究流程里，可能先出现的是：

- `backlog.md`
- `candidates.md`
- `landscape.md`
- 或最终才轮到 `overview.md`

关键不是顺序，而是**哪个问题先稳定出现**。

### 7.4 误判四：只要能下沉，就应该尽量下沉到最细层

不是。

如果更细一层的归属仍不稳，强行下沉只会把不稳定判断提前固化进目录结构里。

## 8. 一个最小决策骨架

拿不准时，可以先按下面这个骨架快速判断：

```text
已找到大致目标区域
  → 这一层是不是最高稳定归属？
    → 不是：先回到更高层
    → 是：继续
  → 更细一层的归属是否也稳定？
    → 是：继续下沉
    → 否：停在当前层
  → 当前最先稳定的是哪类问题？
    → 已整理知识：正文
    → 缺口：backlog
    → 对象：candidates
    → 结构：landscape
    → 冲突：conflict
    → 顺序：roadmap
    → 目录说明 / 导航：README / index
```

## 10. 本文的边界

本文不负责：

- 代替具体规则文件给出所有 stop-line
- 定义每类元信息文件的全部细节
- 代替 Navigate 去解释如何搜索整棵仓库树
- 对具体材料直接做最终落位裁决

它只负责把一个经常被混写的能力单独说清楚：**找到位置以后，如何做落位决策。**

# Repository Search and Navigation

本文是仓库的公共搜索 / 导航指引，回答的是：**当你只有一个问题、还不知道具体目录名时，如何借助 `README.md`、`index.md` 与候选范围内的定向搜索，找到最可能的入口。**

它面向普通读者、人类维护者与 AI 协作者，不负责决定该写什么文件，也不替代 `docs/contributing/` 下的维护规则。

## 1. Navigate 解决什么问题

`Navigate` 解决的是“从问题找到位置”，不是“找到位置以后写什么”。它的目标是帮助读者、人类维护者或 AI 协作者尽快缩小到最可能的入口目录、目标目录或候选支路。

它默认不把全文搜索和目录导航对立起来。更稳的策略是：**先用目录树缩范围，再在候选范围内做定向搜索，最后回到 `README.md` / `index.md` 判断主入口与旁证。**

一个最小顺序可以先这样理解：

```text
问题
  → 找最高稳定上层目录的 README
  → 用 index 缩小到 1–2 条最可能支路
  → 只在候选支路内做定向搜索
  → 回到 README / index 判断哪篇是主入口、哪些只是旁证
```

这种做法的目的不是把树结构改成搜索引擎，而是同时保留：

- `README.md` / `index.md` 提供的结构感
- 定向搜索带来的速度与跨支路补证据能力

## 2. README / index 分别承担什么

在这个仓库里，搜索不是先记住所有目录名，而是先利用两类入口接口：

- `README.md`：语义定向接口。先判断“这是不是我要找的地方”。
- `index.md`：结构路由接口。进入目录后，再判断“下面有哪些子方向、我该往哪层继续走”。

也就是说：

- `README.md` 回答“是不是这里”
- `index.md` 回答“既然是这里，那下一步去哪”

`index.md` 主要解决的是：

- 这个目录下面有哪些子方向
- 我已经知道想找什么时，该直接去哪一支
- 哪些入口属于结构导航，哪些属于后续分流

## 3. 不知道目录名时，先按问题类型分流

如果你还不知道该去哪个主题目录，先判断你的问题更像哪一类：

- **某个具体主题、模型、方法、框架或研究方向**：优先进入对应主题目录，例如 `llm/`、`agentic/`、`deep-learning/`、`training-infra/`
- **跨主题、跨学科或横向映射问题**：优先进入 `interdisciplinarity/`
- **仓库规则、文档治理、Evidence、README/index 分工**：优先进入 `docs/`，再看 `docs/contributing/`
- **学习资源、书籍、课程、论文导航**：优先进入 `learning-materials/`

如果一个问题同时像多个方向，不要一开始就下钻到很深层；先从更高层的 `README.md` 读起。

## 4. 默认路径：先缩范围，再下钻

进入某个目录后，先读 `README.md`，重点不是找所有细节，而是先回答三个问题：

1. 这个目录主要承接什么问题？
2. 哪些内容适合放这里，哪些不适合？
3. 如果问题确实属于这里，下一步该继续看结构导航还是其他入口？

如果 README 读完以后，发现问题其实不属于这个目录，就应立即回退到更高层继续换入口，而不是硬往下钻。

只有当你已经基本确认“这个目录就是我要找的地方”时，才进入 `index.md` 继续缩小范围。

所以默认导航路径是：

```text
问题
  → 找最可能的上层目录 README
  → 判断语义归属
  → 必要时进入 index
  → 继续下钻到更细目录或候选支路
```

## 5. 定向搜索如何配合目录导航

在这个仓库里，常见的搜索方式至少有两种：

- **目录导航式**：先读上层 `README.md` 判断语义归属，再进 `index.md` 继续下钻。
- **定向搜索式**：当你已经有比较具体的问题、关键词或对象名时，在最可能的区域内再做定向搜索，用来快速命中文档。

两者各有优缺点：

- 目录导航式更稳，能先恢复目录边界与主入口判断，但通常更慢。
- 定向搜索式更快，适合具体问题或跨支路问题，但更容易先命中“提到这个词”的文件，而不是最该承接它的主入口。

因此，定向搜索更适合放在目录导航之后使用：先借助 `README.md` / `index.md` 缩小到候选支路，再在候选支路内搜索关键词、对象名或具体问题，最后回到入口文件判断主入口与旁证关系。

一个简短例子：如果问题是“agent 系统里的 tool 是否会直接调用大模型”，先从 `agentic/README.md` 判断主领域，再用 `agentic/index.md` 缩小到 `02-single-agent/` 与 `06-frameworks-and-tools/` 两条候选支路。随后只在这些候选支路内定向搜索 `tool executor`、`MCP`、`orchestration` 等关键词，最后再回到对应 README / index 判断哪些文档是主入口，哪些只是旁证。

## 6. 搜索时默认先停在最高稳定归属

如果你一开始还拿不准，不要急着深钻。

更稳的做法是：

- 先停在你能确定的最高稳定目录
- 先读那一层的 `README.md`
- 只有当目录边界已经相对明确时，再进入 `index.md`、继续下钻或做候选范围内的定向搜索

例如：

- 拿不准是训练方法还是训练基础设施，先去 `llm/README.md` 或 `training-infra/README.md`，不要直接跳到深层子目录
- 拿不准是 Agent foundations 还是具体框架实现，先去 `agentic/README.md`，再决定是否进入更细目录

## 7. 几种常见进入场景

### 7.1 从主题名出发

如果你已经知道主题名，例如：

- `Transformer`
- `RAG`
- `DeepSpeed`
- `Mechanistic Interpretability`

先去最可能的上层主题目录，再用该目录的 `README.md` / `index.md` 缩小范围。

### 7.2 从问题出发

如果你只有问题，例如：

- “训练方法与训练基础设施分别在哪个目录？”
- “Agent foundations 与 coding agents 分别在哪个目录？”
- “某个跨学科问题和哪类 AI 主题最接近？”

先找最可能的上层目录 README，看边界说明与相邻目录关系；如果已经缩小到候选支路，再做定向搜索。

### 7.3 从任务出发

如果你的目标不是找知识，而是做事：

- 想改规则：去 `docs/contributing/`
- 想理解系统设计：读 `docs/design/system-design.md`
- 想找学习资料：去 `learning-materials/`

## 8. 仓库是树，知识可能更像图

这套仓库首先是树结构，而不是图结构。它的好处是稳定、直观、维护成本可控；代价是很多知识关系天然会跨目录。

因此搜索时要接受两件事：

- 你通常先找到**最可能的主入口**，而不是一次命中唯一正确答案
- 横向映射、相邻目录边界和少量交叉引用，是这棵树用来近似图结构的方式

所以当一个问题同时触及多个方向时，先找到“主入口目录”比试图一次找到“唯一正确的最深目录”更重要。

## 9. 不同角色定位后的下一步不同

`Navigate` 是公共底座，但不同角色使用它时，定位目标并不完全相同。

- **普通读者**：通常想找到可阅读的知识入口；定位后更常进入 `Answer`，判断当前内容能回答什么。
- **文档贡献者**：不仅要找到相关区域，还要判断内容应停在哪一层、写成正文还是元信息文件；定位后更常进入 `Place` 或 `Ingest`。
- **结构维护者**：关注目录树、README、index 和元信息文件分工是否仍然稳定；定位后更常进入 `Structure` 或 `Maintain`。
- **AI 协作者**：除了定位，还必须遵守 `AGENTS.md`、`CONTRIBUTING.md` 与 `docs/contributing/` 下的规则路由，尤其不能把未验证材料写成稳定结论。

因此，Navigate 只回答“先去哪里、如何缩小范围”。找到位置以后，应根据当前角色和任务继续接到对应能力或规则层，而不是把所有后续判断都塞回搜索过程里。

## 10. 什么时候该停止搜索

搜索的目标不是把整棵树都读一遍，而是尽快缩小到：

- 你已经找到了最可能的目标目录
- 你已经知道下一层该去哪
- 你已经得到足够的候选文档，可以回到 README / index 判断主入口与旁证
- 或者你已经确认当前目录不对，应回退上层重新选入口

当这些条件满足其一，就不必继续横向乱跳。

## 11. 本文的边界

本文不负责：

- 决定某条新材料最终该写成正文还是哪类元信息文件
- 解释 README、index、overview、landscape 的全部规则边界
- 代替 `docs/contributing/` 下的维护流程与文档治理规则

如果你已经找到目标区域，下一步需要决定“写什么文件”，应回到维护规则层：

- 文档处理流程：`docs/contributing/documentation-workflow.md`
- 元信息文件模型：`docs/contributing/metadata-files.md`
- README 规则：`docs/contributing/readme-rules.md`

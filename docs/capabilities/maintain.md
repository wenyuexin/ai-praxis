# Maintain

本文回答：**当仓库已经积累出规则、案例、入口文件与知识结构之后，如何长期维护这套系统，使它在样本漂移、结构演化和规则迭代中不逐步失真。**

它面向普通读者、人类维护者与 AI 协作者，关注的是一种公共能力：**系统进入长期运行阶段后，如何持续校准而不把维护动作混成一次次局部补丁。**

它不替代 `docs/contributing/meta-rules.md` 的规则维护约束，也不替代 `docs/contributing/structure-refactoring-rules.md` 的低风险重构执行细则。本文只把 Maintain 这项能力单独显式化，避免它长期分散在规则、案例和设计讨论里。

## 1. Maintain 不是日常写作，也不是单次重构

在这个仓库里，`Maintain` 很容易被误当成两类更具体的事情：

- 日常写一篇文档、补一条 backlog、修一处 README
- 某次具体的目录改名、迁移或链接修复

但这两者都还不是 Maintain 本身。

`Maintain` 关注的是更上层的问题：

- 这套系统有没有开始漂移
- 哪些文档还在代表原先要说明的问题模式，哪些已经不再代表
- 哪些规则仍然稳，哪些已经过时、冲突或路由不足
- 结构演化时，怎样把长期成本高的问题提前处理，而不是等它们在多个地方同时爆开

所以，Maintain 的起点不是“我要改哪一篇文档”，而是“这套系统接下来怎么继续保持长期可用”。

## 2. Maintain 的四类核心对象

长期维护时，最常需要持续检查的是四类对象：

1. **入口文件**：`AGENTS.md`、`CONTRIBUTING.md`、`docs/README.md`、`docs/contributing/README.md`、`docs/capabilities/README.md`
2. **规则文件**：`meta-rules.md`、`metadata-files.md`、`readme-rules.md`、`documentation-workflow.md` 等
3. **案例文件**：`docs/contributing/cases/` 下的样本与误判复盘
4. **知识结构本身**：主题目录、README / index / 元信息文件与相邻目录边界

Maintain 的难点，不是单个对象怎么改，而是它们之间会相互拖动：

- 规则变了，案例可能过时
- 结构变了，README / index 可能漂移
- 能力层长出来后，入口文件的路由方式也需要跟着更新

## 3. 先判断问题属于哪一类维护

长期维护时，最稳妥的第一步不是直接开改，而是先判断当前问题属于哪类维护：

- **规则维护**：规则缺失、冲突、过时、太长、职责不清
- **结构维护**：目录名、分类轴、导航接口、边界分工开始老化
- **案例维护**：样本目录变化后，案例不再能代表原先问题模式
- **入口维护**：入口文件已无法把人类或 AI 稳定路由到正确的层

只有先分清类型，才能决定下一步应该去：

- `meta-rules.md`
- `structure-refactoring-rules.md`
- `cases/README.md`
- 或某个具体入口文件

## 4. 三个最常见的维护触发信号

### 4.1 同类误判反复出现

如果同类错误被反复指出，通常说明当前问题已经不是一次性偏差，而是系统层的维护信号。

常见例子：

- AI 总把 `README.md` 和 `index.md` 混写
- 总把 `overview.md`、`landscape.md` 当成固定顺序槽位
- 总在入口层停不住，找不到该读哪篇规则

这类问题不应只靠一次口头提醒解决，而应回到规则或入口层修正。

### 4.2 样本已经变了，但案例还在讲旧问题

案例目录最容易出现这种维护需求。

一个案例仍然值得保留，不取决于样本目录是否一字未动，而取决于：

- 它是否还在代表原先想说明的问题模式
- 核心判断是否仍可迁移
- 读者是否还会从它那里得到正确的边界感

如果答案是否定的，就不该继续把旧案例当默认依据。

### 4.3 现在不改，未来迁移成本会明显升高

这是最典型的长期主义维护信号。

例如：

- 目录名正在系统性误导落位
- 入口文件已经无法稳定路由到新的能力层或规则层
- 某条规则明明已经不够，但大家还在靠记忆补

这种问题的危险不在于今天看起来还能凑合，而在于它会把后续维护成本成倍放大。

## 5. Maintain 的最小判断顺序

拿不准时，可先按下面顺序判断：

```text
发现系统层问题
  → 这是规则问题、结构问题、案例问题，还是入口问题？
  → 这是一次性偏差，还是复发模式？
  → 现在不改，后续成本会不会明显升高？
  → 最小修复是：改一句、补路由、更新案例、还是做结构调整？
  → 修复后，哪些相邻层也需要一起复核？
```

这里最后一步很重要。长期维护几乎总是联动的：

- 改规则后，要看入口和案例
- 改结构后，要看 README、index、链接和案例
- 改能力层后，要看 `docs/README.md` 和 `docs/design/system-design.md`

## 6. 四种最常见的维护样式

长期维护并不总是同一种动作。更实用的做法，是先识别当前更像哪一种维护样式。

### 6.1 路由补强

适用场景：规则本身大致是对的，但入口文件已经不能把读者或 AI 稳定带到正确位置。

常见信号：

- `AGENTS.md`、`CONTRIBUTING.md`、`docs/README.md`、`docs/contributing/README.md` 之间的路由层级开始含混
- 新增了 capability layer，但入口文件还停留在旧结构
- 同类任务总是从错误入口起步

这类维护通常不是去重写一整套规则，而是：

- 补一句更直接的入口提示
- 收紧任务路由文案
- 删除已经过时的旧路径或旧说法

### 6.2 案例校准

适用场景：规则本身仍然成立，但旧案例已经不再代表原先要说明的问题模式。

常见信号：

- 样本目录的结构或职责已经明显变化
- 案例还能引用旧路径，但已经不能稳定传达原先判断
- 读者开始从旧案例里学到错误边界感

这类维护通常不是优先改主规则，而是先判断：

- 应更新案例
- 应改写案例中的核心判断
- 还是应直接退役该案例

### 6.3 规则收紧

适用场景：同类误判反复出现，说明现有规则已经拦不住问题。

常见信号：

- 人类已经提醒过，但 AI 仍反复在同一处犯错
- 规则之间职责边界不清，导致反复走错文件
- 某条规则需要靠维护者口头补充才能真正执行

这类维护通常才需要回到 `meta-rules.md` 或对应专题规则，判断是否要：

- 补 stop-line
- 补最小顺序提示
- 删掉歧义表述
- 重写职责边界

### 6.4 结构前置修复

适用场景：今天看似还能工作，但如果继续拖，未来会形成更高迁移成本。

常见信号：

- 目录名或入口方式正在系统性误导落位
- 某层 README 持续吞并 `index.md` 或元信息文件职责
- 新能力层已经形成，但旧入口和旧说明还在把系统往回拉

这类维护最需要长期主义判断：不要因为今天还能手工补，就继续把系统问题往后拖。

## 7. 三个对象级维护例子

### 7.1 入口文件例子：能力层长出来以后，入口要跟着改

如果 `docs/capabilities/` 已经形成稳定层，但 `docs/README.md` 或 `docs/contributing/README.md` 还在按旧结构路由，维护重点通常不是先改规则正文，而是先修入口。

这里的核心判断是：

- 系统结构已经变了
- 但入口还在把人带向旧路径
- 如果不先修入口，后续所有新文档都会继续被旧路由拖偏

### 7.2 案例文件例子：样本还在，案例却已经失真

一个真实目录可能仍然存在，但它后续长出了更多文件、职责也发生了变化。此时案例是否继续成立，关键不在于“路径还在不在”，而在于：

- 它最初想说明的问题模式还在不在
- 当时的核心判断是否仍可迁移
- 今天的读者会不会被旧案例误导

如果答案是否定的，就应更新、改写或退役，而不是继续把它当默认样本。

### 7.3 规则文件例子：短规则已经不够，就该承认它不够

当某条短规则总要靠维护者口头解释，或者 AI 反复在同一个边界上忽略它时，维护重点就不再是“这次帮它补一下”，而是要承认：规则本身可能已经不够直接、不够显眼或不够稳。

这时真正的维护动作通常是：

- 回到 `meta-rules.md` 判断这是不是复发问题
- 再决定是补一句、更换位置，还是重组规则层分工

## 8. 一个最小维护分流骨架

如果你已经确认“这不是一次性偏差，而是系统层维护问题”，下一步最重要的不是立刻开改，而是先把问题送到正确层。

可先按下面顺序分流：

```text
识别到维护问题
  → 主要问题是在“怎么识别与分型”吗？
    → 是：留在 capability layer（如 `maintain.md`）
  → 主要问题是在“具体规则怎么执行、怎么收紧”吗？
    → 是：进入 `docs/contributing/` 主规则或 `meta-rules.md`
  → 主要问题是在“为什么这条规则这样定、最容易被怎样误读”吗？
    → 是：进入 `docs/contributing/intent/`
  → 主要问题是在“复杂对象场景里具体长什么样”吗？
    → 是：进入 `docs/contributing/cases/`
```

这个骨架的关键，不是把四层绝对割裂，而是先防止最常见的错位：

- 把 capability 写成第二份 rule
- 把 intent 写成第二份 capability
- 把 case 写成第二份通用规则
- 把一次对象级误判过早上升成仓库级制度

### 8.1 四层各自该承接什么

更实用的最小判断是：

- `docs/capabilities/`：承接“系统如何识别这类问题、如何给动作分型、如何决定去哪一层修”
- `docs/contributing/` 主规则：承接“真正动手时必须遵守什么 stop-line、顺序和边界”
- `docs/contributing/intent/`：承接“为什么要这样定、最容易被误读成什么”
- `docs/contributing/cases/`：承接“真实对象里这类问题具体怎样出现、怎样复盘”

### 8.2 一个反向判断

如果你正准备补的内容更像下面这些句子，它大概率就不该继续留在 `maintain.md`：

- “改规则前必须先……”
- “README 不应……”
- “当目录下已有两个及以上稳定子目录时……”
- “这个真实目录当时为什么先长出……”

因为这些句子通常已经分别属于：

- 主规则 / `meta-rules.md`
- README 或元信息规则
- 具体案例文件

`maintain.md` 更应该保留的是这种句子：

- “先判断这是入口问题、案例问题、规则问题还是结构问题”
- “先决定该去哪一层修，而不是直接改最大那层”
- “先识别这是复发模式还是一次性偏差”

## 9. 四个最常见的误判

### 9.1 误判一：只要内容还能工作，就不算维护问题

不是。

很多高成本问题在早期都还能“勉强工作”，但已经开始积累：

- 路由越来越靠历史记忆
- 样本越来越不像案例原意
- 规则越来越依赖维护者口头补充

这些都属于已经发生的维护问题。

### 9.2 误判二：维护就是把旧东西全都同步成当前状态

不是。

长期维护的目标不是机械同步，而是保留真正还在起作用的判断，更新已经失效的那部分。

比如案例文件里最重要的通常不是样本目录名，而是：

- 什么问题先稳定出现
- 为什么某类文件先出现或暂不出现
- 哪些职责不应继续混写

### 9.3 误判三：维护问题一定要回到规则层解决

也不是。

有些问题应改规则；有些问题只需：

- 更新入口文件路由
- 退役旧案例
- 收紧某个目录 README 的边界说明
- 在能力层补一篇更清楚的说明文档

如果所有维护都往规则层塞，规则层会越来越重，反而更难用。

### 9.4 误判四：只要是长期主义，就应该优先大改规则或结构

也不是。

长期主义强调的是尽早识别高成本问题，不是把所有问题都升级成大改。很多时候，真正稳的维护动作反而是：

- 先补入口路由
- 先更新失真的案例
- 先删掉一句过时说法
- 先把复发问题暴露出来，再决定是否升级到规则层

如果把“长期主义”误读成“每次都该大改”，维护本身也会变成新的噪音来源。

## 10. Maintain 与其他文档的分工

`Maintain` 这项能力最容易与下面三个层混淆：

### 10.1 与 `meta-rules.md` 的区别

- `meta-rules.md`：回答“什么时候改规则、规则应怎么改”
- `maintain.md`：回答“整个系统长期运行时，如何识别并处理规则、结构、案例、入口层的老化与漂移”

前者更像规则维护约束，后者更像系统维护能力。

### 10.2 与 `structure-refactoring-rules.md` 的区别

- `structure-refactoring-rules.md`：回答“已经决定重构时，怎样低风险执行”
- `maintain.md`：回答“何时该把结构问题识别为维护问题，以及为什么这类问题不能继续拖”

前者偏执行，后者偏识别与调度。

### 10.3 与 `cases/README.md` 的区别

- `cases/README.md`：回答“复杂场景里有哪些可复用样本”
- `maintain.md`：回答“这些样本何时该更新、改写或退役”

一个提供样本，一个提供样本生命周期判断。

## 11. 当前最值得优先维护的对象

按当前仓库状态，长期维护里最值得优先持续观察的对象通常包括：

- 入口文件是否还能稳定路由到 `capabilities/` 与 `contributing/`
- `cases/` 中的样本是否仍在代表原先的问题模式
- capability layer 增长后，`docs/design/system-design.md` 与 `docs/README.md` 是否继续同步
- 规则层是否又开始把应该进入能力层的公共说明重新吞回去

这些问题看起来分散，但本质上都属于同一件事：**系统层是否还在按原先设计稳定工作。**

## 12. Maintain 与“什么时候不该大动”

长期维护强调长期主义，但这不等于每次发现问题都应该立刻大改。

更稳的判断是：

- 如果只是一次性偏差，先不要上升为系统维护动作
- 如果补一句入口说明就能解决，不要直接重组整层文档
- 如果主规则还稳，只是样本失真，优先改案例而不是改规则
- 如果问题已经会反复复发，再考虑是否需要进入规则层或结构层修复

也就是说，Maintain 的关键不是“总是改大”，而是**知道什么时候该前置修，什么时候只做最小修复，什么时候暂时不动。**

## 13. 使用边界

如果你当前要做的是：

- 判断什么时候该改规则：优先读 [`../contributing/meta-rules.md`](../contributing/meta-rules.md)
- 已经决定做结构重构，想看执行顺序：优先读 [`../contributing/structure-refactoring-rules.md`](../contributing/structure-refactoring-rules.md)
- 想复核某个案例是否仍然成立：读 [`../contributing/cases/README.md`](../contributing/cases/README.md)
- 想理解长期维护为什么不能只靠局部修补：从本文开始

## 14. 本文的边界

本文不负责：

- 代替 `meta-rules.md` 给出改规则的全部硬约束
- 代替 `structure-refactoring-rules.md` 给出结构重构的详细操作步骤
- 代替 `cases/README.md` 列出所有代表案例
- 对某个具体问题直接给出最终修复方案

它只负责把一个长期容易被隐性化的能力单独说清楚：**这套系统如何在持续演化中保持可用，而不是慢慢靠记忆维持。**

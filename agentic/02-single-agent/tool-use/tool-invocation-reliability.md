# Tool Invocation Reliability

> 适用范围：单智能体中工具调用的稳定性、失败模式、校验机制与恢复路径
> 阶段状态：研究主干，持续补充工具系统案例与实证证据
> 使用说明：本文件回答“为什么 tool use 的关键难点不只是选工具、工具调用可靠性到底由哪些环节决定、常见失败模式有哪些、系统通常如何提升可靠性”，不直接把某一种调用策略写成标准答案

## 一、定位

在 single-agent 里，tool use 很容易被表述成一个看上去很简单的问题：**选对工具，然后把调用发出去**。

但真实系统里的难点通常不在“有没有工具能力”，而在：

- 参数是否构造正确
- 前置条件是否满足
- 返回结果是否被正确解释
- 工具失败后系统是否知道怎么恢复
- 多轮工具调用中间状态是否被稳定传递

因此，本主题讨论的不是“工具可不可以接”，而是：**工具调用如何在真实任务中保持稳定、可校验、可恢复**。

---

## 二、为什么它必须单列成主题

### 2.1 tool selection 不能代表 tool reliability

即使系统总能大致选到合适的工具，也仍然可能在后续环节失败：

- 调错参数
- 忽略约束
- 误读返回结果
- 在失败后继续沿错误状态执行

所以“选哪个工具”与“工具调用是否可靠”并不是同一个问题。

### 2.2 orchestration 已触及它，但还没有专门从可靠性视角展开

`tool-orchestration.md` 已经讨论：

- 调用顺序
- 工具依赖
- 中间状态传递
- 失败恢复路径

但这些内容更像在讨论“如何组织工作流”。

本文件则进一步聚焦：**即便工作流结构看上去合理，为什么工具调用仍然会失真，以及系统如何压低这种失真率**。

### 2.3 它是 tool-centric agent 的稳定性基础

很多更偏 tool-centric 的 single-agent 系统之所以成立，不只是因为它们接了更多外部能力，而是因为：

- 工具调用结果能被验证
- 失败可以被局部定位
- 系统不会因为一次工具误用就整条链失控

这使得 tool invocation reliability 成为单智能体工程化里非常关键的一层。

## 三、工具调用到底可靠在什么地方

### 3.1 工具选择正确

第一层当然还是：

- 当前子任务是否应该调用工具
- 应该调用哪个工具
- 是否存在比当前工具更合适的替代路径

但这只是可靠性的起点，而不是终点。

### 3.2 参数与约束表达正确

很多失败不是因为工具错了，而是因为：

- 参数缺失
- 参数格式错误
- 参数取值越界
- 忽略权限、路径、上下文或 schema 约束

因此 tool invocation 可靠性很大一部分其实发生在 invocation formation 阶段。

### 3.3 前置条件判断正确

有些工具只有在特定条件下才该被调用，例如：

- 文件必须已存在
- 上一步结果必须完成解析
- 用户授权必须已满足
- 当前环境必须支持该执行能力

如果这些前置条件没有被检查，系统就会把“可调用”误当成“适合现在调用”。

### 3.4 返回结果解释正确

工具返回成功，不等于任务真的成功。

系统还需要正确判断：

- 返回值是否完整
- 返回结果是否满足原目标
- 返回的错误是硬失败还是可重试失败
- 工具输出是否需要进一步验证

这说明可靠性不是只在调用前决定，也在调用后决定。

### 3.5 失败后恢复正确

真正的可靠系统还必须知道：

- 是否值得重试
- 是否应该换工具
- 是否应该回到上一步状态
- 是否已经需要终止并交还给用户

没有 recovery path 的系统，很难把工具调用从“偶尔成功”提升为“稳定工作流能力”。

## 四、常见失败模式

### 4.1 Wrong Tool, Right Intention

系统知道自己需要外部能力，但选错了具体工具。

例如：

- 本应先读文件却直接修改文件
- 本应先搜索候选对象却直接调用执行工具
- 本应做结构化提取却调用了更泛化的生成接口

### 4.2 Right Tool, Wrong Arguments

这是非常常见的一类失败：

- 调对了工具，但参数拼错
- path、query、schema、filters、payload 不正确
- 没把上一步结果正确映射到当前参数

这类错误通常说明问题不在工具能力，而在中间表示与参数生成。

### 4.3 Missing Preconditions

系统在不满足调用条件时仍然发起工具调用，例如：

- 文件还没读取就开始 patch
- 搜索结果还没筛选就直接读取目标
- 当前 workspace 状态还没稳定就执行下一步

### 4.4 Misread Tool Output

系统把工具结果理解错了，例如：

- 把 warning 当 success
- 把 partial result 当 final result
- 把结构化输出的字段语义理解错
- 忽略返回码、stderr 或状态标记

### 4.5 State Drift Across Calls

多工具链任务里，一个常见问题是：

- 上一步结果没有被稳定传递
- 系统忘记哪些尝试已失败
- 当前调用基于过时状态继续推进

这类问题会让每一步局部都看似合理，但全局任务逐渐失控。

## 五、为什么它在单智能体里尤其难

### 5.1 工具接口只是表面，真正难的是状态映射

很多工具本身接口并不复杂，真正困难的是把：

- 用户目标
- 当前子任务
- 上一步观察
- 中间 artifact
- 当前环境约束

正确映射成一次有效 invocation。

### 5.2 外部反馈比纯语言生成更“硬”

在纯生成任务里，错误常常只是答案质量下降。

但在工具调用里，错误更容易直接表现为：

- 调用失败
- 副作用错误
- 状态污染
- 错误结果被继续消费

所以 tool invocation reliability 与一般生成质量不是同一量级的问题。

### 5.3 多工具系统会把局部错误放大成链式错误

一旦任务是多步工具链：

- 一次参数错误
- 一次错误读取
- 一次状态误判

都可能在后续调用中被不断放大。

这也是为什么可靠性不能只在单次调用层面看，而要放进整个 agent loop 中看。

## 六、系统通常如何提升工具调用可靠性

### 6.1 显式工具约束与 schema

系统常通过更明确的方式约束调用：

- 参数 schema
- 类型限制
- 必填字段
- 调用前说明
- 权限与路径边界

这能减少一部分格式和约束层面的错误。

### 6.2 调用前检查

在真正调用前做 preflight checking，例如：

- 路径是否存在
- 输入是否齐全
- 当前上下文是否满足前置条件
- 是否需要先执行另一工具

这类机制本质上是在把失败前移。

### 6.3 调用后校验

更可靠的系统不会把工具返回直接当终点，而会继续检查：

- 返回结构是否完整
- 结果是否满足任务要求
- 是否需要二次验证
- 是否与其他反馈源冲突

### 6.4 失败分类与分层恢复

不是所有失败都应该同样处理。

系统通常需要区分：

- 可重试失败
- 参数错误
- 权限/环境错误
- 结果不满足要求但调用成功
- 需要升级到 replanning 或人工接管的失败

### 6.5 显式状态记录

为了避免长链漂移，系统往往需要记录：

- 当前子任务
- 已尝试的工具
- 最近失败原因
- 已验证结果
- 可回退的 artifact 节点

这使得 tool invocation reliability 与 memory、workspace state、traceability 深度耦合。

## 七、哪些场景里它更关键

### 7.1 Coding Agent

代码读取、修改、运行、测试链条中的每一步都可能产生硬失败，因此 reliability 几乎是基本盘。

### 7.2 Browser / Research Agent

页面状态变化、检索结果不稳定、结构化提取误差都会让调用可靠性成为核心问题。

### 7.3 高风险执行任务

只要工具调用带来显著副作用，例如：

- 文件修改
- 数据写入
- 远程操作
- 自动执行命令

系统就不能只追求“能跑通”，而必须追求可校验与可恢复。

---

## 八、最重要的结构性矛盾

### 8.1 Flexible Invocation vs Reliable Invocation

越灵活的调用策略越适合未知任务，但也越难约束、校验和复盘。

### 8.2 Lightweight Tool Use vs Validation Overhead

调用前后检查越充分，可靠性越高；但执行链也会更重、更慢。

### 8.3 Local Success vs End-to-End Correctness

单次调用成功不代表整体任务正确，可靠性必须从端到端视角衡量。

### 8.4 Expressive Tooling vs Control Complexity

工具越多、能力越强，agent 越灵活；但参数空间、失败类型与恢复策略也会同步爆炸。

---

## 九、容易被误写成定论的问题

当前阶段应避免把下列说法直接写成共识：

- 只要工具选对了，调用就基本不会出问题
- tool reliability 主要是模型理解能力问题
- 增加更多工具总会提升系统能力
- 调用成功就等于任务成功
- 只要加一层 orchestration，就自然解决可靠性问题

这些说法都会低估参数构造、状态传递、结果校验与恢复路径在真实系统里的重要性。

## 十、与其他专题的关系

- 与 `tool-use/README.md`：后者定义 tool-use 目录边界；本文件进一步说明工具调用难点不只在协议和接口，而在 invocation 的稳定性与恢复能力。
- 与 `tool-orchestration.md`：后者讨论多工具链如何组织；本文件更聚焦单次与多次调用为什么会失真，以及如何提升调用可靠性。
- 与 `tool-centric-vs-monolithic.md`：tool-centric 路径之所以成立，很大程度上依赖工具调用能被验证、恢复与约束，而不是只靠模型“会用工具”。
- 与 `agent-vs-tool-workflow-boundary.md`：一旦系统形成持续执行闭环，失败工具调用就不再是局部小错误，而会影响整条 loop 的稳定性。
- 与 `long-context-vs-external-memory.md`：多步工具调用中的失败记录、已验证结果与状态传递，常常推动 external memory 或显式状态系统出现。
- 与 `../05-environments/`：权限、执行隔离、workspace state、rollback 与 traceability 都会直接塑造工具调用的可靠性边界。
- 与 `backlog.md`：本文件对应当前 P0 缺口 `Tool Invocation Reliability`，并为后续继续展开 input reformulation、state persistence、tool navigation 等主题提供更具体的落点。

## 十一、当前最值得继续补证的方向

- 哪些 tool invocation failure mode 在 coding agent、browser agent、research agent 中最常见
- 参数生成错误、状态传递错误、结果误读错误三类问题，哪类最常成为真实瓶颈
- preflight checking、post-validation、retry policy、tool substitution 哪些组合最能提升稳定性
- tool reliability 的主要提升来自更强模型，还是来自更好的 workflow / state / validation design
- 是否能抽象出一套更稳的 invocation reliability checklist，用于横向比较不同 agent 系统

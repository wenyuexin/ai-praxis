# Claude Code 最佳实践

> 调研时间：2025-07-03（2026 有效性已复核）
> 标注规则：[官方明确] / [社区实践] / [待验证]
>
> 复核结论：Plan Mode、并行会话、`.claudeignore`、延迟工具发现等原则在 2026 仍有效。

## 1. 高频原则（2025-2026）

1. 不为一次性场景做抽象 [官方明确]
2. 不为低概率问题写过度防御 [官方明确]
3. 每次改动只触达必要范围 [官方明确]
4. 将模糊需求转为可验证目标 [社区实践]
5. 复杂任务先 Plan 后执行 [官方明确]
6. 使用 `.claudeignore` 降噪 [官方明确]

## 2. 推荐做法

| 做法 | 适用场景 | 判定 |
|---|---|---|
| Plan Mode 先规划 | 重构/新功能 | [官方明确] |
| 三文件法（WorkFlow/Review/GitInfo） | 中大型任务 | [官方明确] |
| 并行会话（多 worktree） | 多任务并行 | [官方明确] |
| `/compact` 保留关键架构 | 长会话 | [官方明确] |
| `/resume` 恢复上下文 | 会话中断 | [官方明确] |
| `think hard`/`ultrathink` | 疑难问题 | [官方明确] |

## 3. 不推荐做法

| 做法 | 原因 |
|---|---|
| 顺手重构无关代码 | 边界失控，风险扩大 |
| 把模糊需求直接丢给 AI | 输出不可验证 |
| CLAUDE.md 过长 | 关键约束被稀释 |
| 不做验证直接提交 | 缺乏闭环 |

## 4. 2026 新增/变化（整理）

- MCP 成为核心接入路径 [官方明确]
- Plan Mode 从可选变为强推荐 [官方明确]
- SubAgent 上下文隔离机制更明确 [官方明确]
- 延迟工具发现用于节约上下文预算 [官方明确]

## 5. 落地清单

- 任务开始前：写目标、约束、验收标准
- 修改前：确认影响范围与回滚策略
- 修改后：运行测试/构建/检查命令
- 提交前：保留关键决策与证据链接

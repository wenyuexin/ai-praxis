# Codex Cloud 执行与沙箱

> **专题文档**。本文聚焦 Codex Cloud 执行环境和沙箱机制，适合评估安全、隔离和成本时查阅。不属于主线阅读路径，可与 `codex-agent-mechanisms.md` 的沙箱节搭配阅读。



> 基于 `developers.openai.com/codex` 官方文档 + 社区实测整理（2026.5 最新）。Cloud 是 Codex 企业级落地的核心支撑。

---

## 1. Codex Cloud 执行环境规格

| 维度 | 规格 | 证据等级 |
|------|------|---------|
| **基础镜像** | `openai/codex-universal` — 预装 Python、Node.js 等常见语言和工具 | A-官方 |
| **CPU/内存** | 未明确公布（动态分配），支持在 Environment Settings 中固定版本 | A-官方 |
| **超时** | 任务级超时，长任务支持（7h+ 连续执行） | B-什么值得买 |
| **网络访问** | **Setup 阶段**：有（用于安装依赖）；**Agent 阶段**：默认关闭（可在 Settings 开启） | A-官方 |
| **磁盘** | 独立沙箱环境，任务结束后销毁 | A-官方 |
| **缓存** | 最长 **12 小时**，命中时跳过 Setup，直接进入 Agent 阶段 | A-官方 |

## 2. 审批队列工作流

```
┌─────────────┐    ┌──────────────┐    ┌─────────────┐    ┌──────────────┐    ┌─────────────┐
│  任务提交    │───→│ approval_    │───→│  审批对话框   │───→│  云端执行    │───→│  结果回传   │
│  (CLI/App)  │    │ policy 检查  │    │ (如需要)     │    │  (Sandbox)  │    │  (diff/PR)  │
└─────────────┘    └──────────────┘    └─────────────┘    └──────────────┘    └─────────────┘
                         │
                         ▼
                  ┌──────────────┐
                  │ 4 级策略判断  │
                  │ untrusted → 必确认 │
                  │ on-failure → 失败时确认 │
                  │ on-request → 特定操作确认 │
                  │ never → 全自动 ⚠️ │
                  └──────────────┘
```

| 阶段 | 说明 | 证据等级 |
|------|------|---------|
| **Setup 脚本** | 独立 Bash session，安装依赖、配置环境变量。`export` 不跨 session 传递 | A-官方 |
| **Maintenance 脚本** | 缓存命中时增量更新依赖，比完整重跑快得多 | A-官方 |
| **Secrets 管理** | 额外加密层，**仅 Setup 阶段可用**，Agent 执行前自动移除 | A-官方 |
| **Agent 执行** | 默认网络关闭，需手动开启。所有流量经 HTTP/HTTPS 代理 | A-官方 |
| **结果回传** | 展示 diff → 可开 PR 或继续追问 | A-官方 |

## 3. Cloud vs 本地沙箱对比

| 维度 | Codex Cloud | 本地沙箱（Local/Worktree） | 证据等级 |
|------|------------|---------------------------|---------|
| **隔离层级** | 完全远程隔离，代码不出 OpenAI 云端 | OS 内核级（Landlock+Seccomp+bubblewrap / Seatbelt） | A-官方 |
| **网络策略** | Setup 有网络，Agent 默认关闭，可选开启 | 继承终端网络权限 | A-官方 |
| **数据留存** | 任务结束后环境销毁，无持久化 | 本地持久化 | A-官方 |
| **性能** | 受限于云端调度，冷启动慢 | 本地执行，更快 | B-社区 |
| **成本** | 按 token 计费 | $0（但有机器成本） | A-官方 |
| **适用场景** | 长任务、需要隔离的任务、不信任本地环境 | 快速迭代、敏感代码、低延迟需求 | A-官方 |

> ⚠️ **关键差异**：Cloud 是"代码上云执行"，本地是"代码留在本地执行"。Cloud 的安全优势是**物理隔离**，本地的优势是**数据不出域**。

## 4. 典型成本估算

| 场景 | 成本 | 证据等级 |
|------|------|---------|
| **中等规模业务**（日均 2000 行代码，80-100 次 API 调用，2-3 万 token） | **~$0.2/天**（约 ¥1.4/天） | A-官方 |
| **运维脚本生成**（单次 1000-3000 token，月 200 次） | **<$10/月**（约 ¥70/月） | A-官方 |
| **Express.js 重构** | Codex: ~150 万 token ≈ **$2.50** | B-Composio 实测 |
| **Lint 修复**（100k token） | Codex: ~$0.25 | B-Composio 实测 |
| **GPT-5.5 成本** | 约为 GPT-5.4 的 **2 倍**（智能体基准大幅提升） | A-官方 |

## 5. 与 Worktrees 的配合模式

| 模式 | 说明 | 证据等级 |
|------|------|---------|
| **Local + Worktree** | 本地执行，Worktree 提供隔离环境，多任务并行不冲突 | A-官方 |
| **Cloud + Worktree** | Automations 在专用后台 worktrees 中运行（非 Git 仓库项目直接在项目目录） | A-官方 |
| **Worktree 隔离优势** | "多个任务共享代码库但不互相踩文件"，diff 清晰可审 | A-官方 |
| **缓存配合** | Worktree + Cloud 缓存（12h），频繁提交时第二次几乎无需等待 Setup | A-官方 |

> 🔑 **最佳实践**（来自 OpenAI 官方推荐）：
> 1. 用 **Setup 脚本**而不是 AGENTS.md 安装依赖（AGENTS.md 每次传给 Agent 有 token 开销）
> 2. 用 **Secrets** 而不是环境变量存凭证（额外保护层）
> 3. 善用 **Maintenance 脚本**（缓存命中时增量更新）
> 4. 固定依赖版本用 `--frozen-lockfile` 或 `pip install -r requirements.txt`

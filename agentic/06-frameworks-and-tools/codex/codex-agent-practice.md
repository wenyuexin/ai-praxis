# Codex Agent 使用方法

> 目标：提供可复用的使用方法，确保“可持续提效”，而不是一次性技巧。

## 1. 方法主线

推荐固定流程：

1. 定义任务目标（要产出什么）
2. 限定上下文边界（能用哪些信息）
3. 明确约束（性能/兼容/安全）
4. 规定输出格式（计划、diff、测试）
5. 设置验收标准（通过条件）

## 2. 三类高频任务范式

### A. 需求到实现

- 先让 Agent 给方案对比
- 再选一个最小可行实现
- 最后绑定验收与回归测试

### B. 代码改造

- 先生成变更计划
- 再按模块分批改动
- 每批改动都跑验证

### C. 问题定位

- 输入最小复现信息
- 要求根因排序与验证步骤
- 修复建议必须带测试

## 4. OpenAI 团队实践补充（官方案例）

来自 OpenAI 官方实践页面（How OpenAI uses Codex）的可复用方法：

- 先用“询问模式”拿方案，再切到代码执行，降低一次性大改风险。
- 提示词按“像写 Issue 一样”组织：目标、上下文、约束、验收标准。
- 把重构/迁移拆成模块级任务，优先让 Codex 产出可审阅 PR。
- 对性能优化任务，先让 Codex 识别热点路径，再要求解释优化理由。
- 把测试补齐作为独立任务流：覆盖边界输入、失败路径和异常状态。
- 把“专注恢复”作为工作流目标：用任务队列记录零散想法，支持中断后继续。
- 使用 AGENTS.md 提供持续上下文，降低跨任务提示漂移。
- 对复杂任务启用 Best-of-N 思路，一次生成多方案后择优整合。
- 持续优化启动脚本、环境变量和网络权限，减少环境型失败。

### 4.1 可复用提示词模板

- 代码理解："梳理请求在本服务中，从入口到响应的完整流转流程。"
- 重构迁移："按业务职责拆分当前文件为独立模块，并为各模块生成对应测试。"
- 性能优化："找出此请求处理程序中的重复高开销操作，并给出可使用缓存的优化建议。"
- 测试补齐："为此函数编写单元测试，覆盖边界场景与失败路径。"
- 提升开发速度："为 POST /events 搭建新的 API 路由，并添加基础验证和日志记录。"
- 探索方案："如果系统采用事件驱动而不是请求/响应模式，它会如何运作？"

## 5. 反模式与已知失败模式（避免）

- “一次性重构整个仓库”
- “不给约束直接要最优”
- “聊天结果直接提交”
- “把长上下文当长期记忆”

## 6. 稳定性与成本控制

- 把长任务拆成“分析→实现→验证”
- 对输出长度和轮次设置上限
- 记录每轮关键决策与结果
- 对重复任务沉淀固定模板

## 7. 团队落地最小链路

- PR 前：Agent 预审风险
- CI 中：测试 + SAST 强制门禁
- PR 中：人工 review 语义与边界
- PR 后：规则沉淀到团队指令

## 8. 场景选择建议

参考社区实测和 Benchmark 数据，总结 Codex 最适合和不适合的场景：

| 场景 | 适合度 | 理由 |
|------|--------|------|
| ✅ **快速原型 / 日常编码辅助** | **非常适合** | Token 效率高，成本低，迭代快 |
| ✅ **自动化工作流 / PR 审查** | **非常适合** | 内置 Automations + /review 命令 |
| ✅ **代码理解 / 故障定位上手** | **非常适合** | 可快速梳理模块关系、数据流与故障传播路径 |
| ✅ **预算敏感 / 小团队** | **非常适合** | Token 消耗低，单任务成本 $0.2-0.25 |
| ✅ **异步 / 长任务派发** | **适合** | 云端执行 + 审批队列；适合会议密集与值班场景 |
| ⚠️ **大型代码库重构（>100 文件）** | **需分步进行** | 分步拆解，配合 worktree 并行改造 |
| ⚠️ **极度敏感的生产环境** | **需人工审核** | 配置 approval_policy + 审计日志，关键操作人工确认 |

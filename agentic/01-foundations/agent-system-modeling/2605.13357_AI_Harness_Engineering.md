## 1. 核心论点与定位

### 1.1 系统能力视角转移

论文最本质的论点是：自治软件工程能力属于系统，而不属于模型单体。

作者将系统能力形式化为：

$$ C_{\text{system}} = F(C_{\text{model}}, C_{\text{harness}}, C_{\text{environment}}, T) $$

其中 $C_{\text{model}}$ 是 foundation model 的潜在能力，$C_{\text{harness}}$ 是负责 context、tools、memory、verification 等运行时支撑的 substrate，$C_{\text{environment}}$ 是仓库、测试、构建、日志等环境可供性，$T$ 是任务分布。

这个公式的意义不在于提出一个可优化目标，而在于改变归因：**系统失败未必是模型不会写代码，也可能是 harness 没有暴露必要资源，或者环境没有提供可解释反馈。**

> 原文 `main.tex:153-169`：*"Autonomous software-engineering capability is an emergent property of a model--harness--environment system, not of the model alone."*

### 1.2 Autonomy Gap 与 Missing-Harness Intervention

作者进一步定义了 **autonomy gap**：模型在局部代码生成上表现出的能力，与完整系统在**没有人类替它补运行时支撑**的前提下独立完成软件工程任务的能力之间的差距。这个 gap 不是一个单一失效模式，而是一组典型失败：

- 一个 agent 可能写对了逻辑，但查错了文件
- 选择了相关测试，但误读了测试输出
- 把 patch 应用到了错误的架构层次
- 验证了自己做的修改，但没有检查既有行为是否仍被保持
- 宣称完成但没有记录证据

与此对应的一个关键方法论创新是：**人类介入被重新定义为诊断信号，而不是噪声。** 例如，当人类告诉 agent 该看哪个文件时，这通常指向缺失或不充分的 context manager；当人类为 agent 解释测试失败时，这通常指向缺失的 observability 或 failure attribution 支持。作者将这类介入定义为 **missing-harness human intervention**，并定义了 M-HIR（missing-harness intervention rate）：

$$ \text{M-HIR} = \frac{\text{missing-harness interventions}}{\text{total episodes}} $$

一个能降低 M-HIR 的 harness 就是提供了人类本应提供的运行时支撑。

> 原文 `main.tex:251-281`：全段定义 autonomy gap 和 M-HIR。

### 1.3 Failure Taxonomy

论文进一步提出了 8 类失败归因，使 pass/fail 之外的诊断成为可能：

| 类型 | 含义 |
|------|------|
| $F_{\text{context}}$ | 缺少或误用相关上下文 |
| $F_{\text{tool}}$ | 工具缺失、不稳定或误用 |
| $F_{\text{feedback}}$ | 反馈不可用或不可解释 |
| $F_{\text{verify}}$ | 无法证明任务要求被满足 |
| $F_{\text{recovery}}$ | 无法从失败中恢复 |
| $F_{\text{entropy}}$ | 引入维护负担 |
| $F_{\text{model}}$ | 模型推理或编码失败（尽管 harness 和环境充分） |
| $F_{\text{unknown}}$ | 无法自信归因的失败 |

这套分类法的价值不在于它是否完美，而在于它使得对"失败时缺了什么运行时支撑"的提问成为可能，而不是仅仅问"失败没有"。

> 原文 `main.tex:283-296`：给出 failure taxonomy 的完整定义。

### 1.4 位置定位

作者明确定位了 AI Harness Engineering 与相邻概念的关系。原文的区分（`main.tex:415-437`）如下：

- **非 prompt**：prompt 只影响单次模型调用，harness 治理整个 episode。
- **非 agent framework**：agent framework 提供 agent/tool 组合基础设施，harness 是暴露给 agent 的运行时支撑配置。
- **非 ACI**：ACI（Agent-Computer Interface）指定 agent 如何通过工具行动，是 harness 的一个组件。
- **非 agent OS**：agent OS 面向通用 agent 调度和资源管理，harness 面向软件工程特定 substrate。
- **非 evaluation harness**：evaluation harness 测量行为，development harness 塑造行为。
- **非 DevOps / platform engineering**：它们为人类和机器开发工作流提供基础设施，development harness 则关注 foundation-model agent 与软件开发环境之间的运行时接口。

> Evidence：`Observed`。原文 `main.tex:415-437` 逐条区分了以上概念。作者也指出——harness 可以利用 prompts、agent frameworks、ACIs、DevOps 工具和操作系统服务来构建。研究对象是暴露给 agent 的运行时支撑配置以及在执行期间产生的证据。

## 2. The AI Harness Engineering（论文核心框架）

### 2.1 定义

**AI Harness Engineering 是围绕 foundation-model software agent 的 runtime substrate**，负责管理 context、tools、project memory、task state、observability、failure attribution、verification、permissions 和 maintenance state，使模型的潜在编码能力转化为可审计的软件工程行为。

原文（`main.tex:303-319`）接下来给出了四个推论：

1. **外部于模型**：harness 影响模型行为，但自身不是模型。
2. **任务运行时基础设施**：它决定 agent 如何**观察**项目、如何对其**行动**、如何接收**反馈**、如何建立**完成**判断。
3. **可评估**：其组件可以被暴露、隐藏、消融、追踪和比较。
4. **生产证据**：文件为什么被选中、哪些工具被使用、失败如何归因、哪些要求被验证、人类介入是否必要、引入了哪些维护负担。

> Evidence：`Observed`。这四点推论是原文明确定义的，不是外部推断。

### 2.2 五条设计原则（P1–P5）

原文 `main.tex:321-339` 给出：

- **P1. Expose runtime resources explicitly**：关键资源——上下文、工具可供性、项目记忆、验证证据、人类注意力、权限边界、维护状态——被显式暴露和命名，而不是保持隐式。
- **P2. Traceable mediation**：harness 记录 agent 如何选择上下文、调用工具、尝试验证、从失败恢复以及被介入。
- **P3. Requirement-level verification**：任务完成绑定到证据——确定性检查、针对性测试、回归尝试、lint、补丁审查——而不是自然语言声明。
- **P4. Attribution before recovery**：失败的观察在 agent 再次编辑之前产生分类诊断。
- **P5. Maintenance and entropy awareness**：harness 记录 agent 是否引入了维护负担——陈旧的文档、依赖变动、生成残留、测试弱化、边界违规——而不是将其排除在循环之外。

### 2.3 十一个组件职责

这是论文框架的核心实证部分。原始组件表（`main.tex:350-375`）不仅列出了组件名称和职责，还给出了**每个组件缺失时的典型失败模式**和**产生的证据产物**——这使得组件定义不是抽象的主张，而是可操作的检验标准：

| 组件 | Runtime Contract | 缺失时的失败模式 | 证据产物 |
|------|-----------------|-------------------|----------|
| Task interface | 呈现目标、要求、约束、成功标准 | 目标不明确；在错误目标上工作 | Task record |
| Context manager | 选择和暴露任务相关的项目内容 | 检查错误文件；遗漏约束 | Context trace |
| Tool registry | 声明可用工具和允许命令 | 调用失败；不安全命令；重复超时 | Tool trace |
| Project memory | 提供 agent-readable 的架构、测试、已知失败知识 | 重复重新发现；错误层次的修复 | Memory references |
| Task state | 维护当前假设、已检查文件、开放问题、下一步 | 漂移；重复工作；不连贯 | Task-state file |
| Observability layer | 暴露日志、trace、输出、运行时错误 | 成功的不可验证；失败的不可诊断 | Observation log |
| Failure attribution | 分离观察结果、预期行为和诊断 | 失败后的随机修补 | Attribution log |
| Verification protocol | 将任务要求映射到确定性证据 | 未经验证的成功；虚假自信 | Verification trace |
| Permission boundary | 限制风险动作；暴露审批关卡 | 不安全或无效的 episode | Permission record |
| Entropy auditor | 检测 agent 引入的维护负担 | 陈旧文档；依赖变动；残留 | Entropy audit |
| Intervention logger | 记录人类协助及其可避免性 | 不可见的人类支架 | Intervention log |

> Evidence：`Observed`。`main.tex:350-375` 表 `tab:components`。原始论文中的组件表更进一步显式地把"缺失时的失败"和"证据产物"作为每个组件定义的一部分——这种设计使组件不是抽象描述，而是可操作的检验标准。

### 2.4 资源管理视角

论文还提供了一个辅助性的**资源管理视角**（`main.tex:377-412`）：传统操作系统管理 CPU、内存、文件、进程和设备；Development Harness 管理一组类比但不同的运行时资源：

context budget、tool budget、verification evidence、project memory、task state、human attention、permission boundary、failure signal、entropy budget、test-time compute。

作者特别强调：这个类比只服务于一个目的——**识别出为了让 agent 的行为保持连贯、可验证、可维护，必须管理哪些东西**。作者没有主张传统 OS 机制可以直接迁移到 agent 设置中，也不提出"AI 的操作系统"。类比的价值严格限定在资源管理视角。

## 3. H0-H3 Harness Ladder

### 3.1 设计动机

论文在定义框架之后面临一个实证问题（`main.tex:444-447`）：*"Harness framework 提供了一套词汇，但它本身不允许实证探究。为了使 harness 的贡献与模型的贡献可分离，我们需要一种可控的方式来改变运行时支撑，同时保持任务、仓库和模型固定。"*

### 3.2 四层定义

H0-H3 是一个逐层增加运行时支撑暴露的可控消融实验（`main.tex:462-481`）：

| Level | 名称 | 运行时支撑暴露 | 为什么重要 |
|-------|------|--------------|-----------|
| **H0** | Minimal baseline | 只有任务描述和仓库文件。无 tool registry、project memory、verification protocol。 | 比较基线。代表"无 harness"的典型情形。 |
| **H1** | Tool harness | H0 + tool registry、test-command registry、tool-usage protocol。 | 使动作界面显式化和可追踪，但不提供 agent-readable 项目知识或验证规范。 |
| **H2** | Context--memory harness | H1 + agent-readable project memory（架构、测试约定、已知失败）、task-state file、context-selection protocol。 | 使上下文决策显式化和可追踪。 |
| **H3** | Observability--verification harness | H2 + deterministic behavioral check registry、bug-reproduction protocol、failure-attribution protocol、verification protocol、verification report template。 | 使完成成为证据对象，而不是声明。 |

### 3.3 Visibility Matrix

论文的核心操作定义是一张 visibility matrix（`main.tex:499-531`），明确规定了每种 artifact 在哪个 level 对 agent 可见。这是 ladder 的操作化基础：

| Artifact | H0 | H1 | H2 | H3 |
|----------|:--:|:--:|:--:|:--:|
| Task description | ✅ | ✅ | ✅ | ✅ |
| Repository files | ✅ | ✅ | ✅ | ✅ |
| Tool registry | — | ✅ | ✅ | ✅ |
| Test-command registry | — | ✅ | ✅ | ✅ |
| Tool-usage protocol | — | ✅ | ✅ | ✅ |
| Agent_Guide | — | — | ✅ | ✅ |
| Architecture | — | — | ✅ | ✅ |
| Testing guide | — | — | ✅ | ✅ |
| Task_State | — | — | ✅ | ✅ |
| Known_Failures | — | — | ✅ | ✅ |
| Context-selection protocol | — | — | ✅ | ✅ |
| Deterministic check registry | — | — | — | ✅ |
| Bug-reproduction protocol | — | — | — | ✅ |
| Failure-attribution protocol | — | — | — | ✅ |
| Verification protocol | — | — | — | ✅ |
| Verification report template | — | — | — | ✅ |
| Hidden evaluator notes | — | — | — | — |

### 3.4 五点设计要求

Ladder 满足五点设计要求（`main.tex:483-493`）：

- **R1. Controlled visibility**：每层只暴露本层分配的 artifact；低层看不到高层 artifact。
- **R2. Same task, same repository, same initial state**：所有层从相同的任务和仓库状态开始。
- **R3. Traceable runtime support**：当一个层提供了某项能力，其使用被记录。
- **R4. No hidden evaluator leakage**：预期文件、预期修复和 evaluator notes 在任何层对 agent 不可见。
- **R5. Outcome comparability**：每层在相同的最终 outcome taxonomy 下裁决。

### 3.5 Ladder 测量什么

这是论文的一个关键方法论声明（`main.tex:534-540`）：*"H0-H3 ladder 不把任务成功作为其唯一 outcome。它测量 agent 是否检查了相关上下文、使用了工具、运行了测试、重现了失败、归因了它、验证了每个要求、保持了既有行为、避免了无关修改、引入了熵、以及是否被人类介入。"* 下一节将这些测量形式化为 trace-based evaluation protocol。

## 4. Trace-based Evaluation Protocol

### 4.1 原理

论文评估方法的核心原理（`main.tex:551-560`）很简洁：*"自治软件工程评估不应只测量 patch 是否被产出，而应测量 model-harness-environment 系统是否产出了可审计的证据，证明任务要求被满足。"*

### 4.2 Episode 与 Episode Package

一次 agent 运行被定义为一个 **episode**（`main.tex:562-569`），由 episode identifier、model/agent identity、harness level、repository、initial commit、task specification、visible artifacts、allowed tools、intervention policy、verification procedure 和 final outcome rule 唯一界定。**评估单元是 episode，而不是单一的模型响应。**

每个 episode 产出一个 **episode package**，包含 8 类 trace 加上 patch、verification report、final report 和 final-outcome record。

### 4.3 八类 Trace

| Trace | 捕获的运行时资源 | 对应的失败类型 |
|-------|----------------|---------------|
| Action trace | 全部（agent 操作的序列） | 总体 episode 连贯性 |
| Tool trace | tool budget; test-time compute | $F_{\text{tool}}$ |
| Context trace | context budget; project memory | $F_{\text{context}}$ |
| Verification trace | verification evidence | $F_{\text{verify}}$ |
| Failure-attribution log | failure signal | $F_{\text{verify}}$, $F_{\text{model}}$ |
| Intervention log | human attention | 全部（诊断信号） |
| Entropy audit | entropy budget | $F_{\text{entropy}}$ |
| Outcome record | —（最终分类） | 整体裁决 |

论文还进一步给出了每个 trace 的 JSONL schema（`main.tex:611-633`），其中 action trace 记录了 `read_file`, `edit_file`, `run_tool`, `write_report`, `update_task_state`, `inspect_diff`, `declare_complete` 等对外有意义的操作；verification trace 记录了验证类型（bug reproduction / deterministic behavioral check / registered test / targeted test / full regression / lint / patch review / manual evaluator check）、方法、结果、覆盖的要求和 agent 的解读；entropy audit 记录了 9 类 agent 引入的维护负担（code / documentation / dependency / test / file residue / architecture / workflow / configuration / other）及其 0-3 严重程度。

### 4.4 Outcome Taxonomy

最终 outcome 被分类为五个标签之一（`main.tex:635+`）：

| 标签 | 含义 |
|------|------|
| **Autonomous verified success** | 任务要求被满足，且在没有 missing-harness human intervention 的情况下产出充分证据 |
| **Assisted verified success** | 任务要求被满足，但需要 missing-harness human intervention |
| **Autonomous unverified completion** | 任务完成但缺乏验证证据 |
| **Unresolved episode** | 任务未完成，或存在未解答的 entropy、safety 或 regression 问题 |
| **Invalid episode** | episode 失败于外部因素（工具崩溃、环境故障、上下文溢出） |

> 这五个标签解决了论文反复强调的一个问题：patch 通过了测试，不等于系统是可审计的。

## 5. 验证案例

### 5.1 任务设置

论文使用一个 controlled validation task（`main.tex:700+`）作为说明性案例：

- **仓库**：小型 Node.js login application（controller / validator / authentication service / UI / 测试）
- **Defect**：空密码没有被作为 validation error 拒绝，而是流入 credential matching，被错误报告为 `Invalid credentials.`
- **任务要求**：(1) 空密码返回含 `Password is required.` 的 validation error；(2) 正确凭证仍然成功；(3) 错误但非空凭证返回 `Invalid credentials.`；(4) 增加覆盖 empty-password 的测试；(5) 既有测试继续通过

评估依赖三个 deterministic behavioral checks（empty-password probe / valid-login probe / invalid-non-empty-credentials probe），加上 targeted tests、lint、bounded full-regression attempt。

### 5.2 主要发现

论文的核心"结果"不是某个数值提升，而是：**随着 harness level 提升，episode package 的证据结构系统性增强。** 具体而言：

- **H0**：只产出 final patch 和 evaluator-side deterministic checks。
- **H1**：增加 tool trace、targeted tests、lint、bounded full-regression timeout 记录。动作表面变得显式化和可追踪。
- **H2**：增加 project memory 的 context trace 和 task state。上下文选择和项目记忆的使用可追踪。
- **H3**：形成 reproduction → attribution → fix → verify → report 闭环。failure attribution log、deterministic requirement checks、structured verification report 使 task completion 成为结构化证据对象。

> Evidence：`Observed`。`main.tex:624-715` 给出四种 level 的 episode package 产出对比。论文强调这是"说明性展示"而非大规模统计验证。

## 6. 局限性

### 6.1 作者显式或隐式承认的局限

1. **案例规模小**：controlled task 是"小而可检视"的 illustrative case，不支持人口级性能比较（`main.tex:700-703`）。作者明确声明当前结果更像框架可行性展示。

2. **主要验证证据结构，不是广泛任务成功率**：论文展示的是不同 harness level 的 trace 结构差异，不是跨任务统计优势。作者在 future work 中明确指向"更大规模、多任务、多模型"的研究程序。

3. **长期指标处于 outlook 阶段**：在 Discussion 中提出了 AVSR（autonomous verified success rate）、M-HIR（missing-harness human intervention rate）、VAS（verification autonomy score）、entropy delta 等指标，但它们的多任务和多模型估计被标记为未来工作。

4. **Harness 成本未被充分量化**：论文强调 harness 能减少 missing-harness intervention，但构建和维护 project memory、verification registry、entropy audit 等本身需要工程投入。部署复杂度与维护收益之间的 trade-off 仍待后续研究。

### 6.2 推断性局限

以下局限基于论文整体结构的综合判断（`Inferred`，非原文显式声明）：

1. **case 偏 verification-oriented defect**：当前 case 天然适合展示 deterministic checks、failure attribution 和 verification report 的价值；对更开放的 feature implementation、跨模块重构、长期需求演化任务，H0-H3 ladder 的外推效果还需要更多任务来证明。

2. **框架强，工程实例弱**：概念定义和 protocol 设计非常完整，但实证部分仍然较轻。这使得论文容易被定位为"提出了一套很好的 runtime 语言，但还没有充分证明它能在真实复杂仓库上稳定改进 agent performance"——这是该论文当前阶段的固有特征，不是缺陷。

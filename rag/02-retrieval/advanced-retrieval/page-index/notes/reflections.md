# PageIndex 研究反思

本文件用于记录基于当前 `PageIndex` 客观现状形成的个人观察、直觉、假说与阶段性理解。

它不是正式正文，也不是 `evidence.md` 的替代品；其中较成熟的判断，后续应视情况回流到正文、`evidence.md`、`deep-research-question-tree.md` 或 `backlog.md`。

---

## 1. 当前最强的总体感觉

### 1.1 `PageIndex` 的真正稀缺点，不只是“能给 PDF 建树”

**触发现状**

- 当前源码与文档都表明：`PageIndex` 不把长文档当扁平 chunk 集合处理，而是先尽量恢复一个层级结构，再让后续调用方沿这棵树定位正文。
- `tree-and-navigation.md` 已压实：在本地可见 SDK 中，树主要承担的是结构定位 / 导航作用，而最终回答仍依赖正文页内容。

**当前反思**

我现在更倾向于把 `PageIndex` 看成一种“先恢复结构，再组织检索行为”的对象，而不只是“有树输出的 PDF indexing 工具”。

它的代表性，可能主要不在“树这个数据结构本身”，而在于它把**结构恢复、定位范围、逐步取文**这几件事绑成了一条相对清晰的 agent-friendly 路径。

这也解释了为什么它很容易被误读成普通 retrieval 工具，或者反过来被夸大成一个全新的通用范式：它真正新鲜的地方，可能恰恰在这条中间地带。

## 2. 关于“混合型研究单元”的直觉

### 2.1 `PageIndex` 的尴尬感，反而说明“主题+对象混合型”是有必要的

**触发现状**

- 纯按“主题”看，`PageIndex` 的生态支撑还不够宽，暂时很难直接拉出一个成熟对象簇。
- 纯按“对象”看，它又明显不只是一个普通项目解剖，因为它确实承载了 page-level / structure-aware indexing 这类更大的方法信号。

**当前反思**

`PageIndex` 给人的那种“不完全好放”的感觉，本身可能不是坏事，而是一个很有价值的信号：有些对象的重要性，恰好来自它既是对象本体，又是某种稀疏范式的高信号样本。

如果以后还有类似对象出现，我不应该先急着问“它到底算主题还是对象”，而应先判断：

- 它是不是某个更大想法的主要承载体；
- 离开它以后，这个想法是否还有足够多对象能稳定成立；
- 它挂在当前主题树下后，是否会被稳定误读。

也就是说，混合型不是模糊桶，而更像一种“研究价值来自双重身份”的警报器。

## 3. 关于后续研究推进方式的判断

### 3.1 `PageIndex` 这条线已经不适合继续只磨 `overview`

**触发现状**

- `overview.md` 已经能承担“对象总览”的职责。
- 真正推进理解的是 `notes/deep-research-question-tree.md` 与从中提炼出的 `tree-and-navigation.md`。

**当前反思**

这次实践让我更确信：当对象已经讲清“它是什么”，但研究者仍然答不出“关键机制到底如何工作”，继续打磨 overview 往往只会制造更平滑的模糊感，不会自动带来机制级理解。

换句话说，overview 的上限不是“写得还不够多”，而是它天然不适合承载研究过程中的递进式问题链。

这也是为什么 question tree 不是临时补丁，而像是一种必要的研究驾驶舱。

## 4. 还没压实、但值得盯住的假说

### 4.1 `PageIndex` 可能更接近“结构优先的检索编排层”，而不是单点算法卖点

**触发现状**

- 当前可见本地 SDK 暴露的是文档元信息、文档结构、页内容三个主要接口，而不是一个封装很重的“树搜索器”。
- 示例链路也更像“先看结构，再取正文，再由 agent 作答”。

**当前假说**

`PageIndex` 的长期价值，可能不主要在于某个单独算法组件，而在于它提供了一个更适合 agent 使用的结构化文档工作流底座。

如果这个判断成立，那么未来研究它时，重点可能不只是：

- 构树准不准；

还包括：

- 这种结构优先接口，对 agent 的工具调用顺序意味着什么；
- 它与传统 RAG 中“先召回 chunk 再回答”的默认动作链到底差多远；
- Cloud API / notebook 层是否把这种底座进一步产品化了。

目前这还只是方向性假说，不能直接写成正式结论。

## 5. 下一轮如果继续，最值得追的问题

- 树导航在 Cloud API / notebook 路径里是否有比本地 SDK 更强的显式封装。
- `retrieval-protocol.md` 是否还能继续压实“结构定位”和“正文取用”之间的协议层角色。
- `PageIndex` 的代表性到底更偏“检索增强”，还是更偏“agent 文档操作接口设计”。

# PageIndex 深度研究问题树

本文件不是面向普通读者的对象总览，而是 `PageIndex` 深度研究的**递进式问题树脚手架**。

它的目标不是重复 `overview.md` 已经覆盖的“它是什么、有哪些能力”，而是把后续研究组织成一条可持续推进的主线：

> `PageIndex` 如何定义文档树、构建文档树，并基于这棵树支持搜索 / 导航？

使用方式：

- 当 `overview.md` 已足够支撑初步理解，但不足以支撑机制级研究时，进入本文件。
- 每个问题都区分：**当前已知**、**待确认**、**下一步该查什么**。
- 后续研究不必重新发明问题，而是沿这棵树逐层回答、补证、收口。

---

## 0. 主线问题

### 0.1 主问题

`PageIndex` 如何把长文档组织成树结构，并基于这棵树支持检索 / 导航？

### 0.2 为什么这条主线值得优先研究

当前 `overview.md` 已经能告诉读者：

- `PageIndex` 不是普通 embedding + chunk 检索工具；
- 它依赖页面级、结构化、树状组织来处理长文档；
- 它把“索引”和“导航”绑得比较紧。

但如果想深入理解，还缺少下面这些机制级答案：

- 这里的“树”到底是什么树；
- 树是从什么信号构建出来的；
- 为什么树适合这个对象，而不是别的结构；
- “搜索”和“导航”在系统里分别扮演什么角色；
- 仓库到底开放了哪些接口，允许如何组合使用。

因此，这条主线可以作为后续深度研究的第一优先级。

## 1. 树的定义层

### 1.1 这里的“树”到底指什么

#### 当前已知

- `overview.md`、`pdf-indexing.md` 已明确：`PageIndex` 不是简单页列表，而是试图把文档切成层级节点。
- `pdf-indexing.md` 已描述了 flat TOC list → tree 的构造过程，说明“树”至少包含章节层级。
- 现有文档暗示节点既有文本边界，也可能带摘要、页码范围、父子层级关系。

#### 待确认

- 这个树的最小节点单位更接近“章节节点”还是“文本块节点”？当前源码显示 Markdown 路径直接以标题节点构树，PDF 路径也先产出章节条目，但仍需确认是否存在其他节点类型混入同一树。
- 中间节点和叶子节点在行为上是否有严格分工，还是只是在是否拥有 `nodes` 上表现不同？
- PDF 树是否应理解为“章节树 + 页范围 + 可选摘要”的复合节点，而不是单纯章节标题树？
- Markdown / PDF 的树结构在输出层基本同构，但其生成前驱字段并不相同；这一点还需要继续补证。

#### 第一轮源码核验结论

- `PageIndex` 的统一树节点抽象不是从 PDF 专属逻辑里定义出来的，而是更清楚地暴露在 Markdown 路径的 `build_tree_from_nodes()` 中。
- Markdown 路径生成的标准树节点至少包含：`title`、`node_id`、`text`、`line_num`、`nodes`；见 `pageindex/page_index_md.py:190`。
- PDF 路径在真正树化之前，先维护一组扁平章节条目，核心字段是：`title`、`physical_index`、`appear_start`，随后由 `post_processing()` 补出 `start_index`、`end_index`，再调用 `list_to_tree()` 收敛成树；见 `pageindex/utils.py:433`。
- 这说明 PDF 与 Markdown 的“树”不是完全不同的两套结构：两条路径共享的真正抽象是**带 `nodes` 的层级章节树**；差别主要在于 PDF 需要先从分页文本里恢复章节边界，而 Markdown 直接从标题层级得到树。
- 从输出字段看，PDF 路径在最终结构里更接近“章节树 + 页范围”，Markdown 路径更接近“章节树 + 文本块”；两者之后都可以再叠加 `summary` / `prefix_summary`。

#### 下一步研究动作

- 继续确认 `write_node_id()`、摘要生成和最终输出排序，确认哪些字段属于核心树模型，哪些只是后处理附加信息。

#### 第二轮源码核验结论：层级规则与字段并表

**1）PDF 与 Markdown 的建树规则并不相同，但输出形态收敛**

- Markdown 路径的建树规则是**栈式层级归并**：`build_tree_from_nodes()` 依赖标题级别 `level`，遇到更高或同级标题就持续出栈，再把当前节点挂到最近的更高层父节点下；见 `pageindex/page_index_md.py:190`。
- PDF 路径的建树规则是**编号式父子归并**：`list_to_tree()` 不看标题级别，而是读取 `structure`（如 `1`、`1.2`、`1.2.1`），通过截去最后一段编号得到父节点，再把节点挂到该父节点下；见 `pageindex/utils.py:324`。
- 因此，`PageIndex` 统一的不是“层级恢复算法”，而是**最终递归树输出**：节点通过 `nodes` 挂子节点，形成层级章节树。

**2）第一轮可确认的字段并表**

| 路径 | 树化前关键字段 | 树化后核心字段 | 后处理可能附加字段 |
|---|---|---|---|
| PDF | `structure`, `title`, `physical_index`, `appear_start` | `title`, `start_index`, `end_index`, `nodes` | `node_id`, `text`, `summary` |
| Markdown | `title`, `line_num`, `level`, `text` | `title`, `node_id`, `text`, `line_num`, `nodes` | `summary`, `prefix_summary` |

**3）字段层面的关键含义**

- PDF 路径在树化时**丢掉** `structure`、`physical_index`、`appear_start` 这类恢复过程字段，保留的是页范围语义：`start_index` / `end_index`。
- Markdown 路径在树化时**直接保留正文文本**，并把 `line_num` 作为定位信息留在节点上。
- 这说明 PDF 树更接近“章节树 + 页范围”，Markdown 树更接近“章节树 + 原始文本块”。
- `node_id` 在两条路径中的来源也不同：Markdown 在 `build_tree_from_nodes()` 中一开始就写入；PDF 在树已经形成后，由 `write_node_id()` 统一递归补写；见 `pageindex/utils.py:132`。

**4）对“树是什么”的进一步收敛**

- `PageIndex` 的树不是一个单一静态 schema，而是一个**共享骨架 + 路径特有载荷**的树模型。
- 共享骨架是：`title` + `nodes`。
- PDF 特有载荷主要是页范围；Markdown 特有载荷主要是原始节点文本与行号。
- `summary` / `prefix_summary`、`node_id` 更像附加索引能力，而不是树存在的前提条件。

## 2. 树的构建层

### 2.1 树是从什么输入构建出来的

#### 当前已知

- `PageIndex` 处理的核心输入包括 PDF 和 Markdown。
- `pdf-indexing.md` 已说明 PDF 路径会经过页面解析、TOC 检测、标题校验、树化、节点细分。
- `README.md` 与 `sdk-and-workspace.md` 暗示对象还涉及多文档存储和 workspace，但当前主线应先聚焦单文档树构建。

#### 待确认

- PDF 与 Markdown 是否在更高层共享统一的“索引对象 schema”，还是只在树形输出层相似？
- 是否存在 HTML / 网页 / 其他格式入口，若有，它们是否也落到同一树模型？
- 当前公开接口是否默认把这两条路径视为同一类索引结果，还是调用方需要自己理解差异？

#### 第一轮源码核验结论

- Markdown 路径的主链非常直接：`extract_nodes_from_markdown()` → `extract_node_text_content()` → `build_tree_from_nodes()`；它以标题层级 (`level`) 为结构信号，直接生成树；见 `pageindex/page_index_md.py:32`、`:62`、`:190`。
- PDF 路径的主链则是：`check_toc()` / `meta_processor()` 产出扁平章节条目 → `check_title_appearance_in_start_concurrent()` 校验页首关系 → `post_processing()` 补页范围并树化 → `process_large_node_recursively()` 对超大章节再次递归细分；见 `pageindex/page_index.py:959`、`:1029` 与 `pageindex/utils.py:433`。
- 因此，两条路径共享的不是“前处理方法”，而是**收敛后的树状产物形态**：都以带 `nodes` 的层级结构作为最终索引骨架。
- Markdown 路径是“先有层级，再补文本”；PDF 路径是“先从文本恢复层级，再补页范围 / 文本 / 摘要”。

#### 下一步研究动作

- 把 PDF 主链压缩成更细的 5–7 步机制图，明确每一步的输入/输出字段。
- 继续判断 `process_large_node_recursively()` 是否意味着 PDF 树不是一次性构建，而是“初始树 + 局部再生长”的两阶段结构。

#### 第二轮源码核验结论：PDF 主链中的树化收敛点

- `tree_parser()` 是 PDF 路径收敛成树的主入口：先拿到 `meta_processor()` 产出的章节条目，再做 `add_preface_if_needed()`、`check_title_appearance_in_start_concurrent()`、`post_processing()`，最后进入 `process_large_node_recursively()`；见 `pageindex/page_index.py:1029`。
- 这意味着 PDF 树不是在 `meta_processor()` 内部直接完成的；`meta_processor()` 负责的是**生成并修正扁平章节条目**，真正的树化发生在 `post_processing()`。
- `process_large_node_recursively()` 进一步说明：PDF 路径的树不是“一次生成完毕”，而是“初始树 → 对超大节点再局部无 TOC 生长子树”的两阶段过程；见 `pageindex/page_index.py:1000`。
- 因此，从机制上看，PDF 树更像一个**逐步逼近的结构恢复过程**，而不是一次性静态解析结果。

### 2.2 构树依赖哪些信号

#### 当前已知

从现有 `pdf-indexing.md` 可知，PDF 构树至少涉及这些信号：

- 页面文本解析
- token 统计
- TOC 检测
- 页码对齐
- 标题出现校验
- 标题页首校验

#### 待确认

- 这些信号中，哪些是核心主干，哪些只是修复机制？
- LLM 在构树中的作用是“主构建器”还是“结构修正器 / 缺口补偿器”？
- 有没有显式规则层（正则、启发式、标题级别约束）与 LLM 协同？

#### 下一步研究动作

- 按“确定性处理 / 启发式处理 / LLM 处理 / 后修复处理”四类重画构树流程图。
- 明确每一步的输入、输出和失败后回退路径。

### 2.3 树能处理哪些数据，不能处理哪些数据

#### 当前已知

- 现有文档已说明：PDF 路径不是 OCR-first，文本提取质量会直接影响后续结构判断。
- TOC 不可靠时系统存在降级路径与修复路径。
- 文档结构松散时，系统会尝试无 TOC 路径和大节点递归切分。

#### 待确认

- 哪类 PDF 最适合该树路径：学术论文、技术文档、书籍、规范、报告？
- 哪类数据会明显破坏树质量：扫描件、多栏、复杂表格、目录缺失、章节标题不规范？
- Markdown 路径的限制是否远小于 PDF，还是也存在自己的结构风险？

#### 下一步研究动作

- 从源码和示例里提取“显式失败模式”和“隐式脆弱点”。
- 形成一张“输入条件 → 树质量风险”表。

## 3. 设计动机层

### 3.1 为什么是树，而不是别的结构

#### 当前已知

- 现有文档已暗示：`PageIndex` 想解决的是长文档中“只靠 chunk 很难保持结构语义”的问题。
- 树天然适合表达章节层级、父子范围、粗到细定位。
- 这使它同时适合索引、压缩、逐层定位和局部展开。

#### 待确认

- 官方有没有明确论证“为什么是树而不是扁平 chunk / graph / table of chunks”？
- 树是出于可解释性、结构保真、导航便利，还是模型上下文效率？
- 当前实现是否真的把“树”当核心检索结构，而不是只是一个前处理结果？

#### 下一步研究动作

- 收集官方或源码中体现设计动机的直接证据。
- 把“树的优势”拆成：表达优势、检索优势、推理优势、工程优势。

### 3.2 树在这个项目里解决了什么核心矛盾

#### 当前已知

可以先假设它至少试图缓解这些矛盾：

- 长文档太长，无法一次性平铺给模型；
- 扁平切块丢失章节与上下文边界；
- 只做搜索不够，用户或 agent 还需要逐层定位与理解路径。

#### 待确认

- 哪个矛盾是第一位的：检索精度、解释性、导航性、成本控制，还是索引质量？
- 树是为了让“召回更准”，还是为了让“找到后更会走”？

#### 下一步研究动作

- 把现有说法中的目标拆成主目标 / 次目标，而不是全都并列。

## 4. 搜索 / 导航层

### 4.1 “搜索”和“导航”在这里分别是什么意思

#### 当前已知

- 当前文档使用了“搜索导航”“树导航模式”等说法，但还没把两个词拆开解释。
- 从对象行为推测，“搜索”可能更偏找到相关节点，“导航”可能更偏沿树逐步定位、展开和理解。

#### 待确认

- 本地 SDK 之外（尤其是 Cloud API / notebook 路径）是否存在更强的“树上搜索”封装？
- 当前 demo 里 agent 的 reasoning 到底只是用树做页范围定位，还是还隐含了更细的树上遍历策略？
- “导航”在官方叙述里是用户体验词，还是实现层行为词？

#### 第一轮源码核验结论

- 在本地可见 SDK 中，没有一个显式名为 search / navigate 的树上搜索算法模块；可见接口只有三类：`get_document()`、`get_document_structure()`、`get_page_content()`；见 `pageindex/client.py:28` 与 `pageindex/retrieve.py:81`。
- 因此，本地可证实的最小行为链更像是：
  1. 先取文档元信息（确认状态、页数）
  2. 再取无文本树结构（观察章节树）
  3. 再按页码范围取正文内容
  4. 最后由 agent / LLM 基于这些工具输出答案
- 示例 `examples/agentic_vectorless_rag_demo.py` 的系统提示也明确要求：先 `get_document()`，再 `get_document_structure()` 找相关范围，最后用 `get_page_content()` 抓紧凑页段；见 `examples/agentic_vectorless_rag_demo.py:44`。
- 这说明在本地实现里，“搜索”更接近**基于树结构做相关范围定位**，“导航”更接近**沿结构决定下一步该看哪一段页面**；两者目前主要通过 agent 的工具调用顺序体现，而不是源码里一个独立的检索器对象。

#### 下一步研究动作

- 回到 `retrieval-protocol.md` 与 demo 提示词，补一张“元信息 → 结构 → 页内容 → 答案”的最小行为链图。
- 在后续如果需要扩展到 Cloud API，再单独判断服务端是否提供了更显式的树搜索能力。

### 4.2 树在检索流程里实际扮演什么角色

#### 当前已知

- `retrieval-protocol.md` 已经部分描述了树导航、示例层 reasoning pattern、可能的逐层定位方式。
- 当前可合理猜测：树至少不是纯展示层，而参与了节点级定位与路径选择。

#### 待确认

- 服务端或 notebook 路径是否把树直接变成召回器，而不仅是结构浏览器？
- 本地行为链里，检索单位更接近页还是节点？当前工具接口仍然以页内容返回为主。
- demo 之外是否存在直接按 node_id / section 检索的调用方式？

#### 第一轮源码核验结论

- 在本地 SDK 层，树首先扮演的是**结构浏览与范围定位结构**，而不是一个直接返回答案的检索器。
- `get_document_structure()` 明确会去掉 `text` 字段，只返回树骨架；见 `pageindex/retrieve.py:100`。这说明它的主要用途是让 agent / 用户先看结构，而不是直接把节点内容当答案返回。
- `get_page_content()` 的返回单位是页（PDF）或标题所在行号对应的节点文本（Markdown）；见 `pageindex/retrieve.py:110`。因此，本地最终被消费的正文单位仍然更偏“页内容 / 节点文本”，而不是纯树节点对象。
- 从 demo 的使用方式看，树更像是**导航结构 + 解释结构**：它帮助 agent 决定去哪几页取内容，而真正用于回答问题的还是页内容文本。
- 所以在当前可见实现里，更稳的表述不是“PageIndex 在本地提供了一个完整树搜索器”，而是“PageIndex 提供了树结构索引，供 agent 先定位、再取页、再回答”。

#### 下一步研究动作

- 形成一个“结构树负责定位，页内容负责作答”的二层角色图。
- 再判断 `doc_description`、`summary`、`prefix_summary` 是否在其他调用路径中参与了更强的候选筛选，而不只是展示或压缩。

## 7. 当前研究状态面板

### 已相对清楚

- `PageIndex` 的研究入口应放在“树结构 + 搜索/导航”主线上。
- PDF 构树至少依赖 TOC、标题校验、页码修复、树化与递归细分。
- 在本地 SDK 中，树首先承担的是结构定位与导航作用，而最终作答仍主要依赖页内容文本。
- 递归细化不会引入第二套树 schema，但会产生来源和粒度不同的 PDF 子节点。
- 当前 `overview.md` 更适合初步理解，不足以承载机制级深度研究。

### 仍然模糊

- Cloud API / notebook 路径是否存在比本地 SDK 更强的树搜索封装
- 树接口是否支持按 node_id / section 直接访问
- 官方层面对“为什么是树”的直接论证

### 第三轮源码核验结论：递归细化节点与初始节点的关系

- `process_large_node_recursively()` 不会引入一套全新的树 schema；它对子节点再次调用 `meta_processor(..., mode='process_no_toc')`，随后仍然经过 `check_title_appearance_in_start_concurrent()` 和 `post_processing()`，因此新长出来的子树在结构骨架上仍然是 `title` + `start_index` / `end_index` + `nodes`；见 `pageindex/page_index.py:1000`。
- 但它也不是“完全无差别同构”。差异主要有三点：
  1. **来源不同**：递归细化节点不是从全局 TOC 或初始章节恢复出来，而是从父节点页范围内部重新走一遍无 TOC 生长路径。
  2. **粒度不同**：它只在超大节点内部生长更细子树，因此这些节点天然是“局部再切分”结果，而不是文档第一层主干。
  3. **父节点边界会被回写调整**：如果细化后首个子节点与父标题重复，代码会跳过该重复节点，并用第二个子节点的 `start_index` 回写父节点 `end_index`；否则直接用第一个子节点的 `start_index` 回写。见 `pageindex/page_index.py:1013`。
- 因此，更稳的判断是：**递归细化节点与初始 PDF 节点在输出字段上基本同构，但在生成来源、语义粒度和父边界更新逻辑上并不完全等价。**

### 下一轮推荐研究任务

1. 补一张“初始 PDF 节点 vs 递归细化节点”的对照表，把字段相同、来源不同、语义不同这三层拆开。
2. 再补“结构树定位 → 页内容作答”这条本地搜索 / 导航行为链的正式图示或短说明。
3. 最后再补“接口与组合边界”，避免过早滑向外围生态。

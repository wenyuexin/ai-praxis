# backlog.md：LLM Wiki 研究缺口

本文件记录当前 `llm-wiki/` 目录仍待继续澄清的问题和比较缺口。这些基于一轮去锚点调研（`temp/web-search/5.md` / `6.md` / `7.md` / `8.md`）识别，仅作为上游输入，不应直接当作稳定结论。

---

## 1. llm-wiki 与"查询前组织 / 离线知识编译"的边界

当前 `project-studies/` 主要集中在以 wiki/markdown 为最终产物的编译路线。但在外部调研中出现了多条与 llm-wiki 共享"查询前组织知识"这一核心理念、但产物形态和工程路径不同的方向，例如 GraphRAG 的社区摘要方法、STORM 的主动合成引擎、Sleep-Time Compute 的离线预计算范式。

待验证的问题：

- 这些方向是否与 llm-wiki 共享同一主问题（"查询前如何组织知识"），只是产物形态不同？
- 还是它们的核心约束机制（图拓扑 vs 文档编译 vs 参数化）差异太大，不应归为同一路线？

当前样本不足以判断，需要继续收集。

---

## 2. llm-wiki 与 Agent Memory / 运行时状态的边界

外部调研中出现了多个从持久化记忆角度探索知识组织的方向（Memoria、MemoriesDB、Collaborative Memory）。它们与 llm-wiki 共享"知识需要跨会话持久化"这一前提，但关注点更偏向 Agent 的内部认知状态，而不是外部知识的编译产物。

待验证的问题：

- 持久化知识基础设施（轴 A）与 Agent 记忆 / 运行时状态（轴 B）是否存在清晰的技术边界？
- 还是说，在现有实现中，两者已经开始融合（如某个项目同时覆盖知识编译和记忆演化）？
- 如果边界模糊，llm-wiki 应保持以 wiki 编译为重心，还是应同步扩展记忆方向的研究？

---

## 3. "知识产物"的不同形态与比较维度缺失

当前 overview.md 对"LLM Wiki 产物"的讨论主要围绕 markdown wiki 展开。外部调研显示知识产物的形态可能更丰富：

- 结构化知识库（Dify Knowledge）
- 图社区摘要（GraphRAG）
- 可复用 prompt 知识库（Prompt Cache / Liquid Prompting）
- key-value 注入（KBLaM）
- 参数化补丁（Model Editing）—— 此方向距离较远，暂不作为 llm-wiki 的主动比较对象

待验证的问题：

- 这些不同形态的知识产物，是解决同一主问题的不同方案，还是各自服务于完全不同的使用场景？
- llm-wiki 的 markdown wiki 形态在什么条件下是更优选择，什么条件下不是？
- 是否需要在 overview.md 中补充一个"知识产物形态"比较维度？

---

## 4. 非检索知识注入机制的关系

外部调研发现多条不经过传统检索路径的知识传递机制：KBLaM（key-value attention 注入）、KGA（知识图谱引导注意力）、参数化模型编辑（EasyEdit）。它们与 llm-wiki 的"编译即用"理念有部分重叠，但技术路径完全独立。

待验证的问题：

- 这些机制与 llm-wiki 是互补关系（在不同场景使用），还是竞争关系（最终会取代基于文档的编译模式）？
- llm-wiki 是否需要在自己的 overview 中讨论这些替代路径，还是保持专注于文档编译范式？

---

## 5. 共享知识与权限治理维度

Collaborative Memory、MATRIX ontology 等方向引入了多用户 / 多 Agent 的知识共享、权限控制和冲突解决。这是当前 llm-wiki 研究完全缺失的维度。

待验证的问题：

- 多 Agent 共享知识基础设施是 llm-wiki 范式的自然扩展，还是一个独立的问题域？
- 如果 llm-wiki 需要支持知识共享和权限治理，当前的编译模型是否需要根本性的架构调整？

---

## 6. 评估维度的缺口

外部出现了 AIR-Bench / Knowledge Bench 等评估框架，它们评价的不是检索准确率，而是"AI 使用外部知识的综合能力"（知识更新后能否正确回答、多源知识能否整合、知识冲突时如何处理）。

待验证的问题：

- 这些评估框架是否可以直接用来评价 llm-wiki 范式下的知识质量？
- 还是说，llm-wiki 需要自己的评估方法和基准？

## 一、核心理念：从「检索-生成」到「编译-累积」

传统 LLM 知识应用（RAG）遵循 **“查询时检索 + 即时生成”** 模式：文档被静态存储，查询时检索相关片段，由 LLM 拼凑答案。这种方式每一次回答都是从零开始的“重发现”，知识没有积累，无法形成持续的复利。

LLM Wiki 提出了一种**范式翻转**：让 LLM 在**摄入时**就对知识进行一次性深度加工，构建一个持久的结构化 Wiki，并在后续查询中直接使用已编译好的知识。二者的根本区别可概括为：

| 维度 | 传统 RAG | LLM Wiki |
|------|----------|----------|
| 知识处理时机 | 查询时 | 摄入时 |
| 知识形态 | 原始分块，隐式关联 | 结构化页面，显式链接 |
| 跨文档综合 | 每次临时拼凑 | 持续维护的合成页面 |
| 知识演化 | 无法追溯和更新 | 可增量更新、标记矛盾、修订历史 |
| 维护负担 | 低（无长期维护） | 几乎零（由 LLM 承担） |

其类比是**编译器**：将原始资料视为源代码，通过 LLM “编译” 成结构化的中间表示（Wiki），而后可以反复“执行”（查询），而不必每次重新解析源代码。

## 二、三层架构：知识工程的最小完备抽象

文档提出了一个高度凝练的三层模型，将整个系统解耦为不可变的原始数据层、可生长的知识表示层和定义行为的控制层。

### 1. Raw Sources（原始资料层）
- **性质**：不可变的“事实源”（truth source）。LLM 只能读取，不得修改。
- **形式**：文章、论文、笔记、图片、数据文件等，由人工收集并放入 `raw/` 目录。
- **设计意图**：保证知识建构的**可溯源性**，所有编译结果均能追溯到原始材料。

### 2. Wiki（知识表示层）
- **性质**：由 LLM 全权生成和维护的一组 Markdown 文件。
- **内容**：摘要页、实体页、概念页、对比页、综合合成页、索引与日志。
- **特征**：
  - 页面间通过 `[[wikilink]]` 构成显式的**语义网络**。
  - 知识不是“文档库”，而是一个**正在生长的、一致的、交叉引用的数据库**。
- **核心价值**：将 LLM 的推理结果**固化**为可复用的结构化资产，后续查询直接在已建构的环境中进行。

### 3. Schema（控制层）
- **性质**：一份描述 Wiki 结构、命名约定、页面格式、工作流规则、写作风格的系统规范文档（如 `CLAUDE.md` 或 `AGENTS.md`）。
- **作用**：把通用 LLM 调校为**特定领域的 Wiki 维护者 Agent**，使其行为可预期、输出标准化。
- **进化机制**：由人与 LLM 在协作过程中**共同迭代**，随领域需求逐渐稳定。
- **技术定位**：相当于整个系统的 **“元知识库”**，是知识工程进入长期可维护状态的关键。

三层之间形成严格单向依赖：Schema 约束 Wiki 的生成规则；Wiki 从 Raw 中提取并整合信息；Raw 不受任何上层影响，保证系统根部的纯净和可追溯。

## 三、三种核心操作（Operations）：知识生命期的完整闭环

文档定义了 **Ingest、Query、Lint** 三个核心操作，分别对应知识库的**构建、消费、健康保障**，三者相互配合形成可持续增长的闭环。

### 1. Ingest（摄入）——知识的编译与一致性维护

Ingest 是系统最复杂的操作，可视为一个**多目标、多阶段的编译流水线**。

**标准工作流：**
1. **读取源文件**：LLM 解析新加入的 Raw 文档，理解其主题、实体与关键论断。
2. **人机研讨**：与人讨论关键信息点，人在回路中给出定性引导（如强调方向），但不涉足具体的组织劳动。
3. **创建/更新页面**：
   - 生成该源的**专属摘要页**。
   - 更新**相关实体页**（追加信息，标注新发现或与旧信息的矛盾）。
   - 调整**综合合成页**，将新知识纳入高层综述，强化或挑战既有结论。
   - 更新**索引文件**，新增条目并调整分类。
4. **记录日志**：在 `log.md` 中追加一条结构化日志条目。

一次 Ingest 可能触碰 10–15 个文件，本质上是执行一次**小范围的知识图谱一致性维护**。该操作支持两种自主级别：
- **监督模式**：逐篇摄入，人仔细审核每次更改。
- **批量模式**：多篇同时摄入，减少人工干预，由 Schema 中的规则保证质量。

### 2. Query（查询）——基于结构化知识的消费与反哺

Query 不再是“搜索原始文档然后生成”，而是一套**先导航后检索、生成并可回存**的流程。

1. **粗定位**：LLM 首先读取 `index.md`，根据分类和单行摘要快速锁定可能相关的页面集合——这是一种利用 LLM 语言理解能力的 **“文本式剪枝”**，无需引入向量检索基建。
2. **精读与综合**：深入阅读候选页面，综合信息后生成带引用的答案。
3. **多模态输出**：答案形式可以是 Markdown 页面、比较表格、Marp 演示文稿、图表（Matplotlib）或画布（Canvas）——满足不同表达需求。
4. **回存闭环**：有价值的答案（对比分析、新发现、综合洞察）被**归档为新页面或补充到现有页面**。这意味着每一次深入提问都有可能**沉淀为永久的知识资产**，避免探索成果消失在聊天历史中。

Query 操作使知识库进入**正向复利循环**：浏览即贡献，提问即建设。

### 3. Lint（检查）——知识库的持续质量审计

Lint 是用户**主动触发**的健康检查，利用 LLM 跨页面语义分析能力对 Wiki 进行静态分析，识别：

- **矛盾**：不同页面对同一主题的断言冲突。
- **陈旧内容**：后续摄入的事实已推翻或修正了早期结论。
- **孤岛页面**：缺少入链，未被充分整合进知识网络。
- **缺失概念**：反复被提及的重要术语尚无独立页面。
- **链接缺失**：语义相关但未显式建立双向链接。
- **数据缺口**：某些观点缺少来源支撑或可通过外部搜索补全。

这一机制本质上是将**持续集成（CI）理念引入知识管理**：用 LLM 的“无聊但精确”替代人类难以坚持的周期性审查。Lint 不仅发现问题，更能**主动建议**下一步应寻找的资料或应探索的问题，为知识库的进化提供方向引导。

## 四、导航系统：极简但鲁棒的索引与日志

在刻意不使用向量数据库的情况下，文档设计了两个全局纯文本文件，构成低成本、高可维护性的导航系统。

### 1. `index.md` —— 内容维度地图
- **结构**：按类别（实体、概念、来源等）组织，每个条目包含页面链接和一句话摘要，可附加元数据（日期、源数量等）。
- **更新机制**：每次 Ingest 时由 LLM 自动维护。
- **作用**：Query 时的**第一级过滤**。LLM 先扫描索引，迅速缩小需要深度阅读的页面范围。在“数百个页面、上百个来源”的规模下，该方法的性能足够且不引入任何外部依赖。
- **设计哲学**：利用 LLM 的阅读速度和语义判断代替传统搜索技术，呈现“文本即数据库”的思想。

### 2. `log.md` —— 时间维度日志
- **结构**：仅追加，条目以统一格式记录（如 `## [2026-04-02] ingest | Article Title`），同时记载 Ingest、Query、Lint 等所有活动。
- **可解析性**：统一格式使得日志可被简单 Unix 工具（如 `grep`）查询，无需专门工具。例：`grep "^## \[" log.md | tail -5` 即可获取最近 5 条活动。
- **核心价值**：
  - **LLM 的短期记忆**：当 Agent 再次启动时，读日志可快速了解“最近做了什么”，避免重复劳动，维持工作连续性。
  - **人类的叙事线索**：提供知识库生长的“故事线”，便于回顾思考脉络。
  - **审计与可溯源性**：任何改动都有记录，配合 Git 版本控制，形成完整的变更历史。

## 五、实践技巧（Tips and Tricks）：局部工具链与工作流优化

该部分不涉及宏观架构，而是针对个人落地的**工程微操作**，旨在降低摩擦、增强人类监管能力。

### 1. 素材流入：Obsidian Web Clipper
一键将网页文章清洗为洁净 Markdown，直接放入 `raw/`，消除手动整理格式的负担。

### 2. 本地化离线：图片下载脚本
- 配置 Obsidian 将附件自动存入固定目录（如 `raw/assets/`），并设置快捷键（`Ctrl+Shift+D`）批量下载图片。
- **目的**：防止链接失效，并解决当前 LLM 多模态读取技术限制。工作流上采用“先读文本，再按需喂入本地图片”的两步策略，实现可靠的图文理解。

### 3. 人类宏观视图：Graph View
Obsidian 的图谱功能将 `[[wikilink]]` 可视化，让人可以直观判断知识结构的**中心节点、孤岛页面与模块化程度**，为 Lint 和结构优化提供全局观。

### 4. 输出多样化：Marp 插件
Markdown 直接转换为演示文稿，使 Wiki 不仅能读，还能直接用于“展示”。Query 的结果可被编译为 Slide，拓展了知识的再利用维度。

### 5. 动态结构查询：Dataview 插件
若 LLM 在页面中写入 YAML 前置元数据（标签、日期等），Dataview 可以像数据库查询一样动态聚合内容。这为更高级的 Lint 和复查需求提供了灵活的查询手段。

### 6. 终极安全网：Git
Wiki 的底层就是 Markdown 文件仓库，因此天然享有版本控制、差异对比、回滚与分支协作能力，无需额外基建。

## 六、可行性根基：“为什么这行得通”

文档从**行为经济学**角度解释了为什么 LLM Wiki 能克服传统知识库必然衰亡的宿命。

> 人类维护知识库失败的根本原因是**维护负担的增长速度远超价值获取速度**。

知识的整理、交叉引用更新、矛盾标注、一致性修正等所谓的“胶水工作”——其认知负荷低却极度消耗心力，最终被抛弃。LLM 恰恰在这些维度上具有绝对优势：
- **零边际成本维护**：LLM 不会厌烦、不会遗忘，能一次性精确更新数十个关联文件。
- **角色重分配**：人从“阅读+思考+维护”的三重负担中解放，只保留**策展、引导、高阶思考**——人做高价值决策，机器做高正确率的文书工作。

Karpathy 将这一理念追溯至 Vannevar Bush 1945 年构想的 **Memex**（记忆扩展机）——一种私人化的、通过“信息路径”（associative trails）彼此连接的增强记忆设备。LLM Wiki 在半个多世纪后补齐了 Bush 未能解决的最后一块拼图：**日常维护该由谁承担**。

## 七、设计哲学与边界声明（Note）

文档最后通过明确的自我定位，区别了“通用模式”与“具体实现”，体现了成熟的工程设计智慧。

### 1. 有意为之的抽象性
文档**只描述模式，不给出具体实现**。目录结构、Schema 的具体写法、页面格式等全部留待用户与其选择的 LLM Agent 共同决定。原则是跨领域通用的，但方案必须是领域特化的。

### 2. 一切都是可选的、模块化的
所有提及的工具（图片下载、搜索引擎、Marp、Dataview）都是**可插拔组件**：
- 纯文本用户可忽略图片处理；
- 小规模 Wiki 只需 `index.md`，不需要额外搜索工具；
- 无需演示的领域不使用 Marp。

其核心原则是**从最小可行系统开始，在需要时增补组件**，拒绝过度工程化。

### 3. 文档的接收对象是「人+LLM」协作体
这份文档本身不是给人阅读后就手工执行的指令，而是作为**规范需求文档**，由人类和 LLM Agent 共同解读、讨论，然后落地为属于他们自己的具体系统。这意味着文档的性质是一种**混合团队的 PRD**，而非传统技术教程。

## 八、总结

LLM Wiki 不是一款工具，也不是一个可以直接下载使用的应用，而是一套**以 LLM 为运行时的知识工程方法论**。它通过以下技术洞察重构了个人/团队知识管理：

1. **将 LLM 的昂贵推理从查询时前置到摄入时**，实现知识的编译与累积；
2. **用三层架构解耦数据、知识表示和控制逻辑**，实现系统的可维护性；
3. **通过 Ingest、Query、Lint 三个闭环操作**，把知识库变成一个持续活化的有机体；
4. **用纯文本的索引与日志替代重型基础设施**，在足够好的规模下保持零运维；
5. **把人的角色战略性升维**，让人类只做策展与思考，把“胶水工作”全部外包给不知疲倦的 LLM。

这份文档本身也示范了一种新的设计文档范式：抽象到只传递模式，具体落地由人与 AI 协作完成——其方法论不仅适用于知识库构建，也适用于一切需要长期维护的智能系统设计。

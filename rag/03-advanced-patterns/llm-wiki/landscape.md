# LLM Wiki 研究地形图：板块、边界与下钻路径

> 适用范围：LLM Wiki / knowledge compiler / persistent wiki 这一类知识编译范式的结构化研究。
> 阶段状态：结构化研究文，用于稳定本目录的板块划分、边界判断与下钻协作。
> 使用说明：本文不承担对外主综述职责。它主要回答“本主题内部如何切分、哪些内容归入核心生态、下层文档如何分工、继续深入时应按什么路径下钻”。如果只想快速建立整体认知，先读 `overview.md`。

## 一、这篇文档负责什么

`overview.md` 负责让读者快速建立对 LLM Wiki 范式的整体理解；本文则负责把这份理解进一步拆解为可维护的研究结构。

换句话说：

- `overview.md` 讲 **LLM Wiki 是什么、为什么重要、有哪些主要路线**
- `landscape.md` 讲 **本目录为什么这样切、哪些内容应放哪里、不同层次文档如何协作**

因此，本文优先服务以下任务：

- 判断新材料应归入哪个板块
- 判断某个项目是核心实现、相邻生态，还是应移出本目录
- 为后续对象研究、模式解读和辅助材料提供稳定边界
- 给深入读者提供一张“从全景综述继续往下读”的研究地图

## 二、本目录的核心切分

当前 `llm-wiki/` 目录应稳定为四个互补层次：

1. `overview.md`
   - 人类向主文
   - 负责整体理解、关键主线、路线比较与阶段判断

2. `karpathy-pattern.md`
   - 上游原点专题
   - 负责解释 Karpathy 原始模式提出了什么、其最小架构和操作闭环是什么、为什么它构成独立范式

3. `project-studies/`
   - 对象研究层
   - 负责各个代表实现、工具或项目的对象级分析

4. `notes/`
   - 原始材料与研究辅助层
   - 负责原始文档、摘录、辅助材料与不宜直接写入主线正文的资料

这四层不是任意并列，而是有明确依赖关系：

- `overview.md` 给出全景理解
- `karpathy-pattern.md` 提供原点模式基线
- `project-studies/` 提供对象证据与实现分化
- `notes/` 提供可追溯的原始材料支撑

## 三、板块划分：这个主题内部有哪些核心问题

围绕 LLM Wiki，当前最稳定的研究板块可以分成四组。

### 3.1 原点模式板块

关注问题：

- Karpathy 到底提出了什么最小抽象
- “摄入时编译”与“查询时检索”的差异是什么
- 为什么这个模式不只是另一个检索插件

对应文档：

- `karpathy-pattern.md`
- `notes/karpathy-raw.md`

这一板块的职责是提供整个目录的抽象基线。后续所有实现路线，原则上都应能回到这里回答：它继承了原点模式的哪一部分，又偏离了哪一部分。

### 3.2 实现路线板块

关注问题：

- 这套模式被实现成了哪些产品形态
- GUI、CLI、Agent-native、Obsidian、MCP 服务分别怎么落地
- 它们在存储、摄入、图谱、搜索、质量保障上有哪些关键取舍

对应文档：

- `overview.md` 中的路线综述
- `project-studies/` 下的对象研究

这一板块承接的是“从模式到系统”的问题，也是当前目录最容易膨胀的部分。治理原则是：

- 综述层保留在 `overview.md`
- 具体项目细节下沉到 `project-studies/`
- 不把对象级实现细节长期堆积在地形图中

### 3.3 相邻生态与边界板块

关注问题：

- 哪些项目属于 LLM Wiki 核心生态
- 哪些只是相邻的 Obsidian AI 助手、通用 Agent 工具或 RAG 基础设施
- 为什么有些项目值得作为参照，但不应纳入核心实现

这一板块是本文最重要的职责之一，因为边界漂移是此目录最常见的失真来源。

当前边界判断可稳定为：

- **核心生态**：直接围绕 ingest / compile / query / lint / schema / persistent wiki 展开
- **相邻生态**：对知识工作流有帮助，但核心目标不是“把原始资料编译为持续维护的 wiki 中间层”
- **外部参照**：GraphRAG、LightRAG、通用 MCP 工具、通用 Obsidian 助手等，可用于横向比较，但不应抢占本目录主位

### 3.4 设计模式与工程取舍板块

关注问题：

- 摄入如何做：单步、两步、增量、批量、审查队列
- 图谱如何做：显式双链、多信号、推断边、图可视化
- 质量如何保：Lint、Health、审查、dry-run、自知之明、级联删除
- 自动化如何做：夜间同步、后台管线、MCP 服务化、CLI 维护工具

这一板块横跨多个对象研究，但不等同于任何单一实现。它的正确落位通常是：

- 主结论留在 `overview.md`
- 结构比较和归纳逻辑，由本文点出主轴
- 证据和对象细节落到 `project-studies/`

## 四、什么应放这里，什么不应放这里

### 4.1 应优先纳入本目录的内容

- LLM Wiki / knowledge compiler / persistent wiki 范式本身
- Karpathy 原始模式及其直接演化实现
- 围绕 wiki 编译、持续维护、质量保障展开的代表项目
- 以 Markdown / graph / schema / ingest / query 为核心机制的实现路线

### 4.2 应谨慎作为相邻参照的内容

- Obsidian AI 助手
- 通用 Agent/Skill 运行时
- 通用知识图谱或 GraphRAG 基础设施
- 通用文档搜索、向量数据库或 RAG 平台

这些内容可以在 `overview.md` 的边界章节中作为比较对象出现，也可以在对象研究中说明其与 LLM Wiki 的关系；但如果它们的核心目标并不是构建可持续维护的 wiki 中间层，就不应直接成为本目录的主轴。

### 4.3 通常不应放入本目录的内容

- 一般 RAG 基础、检索算法、向量库综述
- 通用 Agent 框架、Tool registry、Skill 系统总览
- 一般知识管理软件或 PKM 生态的泛综述
- 与 LLM Wiki 仅有弱关联的个人笔记工具

## 五、下层文档如何协作

### 5.1 `overview.md` 与本文的关系

两者的正确关系不是互相重复，而是：

- `overview.md` 负责让读者理解“主题主线”
- `landscape.md` 负责让维护者和深入读者理解“研究结构”

如果某段内容主要在回答：

- “LLM Wiki 是什么？”
- “为什么它重要？”
- “主要路线差异在哪？”

优先写进 `overview.md`。

如果某段内容主要在回答：

- “为什么这里要单独建 `karpathy-pattern.md`？”
- “某类项目为什么算边界对象而不是核心实现？”
- “后续继续深入时应先看哪个板块？”

优先写进 `landscape.md`。

### 5.2 `karpathy-pattern.md` 的职责边界

`karpathy-pattern.md` 不是缩短版 `overview.md`，也不是对象研究。它承担的是：

- 原点模式解读
- 最小架构和操作闭环
- 作为全景理解的抽象基线

因此：

- 只要问题落在“Karpathy 原始模式本身提出了什么”，优先写那里
- 只要问题落在“生态如何扩展、对象如何分化”，优先回到 `overview.md` 或 `project-studies/`

### 5.3 `project-studies/` 的职责边界

`project-studies/` 承担对象级研究，而不是再次复写全景综述。对象研究至少应回答：

- 这个项目是什么形态
- 它解决了 LLM Wiki 的哪个子问题
- 它相对其他路线的独特取舍是什么
- 它在本目录的整体结构里处于什么位置

如果一个对象研究开始反复重复“LLM Wiki 是什么、整体生态如何演化”，那说明它正在越界回到综述位。

### 5.4 `notes/` 的职责边界

`notes/` 用于原始材料和研究辅助，不应用来长期承载已经稳定的结构判断。换句话说：

- 原始摘录、临时辅助材料可留在 `notes/`
- 稳定的主线判断进 `overview.md`
- 稳定的结构判断进 `landscape.md`

## 六、建议阅读路径

### 6.1 面向首次进入本目录的读者

- 先读 `overview.md`
- 再根据兴趣读 `karpathy-pattern.md`
- 最后进入 `project-studies/` 看具体路线

### 6.2 面向准备继续建设本目录的维护者

- 先读本文，理解板块与边界
- 再检查 `overview.md` 是否仍与当前板块划分一致
- 然后判断新增对象应进入 `project-studies/`、`notes/`，还是不该进入本目录

### 6.3 面向需要补对象研究的人

- 先确认该对象是否属于 LLM Wiki 核心生态
- 再确认它主要对应哪个研究板块
- 最后决定是在 `project-studies/` 新建对象研究，还是只在 `overview.md` 中做少量比较提及

## 七、何时应继续调整本目录结构

以下情况通常说明本目录仍需进一步治理：

- `overview.md` 再次膨胀出大量目录级结构说明
- `landscape.md` 再次漂移成第二篇全景综述
- `project-studies/` 中出现大量其实属于相邻生态、应移出的对象
- `notes/` 中长期滞留已足够稳定、应回流主线的判断
- 同一类结构判断同时散落在 `overview.md`、`project-studies/` 和 `notes/` 中

## 八、本文不承担什么

本文不负责：

- 替代 `overview.md` 成为对外主综述
- 逐个评测所有代表实现
- 承接原始资料摘录
- 代替 backlog、candidate 或 conflict 这类元信息文件

如果只想快速知道 LLM Wiki 是什么，读 `overview.md`；如果要判断本目录今后该怎么继续长，读本文。
